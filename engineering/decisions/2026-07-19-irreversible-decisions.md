# Решение: правило практической необратимости архитектурных решений

## Назначение

Фиксирует правило принятия архитектурных решений, введённое архитектором проекта 2026-07-19.

## Содержание

### Правило

**Каждое архитектурное решение должно быть практически необратимым.**

Если есть ощущение, что решение может измениться через неделю — его не нужно кодировать. Оно остаётся в статусе **Decision Required**, а проектирование продолжается до тех пор, пока решение не станет устойчивым.

### Зачем

- Исключает бесконечные рефакторинги: код пишется только по решениям, которые не «поплывут».
- Сохраняет последовательность системы по мере роста (десятки тысяч строк, десятки AI-агентов).
- Прецеденты уже были: Assignment/Result в контракте Agent и имя «Engine» пришлось откатывать, потому что решения (ADR-005, устройство Task Engine) ещё не созрели ([ревью EPIC-002](../reviews/2026-07-19-epic-002-code-review.md)).

### Как применяется

1. ADR принимается только когда команда готова жить с решением годами; сомнение — сигнал оставить Decision Required.
2. Код, зависящий от несозревшего решения, не пишется; вместо него — абстрактные заглушки (пример: `Request`/`Response` агента до ADR-005).
3. Правило внесено в шаблон ADR ([ADR-000](../../docs/adr/ADR-000-template.md)).
