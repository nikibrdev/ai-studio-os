# Решение: Stability Assessment, статус Reference и защита утверждённых спецификаций

## Назначение

Фиксирует финальный, шестой по счёту уровень дисциплины для этапа 1 EPIC-003, введённый архитектором проекта 2026-07-20 при финальном (Final Architecture) ревью PR #36 — первой доменной спецификации проекта (Artifact, TASK-029).

## Содержание

### Контекст

Ревью PR #36 велось не как поиск недостатков, а с одним вопросом: «Готов ли этот документ стать эталоном для всех последующих доменных спецификаций?». Ответ — да, но с одним дополнением (не блокирующим merge) и тремя решениями, которые вступают в силу для последующих спецификаций (TASK-030 и далее), а не для этого PR.

### Решение 1 — Stability Assessment (20-й раздел)

[Specification-Domain.md](../../.claude/templates/Specification-Domain.md) дополнен разделом **Stability Assessment** — самым последним в документе, короче остальных: `Current assessment` (Stable for v1 | Provisional | Experimental), `Confidence` (High | Medium | Low), `Known revision triggers` (конкретный перечень, обычно пересекающийся с «Ожидаемыми» Future Extensions). Назначение: человек, впервые открывший спецификацию через годы, сразу видит, устоялась модель или временна, — без чтения всего текста.

### Решение 2 — статус Reference

Цепочка статусов документа расширена: Черновик → Утверждена → **Reference** → Реализована. Reference не присваивается автоматически при утверждении — это отдельное решение архитектора, означающее не «принято», а «эта спецификация — образец для следующих». Artifact ([docs/specifications/domain/artifact.md](../../docs/specifications/domain/artifact.md)) — первая спецификация со статусом Reference в проекте.

### Решение 3 — защита утверждённых/Reference спецификаций

После присвоения статуса Утверждена (тем более Reference) изменение спецификации допускается только одним из трёх путей:

1. новый ADR;
2. отдельный Domain Revision PR с явным обоснованием изменения;
3. изменение, вынужденное утверждённой спецификацией другой сущности — обязательно с Delta Review (решение 4).

Прямое, не документированное изменение утверждённой спецификации — запрещено. Цель: утверждённые спецификации становятся действительно устойчивыми архитектурными артефактами, а не рабочими черновиками, которые тихо меняются при следующей задаче.

### Решение 4 — Delta Review (вступает в силу с TASK-030)

Начиная со второй спецификации Domain Layer, [чек-лист](../../.claude/checklists/DomainSpecificationReview.md) дополняется обязательным Delta Review — короткой проверкой новой спецификации относительно всех уже утверждённых (в первую очередь — Reference), тремя вопросами:

1. Требует ли она изменить существующую утверждённую модель?
2. Использует ли она уже утверждённые понятия одинаковым образом?
3. Не вводит ли она новое понятие, уже существующее под другим названием?

Назначение — предотвратить постепенное «расползание» терминологии и молчаливый пересмотр уже принятых решений при проектировании каждой следующей сущности. Для TASK-029 (Artifact) Delta Review не проводился — не с чем сравнивать, это первая спецификация.
