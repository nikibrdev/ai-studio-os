# Решение: Three-Pass Review, новые разделы спецификации и сознательный темп для EPIC-003

## Назначение

Фиксирует четвёртый по счёту уровень дисциплины для этапа 1 EPIC-003 (после ADR-016, открытия EPIC-003 и Domain Specification Review), введённый архитектором проекта 2026-07-20, непосредственно перед фактическим стартом TASK-029.

## Содержание

### Контекст

После введения обязательного 12-раздельного шаблона ([Domain Specification Review](2026-07-20-domain-specification-review.md)) архитектор решил, что для официального старта TASK-029 нужен ещё один слой: не только СТРУКТУРА документа, но и ПРОЦЕСС его проверки, плюс несколько разделов, которые структура не покрывала (конкретные примеры, сознательно исключённый функционал, тест на самостоятельность сущности, явный сигнал завершённости).

### Решение 1 — Three-Pass Review

Каждая доменная спецификация перед статусом «Утверждена» проходит три независимых прохода ([чек-лист](../../.claude/checklists/DomainSpecificationReview.md)):

1. **Pass 1 — Internal Consistency**: только сам документ (противоречия, циклы в связях, совместимость инвариантов, допустимость команд).
2. **Pass 2 — Cross-domain Consistency**: не конфликтует ли сущность с остальными (Execution, Task, Project, Executor, Workflow и т.д.), уже утверждёнными или ещё нет.
3. **Pass 3 — Future-proof Review**: «Если через три года у системы появятся новые исполнители, новые типы артефактов и распределённое выполнение — придётся ли менять эту спецификацию?» — ответ «да» означает недостаточную абстрактность, спецификация дорабатывается до утверждения, а не после.

### Решение 2 — новые обязательные разделы шаблона

[Specification-Domain.md](../../.claude/templates/Specification-Domain.md) дополнен четырьмя разделами (итого 16 вместо 12):

- **Examples** — не код, не JSON, не Go: содержательные примеры конкретных экземпляров сущности (например, для Artifact — Markdown-спецификация, Pull Request, Figma-дизайн, test report, ADR). Часто именно примеры показывают, что сущность описана правильно.
- **Non-Goals** — отдельно от Anti-Responsibilities: не «чего сущность не делает», а «что эта версия спецификации сознательно не решает» (например, для Artifact v1 — версионирование, диффы, совместное редактирование, ACL, репликация). Не «никогда», а «не сейчас».
- **Removal Test** — один обязательный вопрос: «Что произойдёт с системой, если удалить эту сущность целиком?». Показывает, действительно ли сущность самостоятельна, а не является атрибутом другой (пример архитектора: без Artifact получился бы обычный Task Tracker; без Execution нельзя отслеживать процесс выполнения; без Executor — нельзя запускать работу).
- **Open Questions** — обязательный финальный раздел, даже пустой: явный сигнал «спецификация завершена полностью» либо «есть вопросы». Становится последним разделом, Decision Log — предпоследним (уточнение порядка относительно [предыдущего решения](2026-07-20-domain-specification-review.md), где Decision Log был «в конце»: теперь конец документа — за Open Questions, как более сильный сигнал незавершённости).

### Решение 3 — сознательный темп работы

Явное указание архитектора: на этапе EPIC-003 приоритет — точность и обратимость решений, а не скорость. Одна спецификация (например, Artifact) может законно занять несколько PR, если по ходу работы всплывают вопросы о жизненном цикле, инвариантах или связях с другими сущностями — это признак зрелого процесса, не недостаток. Small PRs с явными Open Questions предпочтительнее одного большого PR с самостоятельно принятыми решениями там, где решение принадлежит архитектору.
