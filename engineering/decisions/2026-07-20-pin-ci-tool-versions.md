# Решение: закреплять версии инструментов, устанавливаемых в CI

## Назначение

Фиксирует инженерный урок из BUGFIX-001: `gofumpt@latest` в CI сломал бы каждый следующий PR и push в `main`, начиная с выхода несовместимой версии инструмента.

## Содержание

### Инцидент

golangci-lint был запинован версией (`v2.12.2`) ещё в TASK-013 — осознанно, из-за нестабильности установочного скрипта релиза. gofumpt по недосмотру остался на `@latest`. 20.07.2026 вышла `gofumpt v0.10.0`, требующая `go >= 1.25.0`; наш CI пинует Go 1.24 ([ADR-009](../../docs/adr/ADR-009-toolchain.md)). Шаг «Install gofumpt» начал падать — обнаружено на Dependabot PR #11 (никак не связанном по содержанию с gofumpt), что затруднило диагностику: первая гипотеза (проблема в bump `actions/setup-go`) оказалась неверной.

### Правило

Любой инструмент, устанавливаемый в CI через `go install`, скачивание бинарника или аналогичным способом (не через `go.mod`/`package.json` с lock-файлом), **закрепляется точной версией**. `@latest` / без версии — запрещено.

### Область действия сейчас

- `gofumpt` — запинован (`v0.9.2`, BUGFIX-001).
- `golangci-lint` — уже запинован (`v2.12.2`, TASK-013).
- `npx --yes <pkg>` (markdownlint-cli2, js-yaml, ранее — mermaid-cli) — **не запинованы**, та же категория риска; не исправлено в рамках BUGFIX-001 (не блокировало CI на момент решения) — кандидат для отдельной задачи.

### Как применять

При добавлении нового шага CI, ставящего инструмент за пределами `go.mod`/`package.json`, — версия указывается явно и проверяется на совместимость с зафиксированным toolchain (ADR-009) до коммита.
