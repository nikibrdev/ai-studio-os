# Решение: workflow.Rules реализуется по каноническому state-machine.md без отдельной доменной спецификации

## Назначение

Фиксирует решение архитектора (2026-07-21, этап 2 EPIC-003) о том, что реализация контракта `workflow.Rules` (TASK-039) ведётся напрямую по каноническим документам — [state-machine.md](../../docs/architecture/state-machine.md) (переходы) и [workflow.md](../../docs/architecture/workflow.md) (роли), — без предварительной 20-раздельной спецификации по шаблону [Specification-Domain.md](../../.claude/templates/Specification-Domain.md).

## Содержание

### Контекст

Этап 1 EPIC-003 (Domain Specifications First) сознательно ограничил спецификации пятью модулями — Artifact, Execution, Executor, Task, Project; workflow был явно исключён («их спецификации — по решению архитектора отдельно, после этапа 1»). При этом реализация правил переходов заявлена как часть этапа 2 (PROJECT_HEALTH: «реализация правил — EPIC-003 этап 2»), а сущность Task (TASK-037) уже зависит от контракта `Rules`.

### Решение

`workflow.Rules` реализуется без отдельной доменной спецификации, потому что для этого модуля она уже существует под другим именем: [state-machine.md](../../docs/architecture/state-machine.md) — канонический документ уровня спецификации (единственный источник истины, полная таблица переходов, guard-условия, восемь инвариантов), а таблица ролей — [workflow.md](../../docs/architecture/workflow.md). Написание 20-раздельной спецификации поверх них дублировало бы источник истины — прямое нарушение правила документации «не дублировать содержание других документов — ссылаться на источник истины» (CLAUDE.md).

### Границы решения

1. Решение касается **только** реализации таблицы переходов/ролей (`Machine`), чья предметная область уже канонизирована. Definition/Step (версионируемые определения процессов) остаются контрактами; их реализация потребует либо расширения канона, либо полноценной спецификации — отдельное решение при открытии соответствующей задачи.
2. Это **не прецедент** пропуска Domain Specifications First: для новых сущностей, чья модель не канонизирована архитектурными документами, полная спецификация обязательна, как и раньше.
3. Любое изменение таблицы переходов или ролей — изменение замороженной архитектуры: только через ADR (правило модуля shared распространяется на канон state machine).
