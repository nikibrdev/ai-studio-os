# Решение: обязательная спецификация для каждого доменного пакета

## Назначение

Фиксирует усиление правила «README + Specification» ([2026-07-19-architecture-spec-implementation.md](2026-07-19-architecture-spec-implementation.md)) специально для Domain Layer (v0.3, EPIC-003), введённое архитектором проекта 2026-07-20.

## Содержание

### Решение

Начиная с Domain Layer ни один доменный пакет не переходит к реализации без **утверждённой** спецификации в `docs/specifications/domain/<модуль>.md`, обязательно содержащей:

1. **Назначение** — зачем пакет существует.
2. **Инварианты** — что пакет гарантирует всегда.
3. **Допустимые состояния** — для пакетов с жизненным циклом (task, workflow) — полный перечень состояний и переходов, либо явная отсылка к [state-machine.md](../../docs/architecture/state-machine.md), если состояния уже канонизированы архитектурой.
4. **Входные и выходные события** — что пакет публикует и потребляет.
5. **Acceptance Criteria** — проверяемые критерии приёмки реализации.

Цикл: **Architecture → Specification → Implementation → Review → Merge**. Задача на реализацию доменного пакета без утверждённой спецификации не соответствует Definition of Ready ([CLAUDE.md](../../CLAUDE.md)).

Этот минимум конкретизирован и расширен до двенадцати обязательных разделов решением [Domain Specification Review](2026-07-20-domain-specification-review.md) (2026-07-20, перед TASK-029) — актуальный шаблон для `docs/specifications/domain/` см. там.

### Последствия

- [Specification.md](../../.claude/templates/Specification.md) дополнен разделом «Допустимые состояния».
- [docs/specifications/README.md](../../docs/specifications/README.md) — уточнено правило для Domain Layer.
- Первая задача EPIC-003 (Domain Layer) — не реализация, а написание и утверждение спецификаций всех доменных модулей.
