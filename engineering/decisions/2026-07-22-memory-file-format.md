# Решение: формат файлов memory/ и политика записи Memory System

## Назначение

Фиксирует решение архитектора (2026-07-22, EPIC-007 TASK-058) по двум из трёх открытых вопросов [memory.md](../../docs/architecture/memory.md): структуре каталога `memory/` и политике записи. Третий вопрос (модель эмбеддингов/схема Qdrant) закрыт отдельно — [ADR-018](../../docs/adr/ADR-018-memory-embeddings-and-qdrant-schema.md) (выбор технологии, требовал ADR; формат файла и организационная политика — нет).

## Содержание

### Формат файла

Один файл на запись: `memory/<projectID>/<id>.md`.

```markdown
---
id: <id>
kind: <kind>
source: <source>
recorded_at: <RFC3339>
---

<содержимое записи, произвольный Markdown>
```

Frontmatter покрывает ровно поля `platform.MemoryEntry`, кроме `ProjectID` (уже выражен путём — каталог) и `Content` (тело файла после разделителя, не дублируется во frontmatter). Формат намеренно близок к уже проверенному в этой сессии шаблону авто-памяти (frontmatter + тело) — не копия кода, а повторное использование того же структурного решения, доказавшего простоту чтения и парсинга.

### Путь и изоляция

Один каталог на проект: `memory/<projectID>/`. Изоляция знаний между проектами (memory.md, принцип «Разделение по проектам») обеспечена на уровне файловой системы, не только фильтром в коде — тот же уровень строгости, что и разделение по репозиториям в `docker/execution` (сеть) или по `project_id` в PostgreSQL-таблицах (EPIC-005): изоляция должна быть видна на носителе, а не только подразумеваться контрактом.

### `Kind` — открытый таксон

`Kind` остаётся произвольной строкой, не закрытым перечислением — тот же принцип, что `artifact.Origin` (спецификация Artifact, EPIC-003): точный список категорий знания не может быть предугадан до появления реальных примеров записи. Примеры (не исчерпывающий список): `decision`, `fact`, `lesson`, `execution-note`.

### Политика записи

`Record`/`Search` — операции, вызываемые явно кодом, который ими пользуется; сам `internal/infrastructure/memory` не решает, когда его вызывать. Автоматическая запись знаний из существующих application-сервисов (`TaskPlanningService` и т.д.) или из будущего Orchestrator/агентских процессов — решение уровня v0.9+ (Multi Agent), не этого эпика: реализуется работающая возможность, не автоматическое поведение.

### Политика устаревания — НЕ решена

Сознательно оставлена открытым вопросом (memory.md продолжает её называть нерешённой после этого эпика): `Record` только добавляет записи, автоматической очистки/устаревания нет. Причина: без реального опыта эксплуатации (какого рода знания накапливаются, с какой скоростью, что оказывается «протухшим») любое правило устаревания было бы придумано, а не выведено из потребности — то же обоснование, что и решение не строить Orchestrator раньше времени (EPIC-006).
