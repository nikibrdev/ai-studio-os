# Решение: порядок реализации v0.8 Dashboard и v0.9 API — сначала API

## Назначение

Фиксирует решение архитектора (2026-07-22) о порядке реализации, отличном от нумерации версий в [ROADMAP.md](../../ROADMAP.md).

## Содержание

### Проблема

ROADMAP.md ставит v0.8 Dashboard раньше v0.9 API по номеру версии. Но [module-boundaries.md](../../docs/architecture/module-boundaries.md) и [ADR-003](../../docs/adr/ADR-003-api-protocol.md) фиксируют жёсткое ограничение: `apps/dashboard` разрешено обращаться к платформе **только** через `apps/api` — прямой доступ к `internal/`, хранилищам или шине событий запрещён. Открытие эпика Dashboard раньше API оставило бы Dashboard без единого источника реальных данных.

### Решение

Порядок **реализации** — API, затем Dashboard: следующий эпик после EPIC-007 — [EPIC-008](../../docs/roadmap/EPIC-008-api-layer.md), реализующий v0.9 API; эпик Dashboard (v0.8) открывается после него.

**Нумерация версий в ROADMAP.md не меняется** — v0.8 остаётся названием Dashboard, v0.9 — названием API, потому что эти номера уже использованы во множестве ссылок, документов и записей CHANGELOG этой сессии; переименование версий создало бы больше путаницы, чем несовпадение номера с порядком сборки. ROADMAP.md явно документирует это несовпадение в обоих разделах (v0.8 и v0.9), чтобы читатель не считал его ошибкой.

### Почему не тонкий мок-слой в Dashboard

Альтернатива — открыть Dashboard первым с временным мок-слоем данных, реальный `apps/api` — отдельным эпиком позже. Отклонено: временный мок-слой в Dashboard — код, который придётся выбросить, и риск того, что мок переживёт своё назначение (тот же класс риска, что и решение не строить Orchestrator раньше времени, EPIC-006/EPIC-007) — учитывая, что API и Dashboard в любом случае должны появиться один за другим в этой же сессии, экономии от параллельной работы здесь нет.
