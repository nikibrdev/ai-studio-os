# Changelog

## Содержание

Типы изменений:

- **Added** — новая функциональность.
- **Changed** — изменения существующей функциональности.
- **Deprecated** — функциональность, которая будет удалена.
- **Removed** — удалённая функциональность.

- **Этап 1 EPIC-003 (Domain Specifications First) закрыт (2026-07-21, TASK-030…033)** — спецификации Execution, Executor, Task и Project утверждены (`docs/specifications/domain/{execution,executor,task,project}.md`), каждая тремя PR с обязательным Delta Review против уже утверждённых. Final Architecture Review внёс по одному содержательному изменению в каждую: Execution — гонка команд Fail/Aborted разрешена порядком выполнения (Behavioral Invariant); Executor — добавлен прямой переход Registered → Retired; Task — расхождения с уже принятым `contracts.go` (`epicID`, запись scope/критериев приёмки) признаны целевой моделью, контракт целенаправленно расширяется на этапе 2, а не наоборот; Project — гипотеза о неявном триггере Created → Active заменена явной командой Activate с проверяемым guard-условием, отсутствие отключения Repository формализовано как сознательное ограничение v1. Все пять спецификаций Domain Layer теперь утверждены; этап 2 (реализация на Go) открывается отдельными задачами.
- **EPIC-003 Domain Layer закрыт целиком — v0.3 завершён (2026-07-21, TASK-034…039, PR #42–#47)** — реализованы все пять доменных сущностей строго по утверждённым спецификациям (единый стиль: сущность + методы-команды, возвращающие события значениями; sentinel-ошибки; ссылки между модулями — идентификаторы-строки по ADR-015; без Commands/Queries-интерфейсов до появления потребителя): `internal/domain/artifact` (Draft → Published → Archived, immutability после Publish), `execution` (гонка Fail/Abort — порядком команд), `executor` (реестр бэкендов, набор Role никогда не пуст), `task` (сущность + расширение контракта `epicID`/SetScope/SetAcceptanceCriteria; правки только в Backlog; допустимость переходов решает исключительно `workflow.Rules`), `project` (явная команда Activate с guard «≥1 Repository»). Каноническая state machine реализована (`workflow.Machine`, ровно 20 переходов, исчерпывающий перебор 81 пары, 100% покрытия) по решению [2026-07-21-workflow-rules-canonical-source.md](engineering/decisions/2026-07-21-workflow-rules-canonical-source.md) — без дублирующей спецификации, state-machine.md признан каноническим источником уровня спецификации. Итог подтверждён сквозным сценарием слоя (`internal/domain/goldenpath_test.go`): Task проходит все девять канонических состояний через реальную Machine, порождая Execution и опубликованный Artifact; всего 82 unit-теста Domain Layer.
- **Три архитектурных блокера сняты — ADR-011, ADR-008, ADR-006 приняты (2026-07-21, PR #49/#50/#51)**: идентификаторы `TASK-NNN`/`EPIC-NNN` последовательные в рамках Project с суррогатным ключом в БД (существующая практика легализована без миграции); git-политики — merge commit, слияние **после** Testing (порядок TestsPassed → MergeCompleted → TaskCompleted), 1 обязательный ревьюер (агент допустим); среда агентов — Docker-контейнер на Execution (сеть по allowlist, короткоживущие секреты), до v0.6 — только локально под надзором человека. Принято 12 ADR, открыто 4 (007/010/012/013), ни один не блокирует v0.4–v0.6.
- **EPIC-004 Application Layer закрыт целиком — v0.4 завершён (2026-07-21, TASK-040…045, PR #53–#57)** — реализованы четыре use-case-сервиса золотого пути поверх Domain Layer: `TaskPlanningService` (постановка задачи), `WorkService` (запуск работы — порождение Execution), `ResultService` (производство результата — Artifact + завершение Execution), `CompletionService` (Review → Testing → Done). ADR-008 выражен кодом, не только текстом: `CompleteTesting` публикует `TestsPassed` → вызывает `RepositoryProvider.MergePullRequest` → публикует `MergeCompleted` → и только тогда переводит задачу в Done с `TaskCompleted`; отказ merge — код-гейт, задача остаётся в Testing. `TaskProjection` строит читаемое состояние задачи только из событий (ADR-014) и доказанно пересобираема с нуля. Попутно закрыты два пробела: каталог `internal/domain/event` не включал 16 событий Artifact/Execution/Executor/Project из утверждённых спецификаций EPIC-003 (добавлены, `events.md` синхронизирован); `platform.Event` не оставлял места для данных, специфичных для события (`Envelope.WithData`, метод сверх контракта, не изменяющий его). Сквозной тест приложения (`internal/application/e2e_test.go`) проводит задачу через весь golden path на in-memory адаптерах, включая ветки «changes requested» и «tests failed». Обнаружено и задокументировано (не устранено — решение для EPIC-005) отсутствие межагрегатной транзакции при последовательном сохранении нескольких агрегатов в одном use-case. 136 unit-тестов во всём проекте.
- **EPIC-005 Infrastructure Layer закрыт целиком — v0.5 завершён (2026-07-21, ADR-017, TASK-046…051, PR #59–#65)** — те же use-case'ы EPIC-004, без единой строки изменений в `internal/application`/`internal/domain`, впервые работают на реальной инфраструктуре вместо in-memory фейков. ADR-017 (принят): драйвер PostgreSQL — `pgx/v5`, миграции — самописный раннер по встроенным `.sql` (без внешней библиотеки, тот же принцип, что и отказ от UUID-зависимости в EPIC-004). Реализовано: пять Postgres-адаптеров хранения (`ProjectStore`…`ArtifactStore`, upsert-семантика, `application.ErrNotFound`); производственный `EventBus` (та же семантика доставки, что и тестовый фейк EPIC-004) с журналом в PostgreSQL — `Publish` журналирует прежде доставки, `eventbus.ReadJournal` восстанавливает события из журнала для перестроения проекций (назначение журнала из ADR-002/event-model.md доведено до работающего кода); `RepositoryProvider` — адаптер GitHub REST API напрямую на `net/http` (без клиентской библиотеки), с двумя явно задокументированными решениями по неоднозначности контракта (целевая ветка PR всегда `main`; `RequestReview` — комментарий к PR, а не назначение GitHub-ревьюера, поскольку ADR-008 возлагает обязательность ревью на workflow, а не на GitHub); composition root (`internal/infrastructure/wiring`) собирает все адаптеры в одну систему. Golden path подтверждён интеграционным тестом на настоящем PostgreSQL (`TestGoldenPath_Infrastructure`) — три прогона подряд зелёные; новый CI-job `integration` с сервис-контейнером PostgreSQL, не блокирующий обязательный статус-чек `verify`. Попутно обнаружены и исправлены два пробела: раннер миграций был уязвим к гонке при конкурентном вызове `Migrate` (исправлено advisory lock'ом на выделенном соединении); пять доменных агрегатов не давали способа реконструкции из персистентных данных без бизнес-команд (добавлен `Restore(...)` в `project`/`task`/`executor`/`execution`/`artifact`, чистая функция без бизнес-правил и событий). Принятый открытый риск: `RepositoryProvider` не проверен вживую на реальном GitHub (нет тестового репозитория и токена в этой сессии) — покрыт unit-тестами на HTTP-уровне (89.6%), golden-path тест использует in-memory фейк для этой части с явной пометкой. 170 unit-тестов во всём проекте.
- **EPIC-006 AI Agent Runtime закрыт целиком — v0.6 завершён (2026-07-22, TASK-052…057, PR #67–#72)** — целевая модель исполнения из ADR-006 реализована и подтверждена реальным прогоном: платформа впервые запускает исполнителя сама, без человека за терминалом на шаге выполнения. `platform.ExecutorTask`/`Artifact`/`ExecutionStatus` конкретизированы (плоские структуры вместо `any`, `internal/platform` остался домен-агностичным — ADR-015). Docker-образ исполнения (`docker/execution`) — на базе Dev Container (`golang:1.24-bookworm`), с Claude Code CLI (Node 22.x из NodeSource — Debian-пакет не удовлетворяет требованию CLI `>=22`, обнаружено сборкой). `agents/claude-code/container.Manager` управляет контейнером Execution через `docker` CLI напрямую (`agents/` не может импортировать `internal/infrastructure`): сетевой allowlist — две Docker-сети и Squid-прокси между ними (контейнер исполнения физически не имеет иного пути наружу), секреты — только переменные окружения, токен читается `git`'ом через `GIT_ASKPASS`, не встраивается в URL/argv. `agents/claude-code.Executor` реализует `platform.Executor` (Accept/Artifacts/Status/Finish) поверх этого жизненного цикла. Демонстрационный реальный прогон (TASK-056) вскрыл и позволил исправить два реальных дефекта: контейнер не оставлял окна для `docker exec` между завершением команды и собственным выходом (исправлено — код возврата в файл, контейнер простаивает после команды вместо немедленного завершения); `docker exec` не указывал рабочую директорию, попадая не в клон, а в `WORKDIR` образа. После исправлений — стабильный честный прогон дважды подряд: реальный контейнер, реальный клон `nikibrdev/ai-studio-os`, реальная блокировка неразрешённого хоста, реальный сбор 20 артефактов-коммитов, чистое освобождение ресурсов. Принятый открытый предел: сам вызов AI-провайдера не проверен — в этой сессии нет отдельного `ANTHROPIC_API_KEY`, и извлечение/переиспользование учётных данных сессии для этой цели не рассматривалось (пользователь явно выбрал зафиксировать честный предел вместо предоставления ключа); адаптер уже готов принять реальный ключ без доработок кода. Auto-подбор исполнителя (Orchestrator) и Tool Layer — сознательно вне scope (ADR-007 Decision Required только для ролей PM/QA, не Developer; Tool Layer — v0.8). 202 unit-теста во всём проекте.
- **EPIC-007 Memory System закрыт целиком — v0.7 завершён (2026-07-22, ADR-018, TASK-058…063, PR #75–#78)** — уже принятый контракт `platform.MemoryProvider` реализован целиком, без изменения самого контракта: файлы (`memory/<projectID>/<id>.md`, frontmatter + Markdown, [decision-документ](engineering/decisions/2026-07-22-memory-file-format.md)) — durable источник истины; Qdrant — производный, перестраиваемый индекс (тот же принцип, что журнал событий и проекции чтения, EPIC-004/005). ADR-018 (принят): эмбеддинг — наивный детерминированный feature hashing (256 измерений, `hash/fnv`+`math`, без внешних зависимостей и секретов — у сессии нет ключа ни к одному провайдеру эмбеддингов, то же решение, что отказ от реального AI-вызова в EPIC-006); коллекция Qdrant `memory_entries` общая на все проекты, `project_id` — поле payload, а не отдельная коллекция (по аналогии с PostgreSQL-таблицами EPIC-005); payload самодостаточен для реконструкции записи без обращения к файлам. `internal/infrastructure/memory.Provider.Record` пишет файл, затем индексирует в Qdrant — в этом порядке, чтобы сбой Qdrant не терял данные; `Reindex` восстанавливает индекс проекта из файлов с нуля. Собран в `wiring.System.Memory` — best-effort по аналогии с `Repository` при отсутствии `GITHUB_TOKEN` (пустой адрес Qdrant оставляет `Memory` равным `nil`, а не ошибкой). Реально проверено вживую на настоящем Qdrant (не только на фейках): запись, поиск, изоляция между проектами (поиск в одном проекте не возвращает записи другого) и восстановление после симулированной потери индекса; CI-job `integration` дополнен сервис-контейнером Qdrant. Принятое честное ограничение MVP: поиск не семантический по сути — совпадение хешированных токенов, а не смысла; замена на реальную модель эмбеддингов при появлении провайдера/ключа — локализованное изменение одной функции, предмет отдельного будущего ADR. Политика устаревания и очистки памяти сознательно не решена — `Record` только добавляет записи, без опыта реальной эксплуатации любое правило было бы придумано, а не выведено из потребности. 253 unit-теста во всём проекте.
- **EPIC-008 API Layer закрыт целиком — v0.9 завершён (2026-07-22, ADR-012, TASK-064…071, BUGFIX-003, PR #80–#87)** — `apps/api` реализует весь golden path через REST (ADR-003) поверх уже готового `internal/application` (EPIC-004): 15 операций по четырём ресурсам (Projects/Tasks/Artifacts/Executions), написанных Documentation First (`docs/api/`), до реализации хендлеров. Порядок реализации — API раньше Dashboard, вопреки номеру версии: `apps/dashboard` структурно зависит от `apps/api` (module-boundaries.md), [decision-документ](engineering/decisions/2026-07-22-api-before-dashboard-build-order.md). ADR-012 принят (Вариант 1): аутентификация отложена — доверенная однопользовательская установка, пересмотр при появлении внешнего потребителя. Два точечных, явно обоснованных дополнения к уже закрытым эпикам: `ProjectService` (Application Layer, EPIC-004) — без него API не мог бы создавать проекты; генерация последовательного `TASK-NNN` на проект (Infrastructure Layer, EPIC-005) — ADR-011 требовал этого с v0.5, но реализация выдачи номера отсутствовала. **BUGFIX-003** (обнаружен живой HTTP-проверкой TASK-069, исправлен до закрытия эпика): публичный `TASK-NNN` уникален только в рамках Project (ADR-011), но `tasks.id` был единственным глобальным `PRIMARY KEY` — два разных проекта с одинаковым `TASK-001` молча портили данные друг друга через `ON CONFLICT (id) DO UPDATE`; исправлено составным ключом `(project_id, id)` по всему стеку (PostgreSQL → шесть методов Application Layer → `TaskProjection` → маршруты `apps/api`, вложенные под `/projects/{projectId}/tasks/...`), воспроизведено и подтверждено устранённым на реальном PostgreSQL. Реально проверено вживую: сквозной HTTP-тест (`TestGoldenPath_HTTP`, три прогона подряд) проводит задачу через весь путь на настоящем PostgreSQL, включая обе ветки ревью (одобрено/отклонено) и тестирования (успех/провал); `RepositoryProvider` — тот же фейк EPIC-004, что уже использовался в `TestGoldenPath_Infrastructure` (EPIC-005) — токена GitHub нет во всех окружениях, где это выполняется. Честные ограничения: без списковых операций сверх `TaskProjection.Get` по одной записи — решение по реальной потребности будущего Dashboard, не здесь. 285 unit-тестов во всём проекте.
- **EPIC-009 Dashboard закрыт целиком — v0.8 завершён (2026-07-23, ADR-009 дополнен, TASK-072…078, PR #90–#95)** — `apps/dashboard` (Next.js 15, TypeScript, App Router) показывает список проектов → задачи проекта → детали задачи (заголовок, тип, scope, критерии приёмки, состояние, время обновления) через реальные HTTP-запросы к `apps/api` — первая версия сознательно read-only (формы создания/действия отложены до реальной потребности), без аутентификации (ADR-012) и без канала реального времени (ADR-003 предвидел этот вопрос). Порядок реализации — вопреки номеру версии: строился после v0.9 API, поскольку `apps/dashboard` структурно зависит от `apps/api`. Frontend-толчейн зафиксирован дополнением [ADR-009](docs/adr/ADR-009-toolchain.md): Node.js 22 LTS, TypeScript, ESLint+Prettier, Vitest+React Testing Library, pnpm (версия закреплена через `packageManager` в `package.json`, тот же принцип, что и у Go-инструментов). По ходу эпика закрыты два реальных пробела API, обнаруженных при проектировании, а не изобретённых заранее: `apps/api` не умел выдавать списки — добавлены `GET /projects`/`GET /projects/{id}/tasks` (`ProjectStore.List`, `TaskProjection.ListByProject`, TASK-072); `TaskProjection` несла только состояние задачи, не её описание — расширена title/type/scope/acceptanceCriteria через `Envelope.WithData` на `TaskCreated` (тот же механизм, что уже применялся для `ReviewCompleted`, TASK-076) — `apps/api` по-прежнему не читает `TaskStore` напрямую (ADR-014). CI: `apps/dashboard` (`pnpm lint`/`format:check`/`test`/`build`) проверяется в том же обязательном статус-чеке `verify`, что и Go-код (TASK-077) — по ходу найдены и исправлены два реальных дефекта конфигурации (`pnpm/action-setup` требует явную версию pnpm; ищет `package.json` в корне репозитория, которого нет в Go-монорепозитории), и негативный сценарий (намеренно испорченный тест) подтверждён реальным красным прогоном перед закрытием задачи. Реально проверено вживую (Playwright, headless Chromium, на настоящем `apps/api`+PostgreSQL): полный путь навигации, кириллица отображается верно, недоступный API и несуществующие проект/задача дают читаемую ошибку вместо падения страницы. 298 unit-тестов Go + 8 unit-тестов `apps/dashboard` во всём проекте.
