# BUGFIX-001: Закрепить версию gofumpt в CI

## Тип

fix

## Эпик

Вне эпика — срочное исправление, обнаружено при обработке Dependabot PR #11 (следствие TASK-019).

## Цель

CI не падает на шаге установки gofumpt; версия закреплена так же, как golangci-lint (v2.12.2), а не `@latest`.

## Контекст

Dependabot PR #11 (`actions/setup-go` 5→7) упал на шаге «Install gofumpt», а не на самом bump: `go install mvdan.cc/gofumpt@latest` резолвнулся в свежевышедшую `v0.10.0`, которой нужен `go >= 1.25.0`; наш workflow пинует Go 1.24 ([ADR-009](../../docs/adr/ADR-009-toolchain.md), принят). `GOTOOLCHAIN=local` не подтягивает 1.25 автоматически — сборка падает.

Это не проблема самого bump-а: любой следующий PR или push в `main` тоже упал бы на этом шаге, начиная с момента выхода gofumpt v0.10.0 — golangci-lint уже был запинован по этой же причине (TASK-013), а gofumpt — по недосмотру оставлен на `@latest`. Меняю Go на 1.25 не стал: это означало бы негласно отступить от принятого ADR-009, а версия tooling — не архитектурное решение и должна решаться независимо от языка проекта.

## Scope

### Входит

- `.github/workflows/verify.yml`: `gofumpt@latest` → `gofumpt@v0.9.2` (последняя версия с `go 1.24.0` в go.mod — совместима с ADR-009).

### Не входит

- Изменение версии Go (ADR-009 не трогается).

## Критерии приёмки

- [x] `gofumpt@v0.9.2` устанавливается и работает локально (`go1.24.5`, 0 issues на репозитории).
- [ ] CI этого PR — зелёный (живой тест исправления).
- [ ] После merge — Dependabot PR #11 (setup-go bump) перепрогоняется и тоже проходит (тот же workflow-файл).

## План реализации

Проверить доступные версии gofumpt через Go module proxy, найти последнюю с `go <= 1.24` в go.mod, запинить в workflow, проверить локально.
