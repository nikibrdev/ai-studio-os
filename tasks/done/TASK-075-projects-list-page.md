## План реализации

1. `src/app/page.tsx` — асинхронный Server Component: `listProjects()`, рендер списка (имя, состояние, ссылка на `/projects/{id}`) или сообщения «Проектов пока нет» для пустого списка.
2. `src/app/loading.tsx` — автоматический fallback Next.js (Suspense) на время запроса.
3. `src/app/error.tsx` — Error Boundary Next.js (Client Component): читаемое сообщение + кнопка «Попробовать снова» (`reset()`), перехватывает исключение `ApiError`/сетевую ошибку из `listProjects`.
4. `export const dynamic = "force-dynamic"` на странице — **обнаружено при первой же сборке**: без этого `next build` пытается статически пререндерить `/` и намертво зависает на `fetch` к недоступному во время сборки `apps/api`; данные всегда актуальны только на момент запроса, статическая генерация здесь в принципе неприменима.
5. `page.test.tsx` — `vi.mock("@/lib/api")`, рендер `await HomePage()` напрямую (простой случай без динамических API Next.js вроде `cookies()`/`redirect()` тестируется вызовом асинхронной функции страницы и передачей результата в `render()`): проверка ссылок по каждому проекту и сообщения для пустого списка.
6. Живая проверка вживую: `docker compose up -d postgres`, `go run ./apps/api`, создание двух реальных проектов через `curl`, `pnpm dev` — Playwright-скриншот показывает оба проекта с верными состояниями, `console --errors` пуст; затем API остановлен — `error.tsx` показывает читаемое сообщение с кнопкой повтора.
7. `make verify` (корень репозитория) — чисто.
