# TASK-054: Жизненный цикл контейнера Execution

## Тип

feature

## Эпик

[EPIC-006 AI Agent Runtime](../../docs/roadmap/EPIC-006-ai-agent-runtime.md)

## Цель

Реализовать управление жизненным циклом контейнера Execution по ADR-006: старт контейнера из образа TASK-053 с рабочей копией ветки задачи (клон при старте), сетевым allowlist (AI-провайдер + git-хостинг), короткоживущими секретами через переменные окружения; уничтожение контейнера и рабочей копии по завершении (успех, неудача или Abort).

## Контекст

Это компонент ниже уровня контракта `Executor` — сама механика запуска/остановки контейнера через Docker (скорее всего `docker run`/Docker SDK или простой exec-обёртка над Docker CLI — решение о способе управления Docker фиксируется в плане). `agents/claude-code` (TASK-055) использует этот компонент, не дублирует его логику.

## Scope

### Входит

- Компонент управления жизненным циклом: старт (клон ветки в рабочую директорию контейнера, сетевой allowlist, инъекция секретов через env), опрос состояния, остановка/уничтожение (контейнер + временная рабочая копия).
- Сетевой allowlist — конкретный механизм (Docker network с ограничениями, или прокси, или firewall-правила внутри контейнера) фиксируется и обосновывается в плане.
- Секреты — короткоживущий git-токен и ключ AI-провайдера передаются только через переменные окружения запускаемого процесса, не сохраняются в образе или в каких-либо файлах, которые переживают контейнер.
- Тесты того, что тестируемо без реального Docker (конструирование команд/конфигурации, обработка ошибок) — реальный запуск контейнера тестируется вручную/в TASK-056.

### Не входит

- Содержимое образа (TASK-053).
- Сам вызов Claude Code внутри контейнера и парсинг его вывода (TASK-055).
- Массовый параллелизм — один контейнер на Execution, без пула/масштабирования.

## План реализации

1. **Расположение пакета:** `agents/claude-code/container`, не `internal/infrastructure` — `module-boundaries.md` прямо запрещает `agents/` импортировать внутренности Core (включая `internal/infrastructure`); разрешён только опубликованный контракт `Executor`, `pkg/` и SDK/CLI своего провайдера. Docker CLI для `agents/` — «SDK/CLI своего провайдера» в широком смысле (среда исполнения, не платформенная инфраструктура) — управляется напрямую через `os/exec`, без клиентской библиотеки Docker (та же причина, что ADR-017 и GitHub-адаптер: горстка команд не оправдывает новую зависимость).
2. **Механизм allowlist:** две Docker-сети — внутренняя (`--internal`, без интернета) и `bridge` (с интернетом). Прокси-контейнер (`ubuntu/squid` — существующий публичный образ) подключён к обеим, разрешает CONNECT/HTTP только к доменам allowlist'а (`dstdomain` ACL — не требует расшифровки TLS, так как Squid видит только целевой хост CONNECT-запроса, не сертификат). Контейнер исполнения подключён только к внутренней сети — физически не имеет иного пути наружу.
3. **Секреты:** `GIT_TOKEN`/`ANTHROPIC_API_KEY` — только переменные окружения контейнера. Токен не встраивается в URL клонирования и не передаётся флагом (оба способа делают секрет видимым в списке процессов контейнера) — вместо этого генерируется `GIT_ASKPASS`-скрипт, который сам читает токен из переменной окружения в момент, когда `git` его запрашивает.
4. `commandRunner` — узкий интерфейс (`Run(ctx, name, args...) (string, error)`), сужающий `os/exec` до заменяемого в тестах контракта (тот же паттерн, что `execer` в `internal/infrastructure/eventbus`, TASK-049).
5. `Manager{Start, Status, Exec, Stop}` — оркестрация сети → прокси → контейнера исполнения; откат частично созданных ресурсов при ошибке любого шага.
6. Юнит-тесты на фейковом `commandRunner`: построение команд, что контейнер исполнения никогда не подключается ко второй (публичной) сети напрямую, что секреты передаются через `-e`, а не встраиваются в URL, откат при ошибке сети/прокси, идемпотентность `Stop`.
7. Прямые тесты `renderSquidConf`/`cloneAndRunScript`/`shellQuote` — эти функции определяют реальную безопасность (что именно разрешено в сети, куда уходит секрет), их корректность не должна проверяться только косвенно через `Start`.
8. Интеграционный тест (тег `integration`, за `TEST_DOCKER` — тот же паттерн, что `TEST_DATABASE_URL`, тест создаёт настоящие Docker-ресурсы): реальное клонирование `nikibrdev/ai-studio-os`, `curl` к `api.github.com` (в allowlist) и к `example.com` (не в allowlist) — доказывает, что блокировка настоящая, не только структура команд правильная.
9. README `agents/README.md` (новый, каталог был пуст — только `.gitkeep`) и `agents/claude-code/README.md` (новый).
