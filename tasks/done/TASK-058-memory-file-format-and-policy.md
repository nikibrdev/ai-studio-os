# TASK-058: Формат файлов memory/ и decision-документ по политике записи

## Тип

docs

## Эпик

[EPIC-007 Memory System](../../docs/roadmap/EPIC-007-memory-system.md)

## Цель

Зафиксировать формат файлов `memory/<projectID>/<id>.md` (frontmatter + содержимое) и решение по политике записи/устаревания — оба вопроса [memory.md](../../docs/architecture/memory.md) называет открытыми и требующими решения на этом этапе.

## Контекст

[ADR-018](../../docs/adr/ADR-018-memory-embeddings-and-qdrant-schema.md) (принят при открытии эпика) уже закрыл вопрос эмбеддингов/схемы Qdrant. Эта задача закрывает оставшиеся два вопроса memory.md, не требующих ADR (формат на диске и организационная политика — не выбор технологии): решение фиксируется как `engineering/decisions/` документ, а не ADR, по аналогии с `2026-07-21-application-ports-placement.md`.

## Scope

### Входит

- Формат файла: YAML-подобный frontmatter (`id`, `kind`, `source`, `recorded_at`) + Markdown-содержимое после разделителя — по образцу, уже проверенному в этой сессии (собственная система авто-памяти использует тот же принцип: frontmatter + тело).
- Путь: `memory/<projectID>/<id>.md` — один каталог на проект (изоляция на уровне файловой системы, не только в коде).
- `Kind` — открытый строковый таксон (не закрытый enum) — по тому же принципу, что `artifact.Origin` (спецификация Artifact): примеры значений документируются, но список не фиксирован закрыто.
- Политика записи: `Record`/`Search` — явно вызываемые операции; автоматический вызов из существующих сервисов или будущего Orchestrator — вне scope этого эпика (см. Scope EPIC-007).
- Политика устаревания: сознательно не решается — `Record` только добавляет; зафиксировать как открытый вопрос, не решение.
- `engineering/decisions/2026-07-22-memory-file-format.md`.

### Не входит

- Сама реализация чтения/записи файлов (TASK-059).
- Модель эмбеддингов/Qdrant (уже решено — ADR-018).
