# TASK-054: Жизненный цикл контейнера Execution

## Тип

feature

## Эпик

[EPIC-006 AI Agent Runtime](../../docs/roadmap/EPIC-006-ai-agent-runtime.md)

## Цель

Реализовать управление жизненным циклом контейнера Execution по ADR-006: старт контейнера из образа TASK-053 с рабочей копией ветки задачи (клон при старте), сетевым allowlist (AI-провайдер + git-хостинг), короткоживущими секретами через переменные окружения; уничтожение контейнера и рабочей копии по завершении (успех, неудача или Abort).

## Контекст

Это компонент ниже уровня контракта `Executor` — сама механика запуска/остановки контейнера через Docker (скорее всего `docker run`/Docker SDK или простой exec-обёртка над Docker CLI — решение о способе управления Docker фиксируется в плане). `agents/claude-code` (TASK-055) использует этот компонент, не дублирует его логику.

## Scope

### Входит

- Компонент управления жизненным циклом: старт (клон ветки в рабочую директорию контейнера, сетевой allowlist, инъекция секретов через env), опрос состояния, остановка/уничтожение (контейнер + временная рабочая копия).
- Сетевой allowlist — конкретный механизм (Docker network с ограничениями, или прокси, или firewall-правила внутри контейнера) фиксируется и обосновывается в плане.
- Секреты — короткоживущий git-токен и ключ AI-провайдера передаются только через переменные окружения запускаемого процесса, не сохраняются в образе или в каких-либо файлах, которые переживают контейнер.
- Тесты того, что тестируемо без реального Docker (конструирование команд/конфигурации, обработка ошибок) — реальный запуск контейнера тестируется вручную/в TASK-056.

### Не входит

- Содержимое образа (TASK-053).
- Сам вызов Claude Code внутри контейнера и парсинг его вывода (TASK-055).
- Массовый параллелизм — один контейнер на Execution, без пула/масштабирования.

## Критерии приёмки

- [ ] Компонент запускает контейнер с рабочей копией указанной ветки, останавливает и удаляет его (включая рабочую копию) по завершении.
- [ ] Сетевой allowlist реализован и задокументирован (конкретный механизм — по решению из плана).
- [ ] Секреты передаются только через переменные окружения контейнера; не логируются, не сохраняются на диске за пределами времени жизни контейнера.
- [ ] Юнит-тесты покрывают то, что не требует реального Docker; ручная проверка реального запуска задокументирована в отчёте.
- [ ] `make verify` — чисто.
