# TASK-081: Каркас apps/orchestrator, бутстрап Developer-исполнителя, цикл опроса

## Тип

feature

## Эпик

[EPIC-010 Orchestrator](../../docs/roadmap/EPIC-010-orchestrator.md)

## Цель

Создать `apps/orchestrator` как отдельный запускаемый процесс: сборка зависимостей через `wiring.System` (по образцу `apps/api/main.go`), идемпотентная регистрация одного Developer-исполнителя при старте, цикл опроса `EventJournal.Since` с курсором в памяти. Без диспетчеризации событий — она в TASK-082/083; эта задача проверяется тем, что процесс стартует, видит собственный курсор и не падает при отсутствии новых событий.

## Контекст

`apps/orchestrator` сейчас — пустой каталог с `.gitkeep` (зарезервирован ещё в EPIC-002). Конфигурация (образ контейнера, GitHub-токен, ключ AI-провайдера, репозиторий проекта) — переменные окружения, тот же принцип, что `apps/api/main.go` (DSN, порт).

## Scope

### Входит

- `apps/orchestrator/main.go` — сборка `wiring.System` (DSN из `DATABASE_URL`, опционально `QDRANT_URL` — как `apps/api`), чтение конфигурации Orchestrator'а из переменных окружения (образ контейнера, `GIT_TOKEN`, AI-провайдер-ключ, репозиторий по умолчанию), запуск цикла опроса.
- Бутстрап: при старте — `ExecutorStore.List`, если Active Developer-исполнителя нет — `ExecutorService.Register` + `Activate` с фиксированным ID/backend (`agents/claude-code`, роль `developer`); идемпотентно — повторный запуск не создаёт дубликат.
- Цикл опроса: курсор в памяти (инициализация — момент запуска), периодический вызов `EventJournal.Since(ctx, cursor)` (интервал — константа, например 5с), обновление курсора по последнему полученному событию; для этой задачи — только логирование полученных событий (обработка `TaskPlanned` — TASK-082).
- Грациозное завершение по сигналу (`SIGINT`/`SIGTERM`), по образцу `apps/api/main.go`, если там уже есть такой обработчик — иначе добавить в обоих местах согласованно не требуется (только `apps/orchestrator` в scope этой задачи).
- `apps/orchestrator/README.md` — пока минимальный (назначение, переменные окружения, запуск); полное описание механизма — TASK-084.

### Не входит

- Обработка `TaskPlanned` (выбор исполнителя, `StartTask`, `Accept`) — TASK-082.
- Слежение за исполнением — TASK-083.
- Docker-образ/деплой Orchestrator'а — не входит в эпик (см. stack.md/toolchain — за пределами этой задачи).

## Критерии приёмки

- [ ] `apps/orchestrator` запускается (`go run ./apps/orchestrator`) против реального PostgreSQL, поднимает `wiring.System`.
- [ ] При первом старте регистрирует и активирует один Developer-исполнитель; при повторном старте — не создаёт дубликат (идемпотентность подтверждена тестом/живой проверкой).
- [ ] Цикл опроса вызывает `EventJournal.Since` с продвигающимся курсором, не пропускает и не дублирует события в рамках одного запуска.
- [ ] Процесс корректно завершается по сигналу.
- [ ] `make verify` — чисто.
