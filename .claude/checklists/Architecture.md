# Чек-лист: Архитектура

## Назначение

Проверка архитектурного решения или изменения, затрагивающего архитектуру, перед принятием. Используется ролями Architect и Reviewer.

## Содержание

### Соответствие принципам

- [ ] Зависимости направлены внутрь; домен не зависит от инфраструктуры (Clean Architecture).
- [ ] Междоменное взаимодействие — только через события ([event-model.md](../../docs/architecture/event-model.md)).
- [ ] Контракт (интерфейс) определён до реализации (Interface First).
- [ ] Решение не требует изменения ядра для будущих расширений этого типа.
- [ ] Выбрано простейшее из работающих решений (KISS); нет преждевременных абстракций.
- [ ] Нет скрытых соглашений и неявного поведения (минимум магии).

### Оформление

- [ ] Решение зафиксировано в ADR по шаблону; рассмотрено ≥ 2 вариантов.
- [ ] Последствия (включая отрицательные) описаны честно.
- [ ] Затронутые документы перечислены и обновлены (или запланированы к обновлению).
- [ ] Изменения структуры репозитория или Конституции — только через ADR.

### Границы

- [ ] Новые технологии не выходят за утверждённый стек либо обоснованы в ADR.
- [ ] Модуль имеет единственную ответственность; границы модулей не размыты.
- [ ] Публичные контракты версионируются/эволюционируют без слома потребителей, либо слом зафиксирован в ADR.
