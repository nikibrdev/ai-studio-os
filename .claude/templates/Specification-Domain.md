# Шаблон: Specification (Domain Layer) — Domain Specification Review

## Назначение

Обязательный шаблон полной спецификации для доменных модулей (`docs/specifications/domain/<module>.md`), усиливающий базовый [Specification.md](Specification.md) двенадцатью разделами — не «по возможности», а всегда, для каждого модуля EPIC-003 и далее ([Domain Specification Review](../../engineering/decisions/2026-07-20-domain-specification-review.md), решение архитектора 2026-07-20). Спецификация без одного из этих разделов не соответствует Definition of Ready на реализацию.

## Содержание

```markdown
# Спецификация: domain/<module>

## Purpose

<Зачем существует сущность. Какую проблему предметной области она решает.>

## Responsibilities

<За что отвечает. За что не отвечает — кратко; исчерпывающий список смежных
границ идёт в раздел Anti-Responsibilities.>

## Invariants

<Утверждения, которые сущность гарантирует ВСЕГДА. Самая важная часть
спецификации — именно инварианты потом определяют код.>

## Lifecycle

<Состояния и правила переходов между ними. Для модулей, чей жизненный цикл
уже канонизирован архитектурой (например, Task — docs/architecture/state-machine.md),
— явная отсылка вместо дублирования.>

## Relationships

<Не просто список связей с другими сущностями — для каждой явно указать:
кто владеет, кто ссылается, кто создаёт, кто удаляет.>

## Domain Events

<Публикуемые события: имя, когда возникает, какие данные несёт.>

## Commands

<Какие действия над сущностью вообще допустимы.>

## Queries

<Что система должна уметь спрашивать об этой сущности.>

## Acceptance Criteria

- [ ] <Проверяемый критерий полноты САМОЙ СПЕЦИФИКАЦИИ — не реализации.>

## Future Extensions

<Что сознательно оставлено за рамками текущей версии — не список TODO,
а осознанно проведённая граница текущего дизайна.>

## Anti-Responsibilities

<Явный, отдельный от Responsibilities список того, что сущность НЕ делает
и не знает — особенно про смежные модули (например: не запускает
исполнение, не знает Executor, не меняет Task, не содержит бизнес-логику
Workflow).>

## Decision Log

<Ссылки на ADR/решения, на которых основана спецификация — что решено и
каким документом (например: «Почему Aggregate Root — ADR-016»).>

## Статус

<Черновик | Утверждена | Реализована>

## Последнее обновление

YYYY-MM-DD
```
