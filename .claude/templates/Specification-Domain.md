# Шаблон: Specification (Domain Layer) — Domain Specification Review

## Назначение

Обязательный шаблон полной спецификации для доменных модулей (`docs/specifications/domain/<module>.md`), усиливающий базовый [Specification.md](Specification.md) шестнадцатью разделами — не «по возможности», а всегда, для каждого модуля EPIC-003 и далее ([Domain Specification Review](../../engineering/decisions/2026-07-20-domain-specification-review.md); дополнено [Three-Pass Review](../../engineering/decisions/2026-07-20-domain-specification-three-pass-review.md), решения архитектора 2026-07-20). Спецификация без одного из этих разделов не соответствует Definition of Ready на реализацию. Перед запросом ревью спецификация проходит [чек-лист трёх проверок](../checklists/DomainSpecificationReview.md).

## Содержание

```markdown
# Спецификация: domain/<module>

## Purpose

<Зачем существует сущность. Какую проблему предметной области она решает.>

## Responsibilities

<За что отвечает. За что не отвечает — кратко; исчерпывающий список смежных
границ идёт в раздел Anti-Responsibilities.>

## Invariants

<Утверждения, которые сущность гарантирует ВСЕГДА. Самая важная часть
спецификации — именно инварианты потом определяют код.>

## Lifecycle

<Состояния и правила переходов между ними. Для модулей, чей жизненный цикл
уже канонизирован архитектурой (например, Task — docs/architecture/state-machine.md),
— явная отсылка вместо дублирования.>

## Relationships

<Не просто список связей с другими сущностями — для каждой явно указать:
кто владеет, кто ссылается, кто создаёт, кто удаляет.>

## Domain Events

<Публикуемые события: имя, когда возникает, какие данные несёт.>

## Commands

<Какие действия над сущностью вообще допустимы.>

## Queries

<Что система должна уметь спрашивать об этой сущности.>

## Examples

<НЕ код, НЕ JSON, НЕ Go — содержательные примеры конкретных экземпляров
сущности, показывающие, что она действительно описана правильно
(например, для Artifact: Markdown-спецификация, Pull Request, Figma-дизайн,
test report, ADR).>

## Acceptance Criteria

- [ ] <Проверяемый критерий полноты САМОЙ СПЕЦИФИКАЦИИ — не реализации.>

## Future Extensions

<Что сознательно оставлено за рамками текущей версии — не список TODO,
а осознанно проведённая граница текущего дизайна.>

## Anti-Responsibilities

<Явный, отдельный от Responsibilities список того, что сущность НЕ делает
и не знает — особенно про смежные модули (например: не запускает
исполнение, не знает Executor, не меняет Task, не содержит бизнес-логику
Workflow).>

## Non-Goals

<Не путать с Anti-Responsibilities (это про поведение сущности, Non-Goals —
про то, чего сознательно не решает текущая версия спецификации; например,
для Artifact v1: версионирование, диффы, совместное редактирование, ACL,
репликация — не «никогда», а «не решено сейчас»).>

## Removal Test

<Один обязательный вопрос: что произойдёт с системой, если удалить эту
сущность целиком? Ответ должен показывать, что сущность самостоятельна,
а не является атрибутом другой.>

## Decision Log

<Ссылки на ADR/решения, на которых основана спецификация — что решено и
каким документом (например: «Почему Aggregate Root — ADR-016»).>

## Open Questions

<Обязательный финальный раздел, даже если пуст — явный сигнал, что
спецификация либо завершена полностью, либо есть нерешённые вопросы.>

## Статус

<Черновик | Утверждена | Реализована>

## Последнее обновление

YYYY-MM-DD
```
