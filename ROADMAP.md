# Roadmap AI Studio OS

## Назначение

Определяет релизные вехи проекта от фундамента до первого публичного MVP. Каждая веха отражает готовность продукта, а не только объём выполненных задач; содержание уточняется эпиками (`docs/roadmap/EPIC-*.md`) и задачами (`tasks/`). Вехи служат горизонту 1 года из [VISION.md](VISION.md); горизонты 2 и 5 лет там же выходят за рамки этого документа.

## Содержание

### История ревизии

Версия 2026-07-19 (первоначальная) была написана до проектирования архитектуры и описывала стек ролей (Project Management, Developer Engine, QA Engine, Workflow) как последовательные версии. После Architecture Freeze (ADR-002…015, [overview.md](docs/architecture/overview.md)) и завершения EPIC-002.5 стала известна реальная слоистая структура (`domain → application → platform → infrastructure`, [ADR-015](docs/adr/ADR-015-internal-layering.md)). Roadmap пересмотрен 2026-07-20 архитектором проекта, чтобы версии отражали готовность продукта по слоям, а не предположения этапа Foundation. Решение зафиксировано: [engineering/decisions/2026-07-20-release-milestones.md](engineering/decisions/2026-07-20-release-milestones.md).

Прежние роли-версии не исчезли — они распределены по новым вехам: Project Management → часть Domain/Application Layer; Developer Engine и Multi Agent → AI Agent Runtime; Workflow → Domain Layer (`internal/domain/workflow`) и Application Layer (оркестрация).

### Внутренние архитектурные вехи (не версии продукта)

Отдельно от пользовательских версий v0.X — рубежи, отмечающие качественные изменения в самом процессе разработки, а не в готовности продукта.

- **First Reference Domain Specification** (2026-07-20) — [Artifact](docs/specifications/domain/artifact.md) стал первой доменной спецификацией проекта со статусом **Reference**: не просто утверждённой, а официально признанной образцом для всех последующих спецификаций Domain Layer ([решение](engineering/decisions/2026-07-20-domain-specification-reference-status.md)). Момент, когда у проекта впервые появился не только код и документация, но и проверенный, воспроизводимый метод доменного моделирования, на который опираются все следующие сущности.
- **Domain Specifications First — этап 1 закрыт** (2026-07-21) — все пять спецификаций Domain Layer ([Artifact](docs/specifications/domain/artifact.md), [Execution](docs/specifications/domain/execution.md), [Executor](docs/specifications/domain/executor.md), [Task](docs/specifications/domain/task.md), [Project](docs/specifications/domain/project.md)) утверждены архитектором. Каждая закрыта в тот же день, когда взята в работу — метод, откалиброванный на Artifact, воспроизвёлся четыре раза подряд без потери в качестве ревью (реальные раунды Changes Requested/Approve на каждой). Два реальных расхождения со старым кодом EPIC-002 (`internal/domain/task`, `internal/domain/project`) не устранялись правкой кода, а получили явное архитектурное решение — контракты целенаправленно расширяются вслед за спецификацией на этапе 2.

### v0.3 Domain Layer — предметная область — **Завершено** (2026-07-21)

- [EPIC-003](docs/roadmap/EPIC-003-domain-layer.md): реализация доменных модулей, начиная не с `task`, а с порядка **Artifact → Execution → Executor → Task → Project** ([domain-model.md](docs/architecture/domain-model.md), [ADR-016](docs/adr/ADR-016-artifact-aggregate-root.md)).
- Этап 1 — Domain Specifications First: полные спецификации всех пяти модулей утверждены архитектором до единой строки Go (закрыт 2026-07-21).
- Этап 2 — реализация: пять сущностей + каноническая state machine (`workflow.Machine`), TASK-034…039 (закрыт 2026-07-21).

**Результат достигнут:** доменная логика (в т.ч. state machine задачи) работает и покрыта тестами, без внешних зависимостей — подтверждено сквозным сценарием слоя ([internal/domain/goldenpath_test.go](internal/domain/goldenpath_test.go)): Task проходит все девять канонических состояний, порождая Execution и опубликованный Artifact.

### v0.4 Application Layer — сценарии использования — **Завершено** (2026-07-21)

- [EPIC-004](docs/roadmap/EPIC-004-application-layer.md): сценарии команд поверх доменных модулей ([ADR-014](docs/adr/ADR-014-module-interaction.md)) — постановка задачи, запуск работы, производство результата, завершение задачи (merge после Testing, ADR-008); проекция для чтения из событий.

**Результат достигнут:** платформа исполняет use-case'ы, не завязанные на конкретную инфраструктуру — подтверждено сквозным тестом ([internal/application/e2e_test.go](internal/application/e2e_test.go)): вся золотая дорожка на in-memory адаптерах, включая ветки «changes requested» и «tests failed», состояние читается только через проекцию.

### v0.5 Infrastructure Layer — инфраструктура — **Завершено** (2026-07-21)

- [EPIC-005](docs/roadmap/EPIC-005-infrastructure-layer.md): адаптеры портов EPIC-004 к реальным технологиям — PostgreSQL (источник истины задач, [ADR-004](docs/adr/ADR-004-task-storage.md), драйвер `pgx/v5` — [ADR-017](docs/adr/ADR-017-postgresql-driver.md)), производственный In-Memory Event Bus с журналом в PostgreSQL ([ADR-002](docs/adr/ADR-002-event-delivery.md)), GitHub (Repository Provider).

**Результат достигнут:** платформа работает end-to-end на реальных хранилищах и интеграциях — подтверждено интеграционным тестом ([internal/infrastructure/wiring/golden_path_integration_test.go](internal/infrastructure/wiring/golden_path_integration_test.go)): та же золотая дорожка, что и в v0.4, на реальных PostgreSQL-адаптерах и производственном EventBus (с журналом, восстановимым отдельным select'ом), без единой строки изменений в `internal/application`/`internal/domain`. Единственное исключение — `RepositoryProvider`: покрыт unit-тестами против GitHub REST API, но не проверен вживую на реальном репозитории (нет тестового репозитория и токена в этой сессии) — принятый открытый риск, не блокирующий результат.

### v0.6 AI Agent Runtime — исполнение агентов — **Завершено** (2026-07-22)

- [EPIC-006](docs/roadmap/EPIC-006-ai-agent-runtime.md): целевая модель исполнения из [ADR-006](docs/adr/ADR-006-agent-execution-environment.md) (принят) — одно Execution = один Docker-контейнер, сетевой allowlist, короткоживущие секреты; первый реальный адаптер `Executor` ([ADR-005](docs/adr/ADR-005-executor-contract.md), принят) — Claude Code.
- Роль Developer исполняется агентом по процессу из [CLAUDE.md](CLAUDE.md); оформление изменений через GitHub.

**Результат достигнут:** платформа сама запускает исполнителя в изолированном контейнере — подтверждено реальным прогоном полного жизненного цикла `Executor` ([agents/claude-code](agents/claude-code/README.md)): контейнер стартует, клонирует рабочую копию, сетевой доступ ограничен allowlist'ом (проверена реальная блокировка неразрешённого хоста), секреты передаются только переменными окружения, ресурсы освобождаются без утечек. Единственное не пройденное вживую звено — сам ответ AI-провайдера: в этой сессии нет отдельного ключа для вложенного Claude Code, и извлечение/переиспользование учётных данных сессии для этой цели не рассматривалось — принятый честный предел, не блокирующий результат (адаптер уже готов принять реальный ключ без доработок кода).

### v0.7 Memory System — память — **Завершено** (2026-07-22)

- [EPIC-007](docs/roadmap/EPIC-007-memory-system.md): реализация уже принятого контракта `platform.MemoryProvider` — знания проекта (`memory/`), поиск на Qdrant ([ADR-018](docs/adr/ADR-018-memory-embeddings-and-qdrant-schema.md) — наивный локальный эмбеддинг, без внешних зависимостей).

**Результат достигнут:** `internal/infrastructure/memory.Provider` реализует `Record`/`Search`/`Reindex` целиком, собран в `wiring.System.Memory` наравне с остальными адаптерами — подтверждено реальным прогоном на живом Qdrant: запись, поиск, изоляция между проектами (поиск в одном проекте не возвращает записи другого) и восстановление индекса из файлов после симулированной потери (`Reindex`) — не только на фейках, но и в CI (`integration`-job поднимает Qdrant как сервис-контейнер). Честный предел, задокументированный ADR-018: поиск не семантический по сути — наивный feature hashing совпадает по хешированным токенам, а не по смыслу; замена на реальную модель эмбеддингов — локализованное изменение одной функции, предмет будущего ADR при появлении провайдера/ключа. Политика устаревания памяти сознательно не решена — `Record` только добавляет записи.

### v0.8 Dashboard — веб-интерфейс — **Завершено** (2026-07-23)

- [EPIC-009](docs/roadmap/EPIC-009-dashboard.md): Next.js — список проектов, список и состояние задач; первая версия — read-only (наблюдаемость), без форм действий, без auth, без канала реального времени.

**Результат достигнут:** `apps/dashboard` (Next.js 15, TypeScript) показывает список проектов → задачи проекта → детали задачи (заголовок, тип, scope, критерии приёмки, состояние, время обновления) через реальные HTTP-запросы к `apps/api`. По ходу эпика закрыты два реальных пробела, обнаруженных при проектировании (не самовольное расширение scope — без них Dashboard физически не мог показать данные): `apps/api` не умел выдавать списки (`GET /projects`, `GET /projects/{id}/tasks` — TASK-072) и `TaskProjection` не несла заголовок/тип/scope/критерии приёмки задачи, только состояние (расширена через `Envelope.WithData` на `TaskCreated`, тот же механизм, что уже использовался для `ReviewCompleted` — TASK-076). Подтверждено вживую (Playwright, headless Chromium, на реальном `apps/api`+PostgreSQL): полный путь навигации, кириллица отображается верно, недоступный API и несуществующие проект/задача дают читаемую ошибку, а не падение страницы. CI проверяет `apps/dashboard` (`pnpm lint`/`format:check`/`test`/`build`) в том же обязательном статус-чеке `verify`, что и Go-код — подтверждено намеренно красным прогоном, что job действительно проверяет, а не пропускает молча (TASK-077). Честные ограничения: read-only (нет форм создания/действия — намеренно отложены до реальной потребности); без аутентификации (ADR-012, доверенная установка); без канала реального времени (обновление только по запросу/перезагрузке страницы, ADR-003 предвидел этот вопрос).

**Порядок реализации:** строился после v0.9 API, несмотря на номер версии — `apps/dashboard` структурно зависит от `apps/api` ([module-boundaries.md](docs/architecture/module-boundaries.md), [ADR-003](docs/adr/ADR-003-api-protocol.md)); решение и его причина — [2026-07-22-api-before-dashboard-build-order.md](engineering/decisions/2026-07-22-api-before-dashboard-build-order.md). Номер версии не меняется, чтобы не переписывать уже существующие ссылки.

### v0.9 API — публичный интерфейс — **Завершено** (2026-07-22)

- [EPIC-008](docs/roadmap/EPIC-008-api-layer.md): REST API ([ADR-003](docs/adr/ADR-003-api-protocol.md)) — тонкий слой доставки над уже готовым `internal/application` (EPIC-004), без auth в первой версии ([ADR-012](docs/adr/ADR-012-identity-and-auth.md), Вариант 1 — доверенная установка).

**Результат достигнут:** `apps/api` реализует весь golden path через REST — создание/активация проекта, постановка и планирование задачи, запуск работы, черновик/публикация артефакта, результат исполнения, ревью, тестирование, завершение — 15 операций по четырём ресурсам (Projects/Tasks/Artifacts/Executions), написанных Documentation First (`docs/api/`). Подтверждено сквозным HTTP-тестом на реальном PostgreSQL (`TestGoldenPath_HTTP`, TASK-070), включая обе ветки ревью и тестирования. По ходу разработки обнаружен и исправлен реальный баг ([BUGFIX-003](tasks/done/BUGFIX-003-task-project-scoped-key.md)): публичный `TASK-NNN` уникален только в рамках Project (ADR-011), а не глобально — задаче-специфичные маршруты вложены под `/projects/{projectId}/tasks/...`. Честные ограничения: без аутентификации (ADR-012, доверенная установка — пересмотр при появлении внешнего потребителя); `TaskProjection.Get` отдаёт только одну запись по (project, id) — списковых операций нет; реальный вызов `RepositoryProvider` не проверен (тот же класс ограничения, что и в EPIC-005/006 — нет токена GitHub в этой сессии). Dashboard (v0.8) становится возможным.

**Порядок реализации:** строился до v0.8 Dashboard, вопреки номеру версии — см. выше.

### v1.0 First Public MVP — первый публичный релиз — **В работе** (открыт 2026-07-23)

При открытии решены три последних Decision Required ADR ([DECISIONS_INDEX.md](docs/adr/DECISIONS_INDEX.md) — принятых теперь 18 из 18): [ADR-007](docs/adr/ADR-007-pm-qa-executors.md) (обе роли PM/QA — агенты, человек подтверждает на контрольных точках), [ADR-010](docs/adr/ADR-010-documentation-language.md) (документация — только русский для v1.0), [ADR-013](docs/adr/ADR-013-managed-projects.md) (управляемые проекты — ретроспективная формализация уже сложившейся реализации).

ADR-007 оказался существенно крупнее «стабилизации»: полная автоматизация ролей PM/QA требует Orchestrator (сознательно отложен в EPIC-006), доступа агентской песочницы к `apps/api` и механизма подтверждения человеком — по объёму сравнимо с EPIC-006. Решение архитектора: v1.0 декомпозирован на отдельные эпики, а не один шаг — детальное планирование каждого при его открытии:

1. **Orchestrator** — подписка на события жизненного цикла задачи, назначение исполнителя роли, запуск через контракт Executor ([module-boundaries.md](docs/architecture/module-boundaries.md): без доменных правил, без durable-состояния).
2. **Роли PM/QA** — role-aware промпты в `agents/claude-code` (один и тот же Docker-адаптер на все три роли, не три отдельных); доступ песочницы к `apps/api`.
3. **Механизм подтверждения человеком** — контрольные точки (приём Definition of Ready, финальное приёмочное решение), которого сейчас не существует ни в API, ни в Dashboard.
4. **Стабилизация и релиз** — синхронизация документации (обнаружена значительная накопленная неточность вне точечных правок этого дня — например, `README.md` корня описывал устаревший roadmap; `domain-model.md` называл Decision Required уже принятые ADR), полная проверка контрактов, публичный релиз `v1.0.0` ([релизный процесс](docs/development/git-workflow.md) — версионирование, тег на `main`, CHANGELOG).

**Результат:** платформа готова к использованию внешними командами.
