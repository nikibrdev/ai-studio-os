# Спецификация: domain/project

## One Sentence

Project — управляемая платформой инициатива разработки программного обеспечения, задающая границу, в которой существуют Epic, Task и Artifact.

## Identity

То, что делает Project именно этим Project, а не Task, не Artifact и не Repository: Task и Artifact существуют *внутри* границы, а сама граница — не сводится ни к одному из них; Repository — техническая деталь, через которую Project подключается к коду, но Project может подключать несколько Repository и не перестаёт быть собой при смене их набора. Identity Project — это сама граница: устойчивая, независимая от того, сколько Epic, Task или Artifact внутри нее уже существует. Если убрать все Task, Epic, Artifact, подключённые Repository и накопленную Memory — останется именно это: различимая инициатива со своей идентичностью, всё ещё «этот Project», даже прежде чем внутри него что-либо создано.

## Purpose

Домен нуждается в Project как отдельной сущности по конкретным причинам:

- **изоляция знания** — Memory накапливается по Project и не должна смешиваться со знаниями другого Project ([memory.md](../../architecture/memory.md)); этой изоляции нужна граница, за пределами которой знание не течёт неявно;
- **общий контекст работы** — Task, Epic и Artifact сами по себе не несут в себе смысла без общего контекста (цели, кода, истории), в рамках которого они существуют; без Project этот контекст пришлось бы восстанавливать заново для каждой единицы работы;
- **единица назначения исполнителей ролей** — то, кто в каком качестве работает, определяется в границах конкретной инициативы, не глобально для всей платформы;
- **единый жизненный цикл инициативы целиком** — архивирование Project замораживает весь контекст разом, отдельно от того, что каждый отдельный Task или Artifact внутри него уже прошёл собственный, независимый жизненный цикл.

Project решает проблему предметной области: как дать множеству Task, Epic и Artifact общий, изолированный контекст — так, чтобы знание, накопленное в одной инициативе, не просачивалось в другую, а жизненный цикл всей инициативы можно было завершить одним действием, не трогая жизненные циклы того, что внутри.

## Responsibilities

Project отвечает за:

- собственную идентичность и жизненный цикл как границы инициативы (см. Lifecycle);
- владение Epic, Task и Artifact, которые существуют в его границах ([domain-model.md](../../architecture/domain-model.md): `Project ├── Task └── Artifact`, [ADR-016](../../adr/ADR-016-artifact-aggregate-root.md));
- подключение 1..* Repository — технических источников кода, в рамках которых ведётся работа.

Project не отвечает за (кратко; исчерпывающий список — раздел Anti-Responsibilities):

- содержание или ход исполнения отдельных Task — это Task и порождаемые им Execution;
- внутреннюю доменную логику Artifact (содержание, инварианты, переходы) — Project владеет границей, в которой Artifact существует, но не его внутренним устройством;
- git-механику самого Repository (ветки, коммиты, PR) — это модуль `git`; Project несёт только ссылку на подключённый Repository.

## Invariants

Инварианты разделены на структурные (форма/строение, неизменны по определению) и поведенческие (выполняются во времени, при любом взаимодействии).

### Structural

1. Project, находящийся в состоянии Active, имеет 1..* Repository ([domain-model.md](../../architecture/domain-model.md): «проект имеет минимум один Repository») — при создании (Created) это ещё не обязательно (см. Lifecycle, Open Questions).
2. Project владеет `0..*` Epic, `0..*` Task и `0..*` Artifact — пустой набор при создании допустим, отсутствие владения невозможно (Epic/Task/Artifact, если существуют, всегда принадлежат ровно одному Project).
3. Project всегда имеет ровно одно текущее состояние из набора Created | Active | Archived — отсутствие состояния невозможно.

### Behavioral

1. После перехода в Archived Project не может вернуться ни в одно из предыдущих состояний — терминально; ни один из атрибутов и связей самого Project не изменяется впредь.
2. Архивирование Project не переопределяет и не форсирует состояния уже существующих внутри него Task или Artifact — они сохраняют собственный, независимо достигнутый Lifecycle-статус (согласовано с тем, как [Artifact](artifact.md) уже определяет: «Archived не означает удалён» — тот же принцип нементального каскадного разрушения применён здесь к границе, а не к содержимому).
3. Подключённый Repository не может быть отсоединён от Project — сознательное ограничение v1 (Architect Review, Final): подключение Repository — необратимая часть истории Project, а не отменяемая операционная деталь; набор подключённых Repository может только расти.
4. Новые Epic, Task и Artifact создаются только в границах Project, находящегося в состоянии Active — не Created (ещё не готов) и не Archived (уже неизменяем).

## Lifecycle

```
Created → Active → Archived
```

- **Created** — Project зарегистрирован, но ещё не обязательно готов к работе: Structural Invariant 1 не требует Repository на этом этапе. Переход в Active выполняется явной командой Activate (Architect Review, Final — не неявным побочным эффектом ConnectRepository: скрытый переход состояния внутри команды с другим именем затруднял бы понимание Lifecycle и не отражён в уже принятом контракте `Registry`, который на сегодня Activate не содержит — расширение контракта запланировано на этапе 2, см. Decision Log). Guard перехода: у Project уже есть хотя бы один подключённый Repository (Structural Invariant 1).
- **Active** — Project полностью операционен: Repository подключён (Structural Invariant 1), в его границах могут создаваться Epic, Task, Artifact.
- **Archived** — терминальное состояние; Project и его собственные атрибуты неизменяемы (Behavioral Invariant 1), содержимое (Task, Artifact) сохраняет собственные статусы (Behavioral Invariant 2).

## Relationships

Project уже полностью определён разделами One Sentence/Identity/Purpose; этот раздел не меняет и не дополняет то, чем Project **является** — он описывает только то, кто и как с Project связан.

- **Project владеет Epic** (`Project "1" o-- "0..*" Epic`) — Epic существует только в границах ровно одного Project.
- **Project владеет Task** (`Project "1" o-- "0..*" Task`) — уже зафиксировано со стороны [Task](task.md) (Structural Invariant 1: Task вне Project не существует); согласовано двусторонне.
- **Project владеет Artifact напрямую** (`Project "1" o-- "0..*" Artifact`, [ADR-016](../../adr/ADR-016-artifact-aggregate-root.md)) — уже зафиксировано со стороны [Artifact](artifact.md) («Project владеет Artifact»); согласовано, не переопределяется.
- **Project подключает 1..* Repository** — ссылка на технический источник кода; git-механика самого Repository (ветки, PR) — вне этой сущности, принадлежит модулю `git`.
- **Memory накапливается по Project, но не принадлежит ему структурно** — Memory ссылается на Project как на границу изоляции знания ([memory.md](../../architecture/memory.md): «знания разных проектов не смешиваются»), владение данными Memory остаётся за модулем `memory`.

## Alternative Interpretations Considered

- **Project как атрибут Repository** — отвергнуто. Project может подключать несколько Repository одновременно (1..*, Structural Invariant 1), а идентичность и жизненный цикл Project (накопленная Memory, назначения ролей) не сводятся ни к одному конкретному репозиторию.
- **Project без собственного жизненного цикла** (статический контейнер/пространство имён) — отвергнуто. Archived — реальное терминальное состояние с конкретными последствиями (неизменяемость собственных атрибутов, Behavioral Invariant 1) — это подразумевает настоящий жизненный цикл, а не пассивную группировку.
- **Project как то же самое, что Epic** — отвергнуто. Epic — единица декомпозиции цели *внутри* Project (Epic принадлежит Project, domain-model.md); Project — сама граница, в которой Epic существует. Отождествление стёрло бы отношение владения между ними.
- **Project как Aggregate Root** — выбрано. Единственная трактовка, выражающая собственную идентичность, независимый жизненный цикл и владение Epic/Task/Artifact, не сводимые ни к одному из них.

## Domain Events

Переходы Lifecycle порождают обязательные публичные события; дополнительно — событие о подключении Repository, поскольку это операционный факт, значимый за пределами Project (уже анонсировано README модуля: «события проекта (создание, подключение репозитория, архивирование)»).

- **ProjectCreated** — публикуется при создании Project (вход в Created). Данные: Identifier, название, момент создания.
- **RepositoryConnected** — публикуется при каждом подключении Repository (не только первом). Данные: Identifier Project, ссылка на Repository, момент.
- **ProjectActivated** — публикуется при переходе Created → Active, вызванном командой Activate. Данные: Identifier, момент перехода.
- **ProjectArchived** — публикуется при переходе Active → Archived. Данные: Identifier, момент перехода.

## Commands

Три из четырёх согласованы с уже принятым `Registry` ([internal/domain/project/registry.go](../../../internal/domain/project/registry.go)); Activate — единственное сознательное расширение контракта, решённое этим ревью (см. Decision Log), а не решение, принятое явочным порядком.

1. **Create** — регистрирует Project в состоянии Created. Обязательный параметр: название.
2. **ConnectRepository** — подключает Repository к Project; допустима в Created и в Active (не в Archived — Behavioral Invariant 1). Не вызывает переход состояния сама по себе (Architect Review, Final) — только удовлетворяет guard-условие команды Activate.
3. **Activate** — переводит Created → Active; допустима только если у Project уже есть хотя бы один подключённый Repository (Structural Invariant 1). Не входит в уже принятый контракт `Registry` — расширение этапа 2 (см. Decision Log), как и `epicID`/scope-AC у [Task](task.md).
4. **Archive** — переводит Active → Archived; после выполнения дальнейшие изменения самого Project недопустимы (Behavioral Invariant 1).

## Queries

Уже принятый код (`registry.go`) не содержит отдельного контракта Queries — эта спецификация фиксирует концептуальный минимум, ожидаемый от будущей реализации (этап 2), не переопределяет ничего уже принятого.

- **Получить по Identifier** — вернуть конкретный Project.
- **Найти по статусу** — например, все Active (действующие инициативы) или все Archived.
- **Получить подключённые Repository** — список Repository, подключённых к Project.

## Examples

Ни кода, ни JSON — только содержательные примеры, показывающие, что модель выдерживает разнородные реальные случаи.

- **Обычный действующий проект** — Create (Created) → ConnectRepository (первый репозиторий) → Activate (переход в Active) → внутри создаются Epic, Task, Artifact.
- **Проект с несколькими репозиториями** — например, монорепозиторий приложения и отдельный репозиторий инфраструктуры, оба подключены к одному Project (Structural Invariant 1 допускает `1..*`).
- **Только что созданный, ещё не активный проект** — Created, ни один Repository ещё не подключён; ни один Task или Artifact не может быть создан в его границах (Behavioral Invariant 4).
- **Архивированный проект** — инициатива завершена, вызван Archive; уже существовавшие внутри Task и Artifact сохраняют собственные, ранее достигнутые статусы (часть — Done, часть — Cancelled) — архивирование Project не переписывает их (Behavioral Invariant 2).

## Acceptance Criteria

Критерии проверяют качество модели, а не факт заполнения разделов; каждый — с самопроверкой против этой спецификации.

- [x] **Сущность имеет единственную ответственность** — Project отвечает только за границу инициативы и владение Epic/Task/Artifact; не совмещает это с содержанием отдельных Task или внутренней логикой Artifact.
- [x] **Все инварианты проверяемы** — каждый Structural/Behavioral инвариант формулирует условие, подтверждаемое или опровергаемое на конкретном Project.
- [x] **Жизненный цикл не содержит неоднозначных переходов** — три состояния, линейная цепочка, Archived терминален (Behavioral Invariant 1); Created → Active — явная команда Activate с чётким guard-условием (Structural Invariant 1), не побочный эффект другой команды.
- [x] **Команды не нарушают инварианты** — ConnectRepository недопустима после Archive; Archive финализирует Project (Behavioral Invariant 1).
- [x] **События являются следствием переходов состояния** — три из четырёх привязаны к Lifecycle; четвёртое (`RepositoryConnected`) явно обосновано как значимый за пределами Project операционный факт, не самовольное исключение из правила.
- [x] **Нет противоречий с принятыми ADR** — согласовано с [ADR-016](../../adr/ADR-016-artifact-aggregate-root.md) (Project владеет Artifact); [ADR-013](../../adr/ADR-013-managed-projects.md) зафиксирован как открытая зависимость, не решается здесь.
- [x] **Спецификация не требует знания Go-кода или реализации** — ни один из трёх PR не содержит упоминаний Go/структур/реализации сверх ссылок на уже принятые контракты по имени.
- [x] **Согласована с Artifact, Execution, Executor и Task** (Delta Review) — без переопределения уже принятых решений об этих сущностях.

### Ожидаемые

- **Контракт назначений исполнителей ролей в Project** — уже анонсирован README модуля, не специфицирован этой версией.

### Возможные

- **Операция отключения или замены Repository**, если возникнет такая потребность (сейчас — сознательное ограничение, см. Non-Goals).
- **Квоты или лимиты** на Project (например, число одновременных активных Task).
- **Иерархия между Project** (например, суб-проекты) — сейчас не рассматривается.
- **Настройки Project как явная доменная сущность** — сейчас не выделены отдельно.

## Anti-Responsibilities

- Project **не управляет содержанием или ходом исполнения отдельных Task** — это Task и порождаемые им Execution.
- Project **не определяет внутренние инварианты или Lifecycle Artifact** — Project владеет границей, в которой Artifact существует, не его внутренним устройством.
- Project **не содержит git-логику** (ветки, Pull Request, ревью) — это модуль `git`; Project лишь ссылается на подключённые Repository.
- Project **не определяет правила Workflow** — это Workflow, Project лишь предоставляет контекст, в котором Workflow применяется.
- Project **не хранит и не интерпретирует содержание Memory** — это модуль `memory`; Project служит только границей изоляции.

## Non-Goals

Сформулированы как сознательные ограничения этой версии, а не как список отсутствующих функций.

- В этой версии Project не специфицирует контракт назначений исполнителей ролей — анонсированная, но пока не выполненная работа.
- В этой версии Project не вводит операцию отключения Repository.
- В этой версии Project не моделирует квоты, лимиты или иерархию между проектами.
- В этой версии Project не определяет формат подключения репозиториев — [ADR-013](../../adr/ADR-013-managed-projects.md), Decision Required.

## Removal Test

Если убрать Project из модели, платформа теряет единую границу, в рамках которой Task, Epic и Artifact получают общий контекст и в рамках которой изолируется накопленное знание (Memory): без Project пришлось бы либо смешивать знания и работу разных инициатив в общем пространстве — нарушая изоляцию Memory ([memory.md](../../architecture/memory.md)), — либо изобретать замену границе, фактически воссоздавая Project под другим именем. Именно эта потеря — невозможность дать множеству Task, Epic и Artifact общий, изолированный контекст с собственным жизненным циклом инициативы целиком — оправдывает Project как самостоятельный Aggregate Root.

## Open Questions

- **Контракт назначений исполнителей ролей в Project** — анонсирован README модуля, не специфицирован этой версией (см. Non-Goals).

### Three-Pass Review — ответы на диагностические вопросы

- **Pass 1, самое слабое место модели:** до Final Architecture Review — механика перехода Created → Active была гипотезой, а не решением; устранено явной командой Activate с проверяемым guard-условием (Decision Log). Текущее самое слабое место — формат подключения репозиториев ([ADR-013](../../adr/ADR-013-managed-projects.md)) остаётся Decision Required, и Repository описан абстрактной ссылкой без деталей.
- **Pass 2, сильнее всего влияющая соседняя сущность:** Artifact — именно через Project реализовано прямое владение Artifact ([ADR-016](../../adr/ADR-016-artifact-aggregate-root.md)), ключевое архитектурное решение всего этапа 1, сильнее всего ограничивающее форму этой спецификации.
  - Delta Review: (1) *Требует ли новая спецификация изменить уже утверждённую/черновую модель?* Нет. (2) *Использует ли она уже утверждённые понятия одинаковым образом?* Да — «владеет» применено как структурная агрегация единообразно по всем четырём предыдущим спецификациям. (3) *Не вводит ли она новое понятие, уже существующее под другим названием?* Нет — ничего нового не введено взамен уже принятого.
- **Pass 3, наиболее вероятное будущее изменение, требующее пересмотра:** принятие [ADR-013](../../adr/ADR-013-managed-projects.md) (формат подключения репозиториев) почти наверняка потребует пересмотра Relationships/Invariants этой спецификации — сейчас Repository описан абстрактной ссылкой без деталей подключения.
