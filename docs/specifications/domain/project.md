# Спецификация: domain/project

## One Sentence

Project — управляемая платформой инициатива разработки программного обеспечения, задающая границу, в которой существуют Epic, Task и Artifact.

## Identity

То, что делает Project именно этим Project, а не Task, не Artifact и не Repository: Task и Artifact существуют *внутри* границы, а сама граница — не сводится ни к одному из них; Repository — техническая деталь, через которую Project подключается к коду, но Project может подключать несколько Repository и не перестаёт быть собой при смене их набора. Identity Project — это сама граница: устойчивая, независимая от того, сколько Epic, Task или Artifact внутри нее уже существует. Если убрать все Task, Epic, Artifact, подключённые Repository и накопленную Memory — останется именно это: различимая инициатива со своей идентичностью, всё ещё «этот Project», даже прежде чем внутри него что-либо создано.

## Purpose

Домен нуждается в Project как отдельной сущности по конкретным причинам:

- **изоляция знания** — Memory накапливается по Project и не должна смешиваться со знаниями другого Project ([memory.md](../../architecture/memory.md)); этой изоляции нужна граница, за пределами которой знание не течёт неявно;
- **общий контекст работы** — Task, Epic и Artifact сами по себе не несут в себе смысла без общего контекста (цели, кода, истории), в рамках которого они существуют; без Project этот контекст пришлось бы восстанавливать заново для каждой единицы работы;
- **единица назначения исполнителей ролей** — то, кто в каком качестве работает, определяется в границах конкретной инициативы, не глобально для всей платформы;
- **единый жизненный цикл инициативы целиком** — архивирование Project замораживает весь контекст разом, отдельно от того, что каждый отдельный Task или Artifact внутри него уже прошёл собственный, независимый жизненный цикл.

Project решает проблему предметной области: как дать множеству Task, Epic и Artifact общий, изолированный контекст — так, чтобы знание, накопленное в одной инициативе, не просачивалось в другую, а жизненный цикл всей инициативы можно было завершить одним действием, не трогая жизненные циклы того, что внутри.

## Responsibilities

Project отвечает за:

- собственную идентичность и жизненный цикл как границы инициативы (см. Lifecycle);
- владение Epic, Task и Artifact, которые существуют в его границах ([domain-model.md](../../architecture/domain-model.md): `Project ├── Task └── Artifact`, [ADR-016](../../adr/ADR-016-artifact-aggregate-root.md));
- подключение 1..* Repository — технических источников кода, в рамках которых ведётся работа.

Project не отвечает за (кратко; исчерпывающий список — раздел Anti-Responsibilities):

- содержание или ход исполнения отдельных Task — это Task и порождаемые им Execution;
- внутреннюю доменную логику Artifact (содержание, инварианты, переходы) — Project владеет границей, в которой Artifact существует, но не его внутренним устройством;
- git-механику самого Repository (ветки, коммиты, PR) — это модуль `git`; Project несёт только ссылку на подключённый Repository.

## Invariants

Инварианты разделены на структурные (форма/строение, неизменны по определению) и поведенческие (выполняются во времени, при любом взаимодействии).

### Structural

1. Project, находящийся в состоянии Active, имеет 1..* Repository ([domain-model.md](../../architecture/domain-model.md): «проект имеет минимум один Repository») — при создании (Created) это ещё не обязательно (см. Lifecycle, Open Questions).
2. Project владеет `0..*` Epic, `0..*` Task и `0..*` Artifact — пустой набор при создании допустим, отсутствие владения невозможно (Epic/Task/Artifact, если существуют, всегда принадлежат ровно одному Project).
3. Project всегда имеет ровно одно текущее состояние из набора Created | Active | Archived — отсутствие состояния невозможно.

### Behavioral

1. После перехода в Archived Project не может вернуться ни в одно из предыдущих состояний — терминально; ни один из атрибутов и связей самого Project не изменяется впредь.
2. Архивирование Project не переопределяет и не форсирует состояния уже существующих внутри него Task или Artifact — они сохраняют собственный, независимо достигнутый Lifecycle-статус (согласовано с тем, как [Artifact](artifact.md) уже определяет: «Archived не означает удалён» — тот же принцип нементального каскадного разрушения применён здесь к границе, а не к содержимому).
3. Подключённый Repository не может быть отсоединён от Project — уже принятый контракт (`Registry`) не предоставляет такой операции; набор подключённых Repository может только расти.
4. Новые Epic, Task и Artifact создаются только в границах Project, находящегося в состоянии Active — не Created (ещё не готов) и не Archived (уже неизменяем).

## Lifecycle

```
Created → Active → Archived
```

- **Created** — Project зарегистрирован, но ещё не обязательно готов к работе: Structural Invariant 1 не требует Repository на этом этапе. Переход в Active, вероятно, наступает при подключении первого Repository (единственная операция, помимо Create и Archive, в уже принятом контракте `Registry`) — предложенное здесь правило, требующее подтверждения архитектора (см. Open Questions), поскольку явной команды «Activate» контракт не содержит.
- **Active** — Project полностью операционен: Repository подключён (Structural Invariant 1), в его границах могут создаваться Epic, Task, Artifact.
- **Archived** — терминальное состояние; Project и его собственные атрибуты неизменяемы (Behavioral Invariant 1), содержимое (Task, Artifact) сохраняет собственные статусы (Behavioral Invariant 2).

## Relationships

Project уже полностью определён разделами One Sentence/Identity/Purpose; этот раздел не меняет и не дополняет то, чем Project **является** — он описывает только то, кто и как с Project связан.

- **Project владеет Epic** (`Project "1" o-- "0..*" Epic`) — Epic существует только в границах ровно одного Project.
- **Project владеет Task** (`Project "1" o-- "0..*" Task`) — уже зафиксировано со стороны [Task](task.md) (Structural Invariant 1: Task вне Project не существует); согласовано двусторонне.
- **Project владеет Artifact напрямую** (`Project "1" o-- "0..*" Artifact`, [ADR-016](../../adr/ADR-016-artifact-aggregate-root.md)) — уже зафиксировано со стороны [Artifact](artifact.md) («Project владеет Artifact»); согласовано, не переопределяется.
- **Project подключает 1..* Repository** — ссылка на технический источник кода; git-механика самого Repository (ветки, PR) — вне этой сущности, принадлежит модулю `git`.
- **Memory накапливается по Project, но не принадлежит ему структурно** — Memory ссылается на Project как на границу изоляции знания ([memory.md](../../architecture/memory.md): «знания разных проектов не смешиваются»), владение данными Memory остаётся за модулем `memory`.

## Alternative Interpretations Considered

- **Project как атрибут Repository** — отвергнуто. Project может подключать несколько Repository одновременно (1..*, Structural Invariant 1), а идентичность и жизненный цикл Project (накопленная Memory, назначения ролей) не сводятся ни к одному конкретному репозиторию.
- **Project без собственного жизненного цикла** (статический контейнер/пространство имён) — отвергнуто. Archived — реальное терминальное состояние с конкретными последствиями (неизменяемость собственных атрибутов, Behavioral Invariant 1) — это подразумевает настоящий жизненный цикл, а не пассивную группировку.
- **Project как то же самое, что Epic** — отвергнуто. Epic — единица декомпозиции цели *внутри* Project (Epic принадлежит Project, domain-model.md); Project — сама граница, в которой Epic существует. Отождествление стёрло бы отношение владения между ними.
- **Project как Aggregate Root** — выбрано. Единственная трактовка, выражающая собственную идентичность, независимый жизненный цикл и владение Epic/Task/Artifact, не сводимые ни к одному из них.

## Open Questions

- **Механизм перехода Created → Active** (Lifecycle) — уже принятый `Registry` не содержит явной команды `Activate`; эта спецификация предполагает, что переход наступает при подключении первого Repository, но это вывод из формы контракта, не зафиксированное архитектором решение. Требует подтверждения.
- **Отсутствие операции отключения Repository** (Behavioral Invariant 3) — уже принятый контракт не предоставляет такой операции; спецификация принимает это как сознательное ограничение, а не пробел, но явного решения архитектора по этому вопросу нет.
- **Контракт назначений исполнителей ролей в Project** — упомянут в Purpose/Responsibilities как обязанность, но не специфицирован этой версией: README модуля [internal/domain/project](../../../internal/domain/project/README.md) называет его «контракт назначений — Domain Layer», то есть отдельной, ещё не выполненной работой. Не входит в эту спецификацию (см. Non-Goals, PR 3).
- **Формат подключения репозиториев** — [ADR-013](../../adr/ADR-013-managed-projects.md), Decision Required; эта спецификация не решает вопрос, только фиксирует зависимость.
