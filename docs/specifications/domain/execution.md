# Спецификация: domain/execution

## One Sentence

Execution — единичный, ограниченный во времени запуск конкретного Executor'а, в рамках которого производятся Artifact и фиксируется статус исполняемой работы.

## Identity

То, что делает Execution именно этим Execution, а не Task и не Executor: Task — организующая единица работы с собственным, уже канонизированным жизненным циклом ([state-machine.md](../../architecture/state-machine.md)), которая может пережить несколько попыток исполнения; Executor — долгоживущий, переиспользуемый технический бэкенд, существующий независимо от того, сколько раз и для каких задач он был запущен. Execution — ни то, ни другое: это конкретный, неповторимый факт «этот Executor сейчас выполняет именно эту работу». Если убрать порождающую Task, использованный Executor и произведённые Artifact — останется именно это: единичность и невозобновимость попытки. Даже если тот же Executor повторно берётся за ту же Task после неудачи, это не продолжение прежнего Execution, а новый — Execution не переживает собственное завершение и не может быть «начат заново» тем же экземпляром.

## Purpose

Домен нуждается в Execution как отдельной сущности по конкретным причинам:

- **разделение организации работы и факта её выполнения** — Task несёт цель, scope и процессный жизненный цикл ([state-machine.md](../../architecture/state-machine.md), девять состояний), который уже закрыт и не рассчитан на то, чтобы отражать в себе каждую техническую попытку исполнения; без Execution эту функцию пришлось бы втискивать в состояния Task, размывая его собственную ответственность;
- **безопасные повторные попытки** — если исполнение завершилось неудачно, повторная попытка не должна ни искажать, ни замещать историю неудачной: обе должны существовать как разные Execution одной Task, а не как перезаписанное состояние одной сущности;
- **единая точка входа для контракта исполнителя** — Executor взаимодействует с платформой через ровно четыре возможности ([ADR-005](../../adr/ADR-005-executor-contract.md): Accept/Artifacts/Status/Finish); Execution — доменный адресат этого контракта, место, куда поступают статус и произведённые артефакты произвольного технического исполнителя, без чего у платформы не было бы, куда «приземлить» эти данные;
- **измерение самого факта исполнения** — когда начато, сколько длилось, какие Tool вызывались, — эти данные принадлежат конкретному запуску, а не задаче в целом, и нужны независимо от семантики того, что именно выполнялось.

Execution решает проблему предметной области: как отличить неизменную цель работы (Task) от переменного, потенциально повторяемого факта её технического выполнения — так, чтобы неудача одной попытки не разрушала ни задачу, ни историю других попыток.

## Responsibilities

Execution отвечает за:

- фиксацию того, кто (Executor), в рамках какой Task и когда выполнял работу;
- отслеживание хода выполнения на протяжении своего запуска;
- производство 0..* Artifact — по ссылке, не по владению ([ADR-016](../../adr/ADR-016-artifact-aggregate-root.md));
- собственный статус исполнения (`ExecutionStatus`), который Execution несёт напрямую, без промежуточной сущности Result ([ADR-005](../../adr/ADR-005-executor-contract.md)).

Execution не отвечает за (кратко; исчерпывающий список — раздел Anti-Responsibilities):

- решение о том, нужно ли начинать исполнение и когда — это относится к Task/Workflow;
- интерпретацию содержания произведённых Artifact — это либо сам Artifact, либо его потребители;
- регистрацию, доступность или отключение Executor — это ответственность модуля `executor`;
- собственный жизненный цикл Task — переходы состояний Task определяются его собственной state machine, Execution на них не влияет напрямую.

## Invariants

Инварианты разделены на структурные (форма/строение, неизменны по определению) и поведенческие (выполняются во времени, при любом взаимодействии).

### Structural

1. Execution всегда связан ровно с одним Executor, назначенным при создании; эта связь не переопределяется другим Executor в течение жизни Execution.
2. Execution всегда связан ровно с одной Task, в рамках которой возник; Execution не существует без порождающей Task ([domain-model.md](../../architecture/domain-model.md): «одно исполнение — один исполнитель и одна задача»).
3. Execution всегда имеет `ExecutionStatus`, начиная с момента создания (состояние Queued) — отсутствие статуса невозможно ни в каком состоянии жизненного цикла.
4. Artifact, на которые ссылается Execution, — это ссылки (0..* Artifact), не владение: удаление ссылки в Execution не удаляет и не изменяет сам Artifact, и наоборот, — согласовано с [ADR-016](../../adr/ADR-016-artifact-aggregate-root.md) и с тем, как сам Artifact хранит обратную, необязательную и единственную ссылку на породившее его Execution (Behavioral Invariant 3 спецификации [Artifact](artifact.md)).
5. Execution состоит из фиксированных при создании данных о запуске (Executor, Task, момент начала) и изменяющегося во времени состояния (`ExecutionStatus`, множество произведённых Artifact) — оба рода данных присутствуют всегда, не опционально.

### Behavioral

1. После перехода в терминальное состояние (Succeeded | Failed | Aborted) `ExecutionStatus` и множество произведённых Artifact для этого Execution более не меняются; повторная попытка выполнить ту же работу — это новый Execution, а не продолжение или мутация старого.
2. Переходы Lifecycle необратимы: Execution не может вернуться из Running в Queued и не может покинуть терминальное состояние.
3. Execution не может перейти в Running, минуя подтверждение Executor'ом принятия работы (`Accept`, [ADR-005](../../adr/ADR-005-executor-contract.md)) — Queued отражает «назначено, но ещё не подтверждено принятым», а не «уже выполняется».
4. Artifact считается произведённым данным Execution только если он появился в системе в период, когда это Execution находилось в состоянии Running, — не задним числом и не заранее; это условие защищает смысл обратной ссылки «породившее Execution» на стороне Artifact.

## Lifecycle

```
Queued → Running → Succeeded
                  → Failed
                  → Aborted
Queued → Aborted
```

Три терминальных состояния (Succeeded, Failed, Aborted) взаимно исключают друг друга и не пересекаются: Succeeded — работа завершена так, как предполагалось; Failed — Executor сам сообщил о невозможности завершить работу ([ADR-005](../../adr/ADR-005-executor-contract.md), `Finish`/`Status`); Aborted — исполнение прекращено внешним решением (например, отменой Task, истечением времени ожидания) независимо от того, как далеко Executor продвинулся, — различие в источнике прекращения (сам Executor против внешнего решения), а не в фактическом результате. Переход Queued → Aborted — отдельный, короткий путь для случая, когда исполнение отменяется до того, как Executor успел принять работу (`Accept` не был вызван или не был подтверждён).

- **Queued** — Execution создан и связан с Executor и Task (Structural Invariants 1–2), но Executor ещё не подтвердил принятие работы. Может находиться здесь ограниченное время (ожидание доступности Executor), но не бессрочно долго в практическом смысле — точный порог не определяется этой спецификацией (см. Open Questions).
- **Running** — Executor принял работу (`Accept`) и активно её выполняет; в течение этого состояния могут поступать Artifact (через `Artifacts`) и обновляться статус (через `Status`).
- **Succeeded / Failed / Aborted** — терминальные состояния; после входа в любое из них Execution не изменяется (Behavioral Invariant 1).

## Relationships

Execution уже полностью определён разделами One Sentence/Identity/Purpose; этот раздел не меняет и не дополняет то, чем Execution **является** — он описывает только то, кто и как с Execution связан.

- **Task создаёт и владеет Execution** ([domain-model.md](../../architecture/domain-model.md): `Task "1" o-- "0..*" Execution`): Execution не существует вне контекста породившей его Task (Structural Invariant 2); одна Task может породить несколько Execution — например, несколько попыток исполнения одного и того же шага.
- **Execution использует Executor, но не владеет им**: Executor — самостоятельная, переиспользуемая сущность со своим жизненным циклом (Registered → Active ⇄ Disabled → Retired), которая продолжает существовать независимо от того, сколько Execution она уже выполнила и выполнит впредь.
- **Execution производит Artifact, но не владеет им** ([ADR-016](../../adr/ADR-016-artifact-aggregate-root.md)): согласовано с тем, как сам [Artifact](artifact.md) описывает эту связь со своей стороны (необязательная, единственная ссылка на породившее Execution). Кардинальность с точки зрения Execution — производит 0..* Artifact; с точки зрения Artifact — не более одного породившего Execution (уже зафиксировано в спецификации Artifact, не переопределяется здесь).
- **Execution вызывает Tool**: в ходе выполнения Execution может обращаться к 0..* Tool; вызов Tool — действие в рамках Execution, не самостоятельная сущность, связанная с Task или Artifact напрямую.

## Alternative Interpretations Considered

- **Execution как часть Task** — отвергнуто. Жизненный цикл Task уже канонизирован как закрытый набор из девяти состояний ([state-machine.md](../../architecture/state-machine.md)), описывающих процессное продвижение работы, а не технические попытки её исполнения. Task может пережить несколько Execution (успешных и неуспешных); растворение Execution в Task потребовало бы либо расширения state machine Task под каждую попытку исполнения, либо потери истории неудачных попыток — оба варианта хуже отдельной сущности.
- **Execution как часть/сессия Executor** — отвергнуто. Executor — долгоживущая, переиспользуемая сущность (Registered → Active ⇄ Disabled → Retired), не привязанная к одному запуску; один Executor выполняет множество Execution за свою жизнь, зачастую для разных Task одновременно или последовательно. Привязка Execution к Executor как его внутреннему состоянию сделала бы невозможным одновременное существование нескольких независимых запусков одного и того же Executor.
- **Execution как Result / обёртка вокруг результата** — отвергнуто явно принятым решением [ADR-005](../../adr/ADR-005-executor-contract.md): контракт Executor сознательно спроектирован без промежуточной сущности Result — статус и артефакты передаются напрямую через `Status`/`Artifacts`, поэтому и в домене Execution сам несёт эти данные, а не оборачивает их в отдельную сущность.
- **Execution как производное Event** — отвергнуто. Event — неизменяемый факт, зафиксированный в момент публикации (Published → Delivered → Archived); Execution, напротив, обладает протяжённостью во времени и собственным продвижением по состояниям (Queued → Running → терминальное) до момента, когда становится неизменным. Execution порождает события о своих переходах (аналогично тому, как это уже установлено для Artifact), но не является событием само по себе.
- **Execution как Aggregate Root** — выбрано. Единственная из рассмотренных трактовок, которая корректно выражает собственный жизненный цикл запуска, отдельный и от организующей его Task, и от переиспользуемого Executor, при этом не разрывая владения: Execution принадлежит Task, но самостоятелен как сущность внутри этого владения.

## Domain Events

Только переходы Lifecycle порождают обязательные публичные Domain Events — согласовано с уже принятым для [Artifact](artifact.md) принципом (Delta Review). Каждое из пяти событий соответствует ровно одному переходу или созданию.

- **ExecutionQueued** — публикуется при создании Execution (вход в Queued). Данные: Identifier, Task, Executor, момент создания.
- **ExecutionStarted** — публикуется при переходе Queued → Running (Executor подтвердил принятие работы). Данные: Identifier, момент перехода.
- **ExecutionSucceeded** — публикуется при переходе Running → Succeeded. Данные: Identifier, момент перехода, ссылки на произведённые Artifact (если есть).
- **ExecutionFailed** — публикуется при переходе Running → Failed. Данные: Identifier, момент перехода, ссылки на произведённые к этому моменту Artifact (Behavioral Invariant 4 не требует их отсутствия — неудача может произойти уже после того, как часть работы произведена).
- **ExecutionAborted** — публикуется при переходе в Aborted (из Queued или из Running — единое событие для обоих путей, поскольку итоговое состояние одно и то же, аналогично `ArtifactArchived` у Artifact). Данные: Identifier, момент перехода, состояние, из которого пришли (Queued | Running).

## Commands

Соответствие четырём возможностям контракта Executor ([ADR-005](../../adr/ADR-005-executor-contract.md): `Accept`/`Artifacts`/`Status`/`Finish`) явно: `Accept` отражается командой **Accept**, `Artifacts` — командой **RecordArtifact** (платформа опрашивает Executor и переносит результат в Execution), `Status`/`Finish` вместе отражаются терминальными командами **Succeed**/**Fail** — единый акт завершения работы Executor'ом, различающийся только исходом. **Abort** не имеет прямого соответствия в контракте Executor — это решение, инициированное платформой (Task/Workflow), а не самим Executor.

1. **Create** — регистрирует новый Execution в состоянии Queued. Обязательные параметры: Task, Executor (Structural Invariants 1–2).
2. **Accept** — переводит Queued → Running; недопустима, если Execution уже покинул Queued.
3. **RecordArtifact** — добавляет ссылку на произведённый Artifact к множеству Execution; допустима только пока Execution в состоянии Running (Behavioral Invariant 4).
4. **Succeed** — переводит Running → Succeeded; после выполнения дальнейшие RecordArtifact для этого Execution недопустимы (Behavioral Invariant 1).
5. **Fail** — переводит Running → Failed; как и Succeed, финализирует множество произведённых Artifact.
6. **Abort** — переводит Queued → Aborted или Running → Aborted; не изменяет и не удаляет уже произведённые к этому моменту Artifact.

## Queries

- **Получить по Identifier** — вернуть конкретный Execution.
- **Найти по Task** — все Execution, порождённые данной Task, включая историю неудачных и отменённых попыток (Relationships: Task владеет Execution).
- **Найти по Executor** — все Execution, выполненные данным Executor (например, для оценки его загрузки или надёжности).
- **Найти по состоянию** — например, все Running (мониторинг активных исполнений) или все Queued дольше заданного порога (эксплуатационная потребность, см. Open Questions).
- **Найти по произведённому Artifact** — какое Execution произвело данный Artifact; сторона запроса, симметричная собственному query Artifact «Найти по породившему Execution» — оба обращаются к одной и той же связи, не дублируют её независимым хранением.

## Examples

Ни кода, ни JSON — только содержательные примеры, показывающие, что модель выдерживает разнородные реальные случаи.

- **Успешная реализация задачи** — Task: написание Pull Request; Executor: Claude Code. Queued (назначен) → Running (Accept, пишет код, по ходу RecordArtifact для промежуточных файлов) → Succeeded. Произведён Artifact типа PullRequest.
- **Неудачный тестовый прогон** — Task: QA-проверка; Executor: автоматический тестовый раннер. Running → Failed (тесты не прошли, Executor сообщил через `Finish`). Artifact типа TestReport всё же произведён — отчёт о неудаче тоже результат работы (ExecutionFailed не запрещает RecordArtifact, произошедший до отказа).
- **Отменённая до начала задача** — Task отменена человеком раньше, чем Executor успел принять работу. Execution: Queued → Aborted, ни один Artifact не произведён.
- **Человек как Executor** — Executor: человек-архитектор, вручную оформляющий решение. Queued → Running → Succeeded, произведён Artifact типа ADR — показывает, что модель не завязана технически на автоматических исполнителях ([ADR-005](../../adr/ADR-005-executor-contract.md): Executor — любой технический бэкенд, включая человека).
- **Повторная попытка после сбоя** — первый Execution той же Task: Running → Failed. Та же Task порождает второй, независимый Execution (тем же или другим Executor): Queued → Running → Succeeded. Оба Execution существуют одновременно в истории одной Task (Behavioral Invariant 1: неудачный Execution не переиспользуется и не мутирует).

## Acceptance Criteria

Критерии проверяют качество модели, а не факт заполнения разделов; каждый — с самопроверкой против этой спецификации.

- [x] **Сущность имеет единственную ответственность** — Execution отвечает только за факт, ход и статус конкретного запуска (Responsibilities); не совмещает это с организацией работы (Task) или регистрацией/доступностью исполнителя (Executor).
- [x] **Все инварианты проверяемы** — каждый Structural/Behavioral инвариант формулирует условие, подтверждаемое или опровергаемое на конкретном Execution, а не декларацию намерения.
- [x] **Жизненный цикл не содержит неоднозначных переходов** — пять переходов (Queued→Running, Queued→Aborted, Running→Succeeded, Running→Failed, Running→Aborted), необратимы; три терминальных состояния взаимно исключают друг друга по источнику прекращения, не по фактическому результату.
- [x] **Команды не нарушают инварианты** — Accept недопустим вне Queued; RecordArtifact допустим только в Running (согласовано с Behavioral Invariant 4); Succeed/Fail финализируют множество произведённых Artifact (согласовано с Behavioral Invariant 1).
- [x] **События являются следствием переходов состояния** — все пять Domain Events привязаны к конкретному переходу или созданию, ни одно не публикуется по причине, не отражённой в Lifecycle.
- [x] **Нет противоречий с принятыми ADR** — согласовано с [ADR-005](../../adr/ADR-005-executor-contract.md) (Executor Contract, без обёртки Result) и [ADR-016](../../adr/ADR-016-artifact-aggregate-root.md) (Artifact — Aggregate Root, ссылка, не владение).
- [x] **Спецификация не требует знания Go-кода или реализации** — ни один из трёх PR не содержит упоминаний Go/структур/реализации.
- [x] **Согласована с уже утверждённой спецификацией Artifact** (обязательный Delta Review) — терминология «ссылка, не владение» и кардинальность связи выровнены, не переопределены заново.

## Future Extensions

Разделены на ожидаемые (вероятны с высокой уверенностью, модель сознательно оставляет для них пространство) и возможные (могут появиться, но пока не имеют достаточных оснований для проектирования).

### Ожидаемые

- **Метрики надёжности/производительности Executor**, выводимые из истории его Execution (среднее время выполнения, доля Failed) — естественное продолжение Query «Найти по Executor».
- **Явная модель тайм-аута** для предельного времени пребывания в Queued (см. Open Questions) — вероятно, потребует нового инварианта или конфигурации на уровне Application Layer, не решается в этой версии.
- **Формализованная политика повторных попыток** — если Workflow/Task введёт явные правила «когда и сколько раз повторять после Failed/Aborted», Execution, вероятно, потребует явной связи «эта попытка — N-я для данной Task» вместо простого перечисления через Query.

### Возможные

- **Сущность Attempt**, группирующая серию Execution одной Task в логическую последовательность попыток.
- **Параллельные под-исполнения** внутри одного Execution — сейчас модель предполагает один непрерывный поток статуса.
- **Приоритизация очереди** Queued-исполнений на уровне платформы.
- **Детальная структура прогресс-лога** (журнал хода выполнения, heartbeat) — domain-model.md уже фиксирует, что такие данные принадлежат Execution, а не Artifact; их устройство не специфицируется этой версией.

## Anti-Responsibilities

- Execution **не решает, когда и нужно ли начинать исполнение** — это относится к Task/Workflow.
- Execution **не управляет собственным жизненным циклом Task** — переходы состояний Task определяются его собственной state machine ([state-machine.md](../../architecture/state-machine.md)), Execution на них не влияет напрямую.
- Execution **не интерпретирует содержание произведённых Artifact** — это либо сам Artifact, либо его потребители.
- Execution **не управляет регистрацией, доступностью или отключением Executor** — это ответственность модуля `executor`.
- Execution **не определяет политику повторных попыток** (когда и сколько раз пробовать снова после Failed/Aborted) — эта политика принадлежит Workflow/Task.

## Non-Goals

Сформулированы как сознательные ограничения этой версии, а не как список отсутствующих функций.

- В этой версии Execution не моделирует параллельные под-исполнения внутри одного запуска — один Execution соответствует одному непрерывному потоку статуса.
- В этой версии Execution не определяет тайм-аут или предельное время ожидания в Queued — операционный вопрос, не решённый на уровне домена (см. Open Questions).
- В этой версии Execution не вводит отдельную сущность Attempt для группировки серии попыток одной Task — история доступна через Query «Найти по Task».
- В этой версии Execution не определяет политику повторных попыток после неудачи — не входит в ответственность модуля (Anti-Responsibilities).

## Removal Test

Если убрать Execution из модели, платформа теряет способность отличить организующую единицу работы (Task, с собственным процессным жизненным циклом) от технического факта её выполнения конкретным исполнителем: результат и статус пришлось бы приписывать либо напрямую Task — стирая грань между организацией работы и отдельными попытками её выполнения, делая невозможным различить несколько независимых попыток одной задачи, — либо Executor, стирая грань между переиспользуемым исполнителем и разовым фактом его использования. Именно эта потеря — невозможность различить задачу, попытку её выполнения и исполнителя как три разные сущности — оправдывает Execution как самостоятельный Aggregate Root.

## Open Questions

- **Порог времени пребывания в Queued** (Lifecycle) — спецификация сознательно не фиксирует, сколько Execution может ожидать принятия Executor'ом, прежде чем это станет операционной проблемой; решение — вне домена (Infrastructure/Application).
- **Различие Failed и Aborted на границе** (Lifecycle) — случай, когда Executor сам не успевает сообщить о сбое до внешней отмены, оставлен на усмотрение реализации Application Layer.
- **Политика повторных попыток и видимость для Task** — как и когда Task/Workflow решает породить повторный Execution после Failed/Aborted — принадлежит Workflow, требует согласования при написании спецификации Task (TASK-032).

### Three-Pass Review — ответы на диагностические вопросы

- **Pass 1, самое слабое место модели:** открытый порог времени пребывания в Queued — ни один Invariant не ограничивает, как долго Execution может ждать принятия Executor'ом, что оставляет потенциально неограниченное «зависшее» состояние вне доменного контроля.
- **Pass 2, сильнее всего влияющая соседняя сущность:** Artifact — почти все структурные решения этой спецификации (кардинальность связи, «ссылка, не владение», паттерн Domain Events строго на переходах) унаследованы через обязательный Delta Review, а не выведены заново.
  - Delta Review: (1) *Требует ли новая спецификация изменить уже утверждённую модель?* Нет — Execution только использует уже зафиксированную Artifact'ом сторону связи (0..1 порождающее Execution), не меняет её. (2) *Использует ли она уже утверждённые понятия одинаковым образом?* Да — «ссылка, не владение» и «породившее Execution» использованы в точности как в спецификации Artifact. (3) *Не вводит ли она новое понятие, уже существующее под другим названием?* Нет — команда RecordArtifact описывает появление ссылки со стороны Execution и не дублирует ничего уже определённого у Artifact.
- **Pass 3, наиболее вероятное будущее изменение, требующее пересмотра:** формализация политики повторных попыток при проектировании Task/Workflow (TASK-032) — как только появится явная сущность Attempt или правило «когда повторять», вероятно, потребуется пересмотр того, как Execution выражает свою принадлежность к серии попыток (сейчас — только через Query «Найти по Task», без явного номера попытки).
