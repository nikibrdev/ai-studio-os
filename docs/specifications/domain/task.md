# Спецификация: domain/task

## One Sentence

Task — единица работы с формализованным жизненным циклом, через которую в границах Project организуется процесс достижения конкретного результата.

## Identity

То, что делает Task именно этой Task, а не Execution, Artifact или Epic: Execution — технический факт одной попытки исполнения, Artifact — производимая ценность, Epic — более крупная цель, слишком большая для одной единицы работы. Task — ни то, ни другое: это организующая единица *процесса*, обладающая собственным владельцем на каждом этапе и продвигающаяся по фиксированной, канонически заданной последовательности состояний ([state-machine.md](../../architecture/state-machine.md)) независимо от того, сколько технических попыток исполнения (Execution) она породила и что именно в итоге произведено (Artifact). Если убрать Epic, которому Task может принадлежать, Execution, которые она порождает, и Artifact, к которым может привести её выполнение, — останется именно это: обособленный, ограниченный по scope объём работы со своими критериями приёмки и собственной позицией в каноническом процессе.

## Purpose

Домен нуждается в Task как отдельной сущности по конкретным причинам:

- **организация процесса, отдельная от факта исполнения и от результата** — платформе нужно единое место, отражающее, кто сейчас отвечает за конкретный объём работы, независимо от того, сколько раз и как технически эта работа исполнялась (Execution) и что именно произведено (Artifact);
- **подотчётность на каждом этапе** — Definition of Ready и Definition of Done формализуют вход и выход из работы; явный владелец на каждом состоянии (Developer, Reviewer, QA Engineer, Project Manager) исключает состояния «ничьей» ответственности;
- **единица декомпозиции Epic** — крупная цель делится на Task именно потому, что Task — минимальный объём работы, который можно независимо начать, завершить или отменить;
- **точка запуска технического исполнения** — переход Task в In Progress — это момент, когда для агента-исполнителя порождается Execution ([events.md](../../architecture/events.md): `TaskStarted`); без Task этому запуску не с чем было бы быть связанным.

Task решает проблему предметной области: как организовать и отследить объём работы с чётким владельцем на каждом этапе, не смешивая эту организационную роль ни с техническим фактом исполнения (Execution), ни с производимым результатом (Artifact).

## Responsibilities

Task отвечает за:

- цель, scope и критерии приёмки, зафиксированные при постановке;
- собственное продвижение по каноническому жизненному циклу ([state-machine.md](../../architecture/state-machine.md)) и историю переходов;
- порождение Execution при переходе в In Progress (для агента-исполнителя) — согласовано со спецификацией [Execution](execution.md): Task владеет порождённым Execution;
- связь с рабочей веткой (Branch) и оформлением изменений (PullRequest) через модуль `git`.

Task не отвечает за (кратко; исчерпывающий список — раздел Anti-Responsibilities):

- технический ход исполнения — это Execution, которое Task порождает, но не заменяет собой;
- владение произведёнными Artifact — по [ADR-016](../../adr/ADR-016-artifact-aggregate-root.md) Artifact принадлежит Project напрямую, не Task; связь Task↔Artifact — опосредованная, через Execution;
- определение правил процесса (какие состояния существуют, кто за что отвечает) — это задаёт Workflow, Task лишь следует уже определённым правилам.

## Invariants

Инварианты разделены на структурные (форма/строение, неизменны по определению) и поведенческие (выполняются во времени, при любом взаимодействии).

### Structural

1. Task всегда принадлежит ровно одному Project; Task вне Project не существует ([domain-model.md](../../architecture/domain-model.md): `Project "1" o-- "0..*" Task`).
2. Task может (необязательно) принадлежать одному Epic — связь 0..1, не обязательная.
3. Task всегда имеет ровно одно текущее состояние из канонического набора девяти состояний ([state-machine.md](../../architecture/state-machine.md)) — отсутствие состояния или состояние вне этого набора невозможно.
4. Task несёт цель, scope и критерии приёмки, зафиксированные при постановке (шаблон Task) — не опциональны, хотя формат хранения (файл/БД) — переходный вопрос (см. Decision Log).

### Behavioral

1. Переходы состояния Task допустимы только в соответствии с [state-machine.md](../../architecture/state-machine.md) (Инвариант 1 того документа) — таблица переходов не дублируется этой спецификацией, используется по ссылке.
2. Переход в Blocked или Cancelled требует зафиксированной причины (state-machine.md, Инвариант 3) — эта спецификация не переопределяет условие, только фиксирует его применимость к домену Task.
3. После перехода в Archived представление Task неизменяемо (state-machine.md, Инварианты 6–7) — терминальное состояние.
4. Каждый переход публикует ровно одно событие жизненного цикла, кроме перехода Testing → Done, который публикует два последовательных события (`TestsPassed`, затем `TaskCompleted`) — согласовано с уже принятым каталогом [events.md](../../architecture/events.md), не переопределяется здесь.

## Lifecycle

Полностью канонизирован — [state-machine.md](../../architecture/state-machine.md): девять состояний (Backlog, Ready, In Progress, Review, Testing, Done, Blocked, Cancelled, Archived), правила переходов, guard-условия и инварианты процесса. Эта спецификация не дублирует state machine — она фиксирует только то, что относится к самой сущности Task (Invariants выше), а не к процессу её продвижения.

## Relationships

Task уже полностью определена разделами One Sentence/Identity/Purpose; этот раздел не меняет и не дополняет то, чем Task **является** — он описывает только то, кто и как с Task связан.

- **Project владеет Task** ([domain-model.md](../../architecture/domain-model.md): `Project "1" o-- "0..*" Task`).
- **Epic может группировать Task** (0..1 на стороне Task, `Epic "1" o-- "0..*" Task`) — опциональная связь, не обязательная для существования Task.
- **Task проходит по Workflow, но не определяет его** (`Task "0..*" --> "1" Workflow`) — использует уже заданные правила, не создаёт их.
- **Task создаёт и владеет Execution** — уже зафиксировано со стороны [Execution](execution.md) (Structural Invariant 2: Execution не существует без порождающей Task); согласовано двусторонне через Delta Review.
- **Task не владеет Artifact напрямую** ([ADR-016](../../adr/ADR-016-artifact-aggregate-root.md)) — уже зафиксировано со стороны [Artifact](artifact.md) («Task не владеет Artifact напрямую… связь опосредованная, через Execution»); эта спецификация не переопределяет и не ослабляет то решение.
- **Task ведётся в Branch и оформляется в PullRequest** — через модуль `git`, за пределами Domain Layer этого модуля; Task несёт лишь связь, не логику git-операций.

## Alternative Interpretations Considered

- **Task как атрибут Project, без собственной идентичности** — отвергнуто. Task обладает собственным процессным жизненным циклом с явным владельцем на каждом этапе ([state-machine.md](../../architecture/state-machine.md)), который не сводится к простому полю Project; несколько Task одного Project продвигаются по этому циклу полностью независимо друг от друга.
- **Task как то же самое, что Execution** — отвергнуто; уже установлено со стороны [Execution](execution.md) (Delta Review, согласовано без изменений). Task — организующая единица с девятью процессными состояниями, Execution — единичный технический факт исполнения; одна Task порождает произвольное число Execution (в том числе неудачных попыток), не переставая быть одной и той же Task.
- **Task как владелец Artifact** — отвергнуто [ADR-016](../../adr/ADR-016-artifact-aggregate-root.md). Artifact переживает породившую его Task (может оставаться ценным после её завершения или архивирования); владение закреплено за Project, чтобы завершение или архивирование Task не обесценивало произведённый результат.
- **Task как Aggregate Root** — выбрано. Единственная трактовка, корректно выражающая собственный процессный жизненный цикл, декомпозируемый из Epic, владеющий порождёнными Execution, но не владеющий произведёнными Artifact.

## Domain Events

Полный каталог — [events.md](../../architecture/events.md); не дублируется, только подтверждение согласованности с этой спецификацией (Behavioral Invariant 4). Пятнадцать событий, по одному на переход, кроме Testing → Done (два подряд).

| Событие | Переход | Источник |
| --- | --- | --- |
| TaskCreated | → Backlog | `task` |
| TaskPlanned | Backlog → Ready | `task` |
| TaskReturnedToBacklog | Ready → Backlog | `task` |
| TaskStarted | Ready → In Progress | `task` |
| ReviewRequested | In Progress → Review | `task` |
| ReviewCompleted | Review → In Progress \| Review → Testing | `git` |
| TestsFailed | Testing → In Progress | `execution` |
| TestsPassed, затем TaskCompleted | Testing → Done | `execution`, затем `task` |
| TaskBlocked | любое активное состояние → Blocked | `task` |
| TaskUnblocked | Blocked → Ready \| Blocked → In Progress | `task` |
| TaskCancelled | Backlog \| Ready \| Blocked \| In Progress → Cancelled | `task` |
| TaskArchived | Done \| Cancelled → Archived | `task` |
| MergeRequested, MergeCompleted | сопутствуют Review/Testing (порядок относительно Testing — [ADR-008](../../adr/ADR-008-git-policies.md), Decision Required) | `git` |

## Commands

Согласовано с уже принятым `Commands` ([internal/domain/task/contracts.go](../../../internal/domain/task/contracts.go)) — эта спецификация не вводит новых команд сверх того, что уже принято.

1. **Create** — регистрирует Task в Backlog. Обязательные параметры: Project, название, тип. Цель, scope и критерии приёмки, а также привязка к Epic, контрактом на сегодня отдельно не фиксируются — расширение контракта на этапе 2 (см. Decision Log).
2. **Transition** — единственный путь изменения состояния; допустимость перехода проверяется модулем `workflow` по [state-machine.md](../../architecture/state-machine.md); причина обязательна там, где это требует state-machine.md (Blocked, Cancelled).

## Queries

Согласовано с уже принятым `Queries` — намеренно узкий контракт (не межмодульный, [ADR-014](../../adr/ADR-014-module-interaction.md)).

1. **State** — текущее состояние Task. Единственный метод уже принятого контракта; более широкие запросы (по Project, по Epic, по статусу) в контракт пока не входят (см. Future Extensions).

Отдельно, вне Commands/Queries: **Exporter** — генерирует markdown-представление Task в `tasks/`; не Command и не Query в доменном смысле, а специфичный для переходного периода механизм экспорта ([ADR-004](../../adr/ADR-004-task-storage.md)) — не источник истины.

## Examples

Ни кода, ни JSON — только содержательные примеры, показывающие, что модель выдерживает разнородные реальные случаи.

- **Обычная документационная задача** — эта самая спецификация (TASK-032): Backlog → Ready → In Progress (порождает Execution для агента-исполнителя) → Review → Testing → Done.
- **Заблокированная задача** — Task, ожидающая внешнего архитектурного решения (например, требует ещё не принятого ADR). In Progress → Blocked, причина и требуемое решение зафиксированы (Behavioral Invariant 2).
- **Отменённая до начала задача** — требования устарели прежде, чем работа началась. Backlog → Cancelled.
- **Задача, вернувшаяся на доработку** — Review → In Progress (вердикт «changes requested») → снова Review → Testing.
- **Задача без Epic** — самостоятельный срочный bugfix, не входящий ни в один Epic (Structural Invariant 2: связь опциональна).

## Acceptance Criteria

Критерии проверяют качество модели, а не факт заполнения разделов; каждый — с самопроверкой против этой спецификации.

- [x] **Сущность имеет единственную ответственность** — Task отвечает только за организацию процесса (цель, scope, критерии приёмки, продвижение по состояниям); не совмещает с техническим исполнением (Execution) или владением результата (Artifact).
- [x] **Все инварианты проверяемы** — каждый Structural/Behavioral инвариант формулирует условие, подтверждаемое или опровергаемое на конкретной Task.
- [x] **Жизненный цикл не содержит неоднозначных переходов** — полностью делегирован каноническому [state-machine.md](../../architecture/state-machine.md), не переопределён и не продублирован этой спецификацией.
- [x] **Команды не нарушают инварианты** — Transition проверяется модулем `workflow` по state-machine.md; Create требует Project (Structural Invariant 1).
- [x] **События являются следствием переходов состояния** — все 15 согласованы с уже принятым каталогом [events.md](../../architecture/events.md), таблица подтверждает соответствие, не переопределяет.
- [x] **Нет противоречий с принятыми ADR** — согласовано с [ADR-004](../../adr/ADR-004-task-storage.md) (PostgreSQL — источник истины), [ADR-016](../../adr/ADR-016-artifact-aggregate-root.md) (Task не владеет Artifact); [ADR-011](../../adr/ADR-011-task-identifiers.md) зафиксирован как открытая зависимость, не решается здесь.
- [x] **Спецификация не требует знания Go-кода или реализации** — ни один из трёх PR не содержит упоминаний Go/структур/реализации сверх ссылок на уже принятые контракты по имени.
- [x] **Согласована с Artifact, Execution и Executor** (Delta Review) — без переопределения уже принятых решений об этих сущностях.

## Future Extensions

Разделены на ожидаемые (вероятны с высокой уверенностью, модель сознательно оставляет для них пространство) и возможные (могут появиться, но пока не имеют достаточных оснований для проектирования).

### Ожидаемые

- **Расширенные Queries** (по Project, по Epic, по статусу, листинг) — естественное продолжение уже принятого узкого контракта `State`.
- **Явные Commands для записи/обновления scope и критериев приёмки** отдельно от Create, и параметр `epicID` в Create — решённое направление расширения контракта на этапе 2 (см. Decision Log).
- **Формат идентификаторов** после принятия [ADR-011](../../adr/ADR-011-task-identifiers.md).

### Возможные

- **Полноценная спецификация Epic** как самостоятельной сущности со своим Lifecycle и Invariants — сейчас Epic упомянут только в Relationships этой спецификации, вне scope TASK-032.
- **Приоритезация Task** внутри Backlog/Ready.
- **SLA или сроки** на отдельных этапах жизненного цикла.

## Anti-Responsibilities

- Task **не выполняет техническую работу сама** — это Execution, которое Task порождает.
- Task **не определяет правила процесса** (какие состояния и переходы допустимы) — это задаёт Workflow, Task лишь следует уже определённым правилам.
- Task **не владеет произведёнными Artifact** — владение закреплено за Project ([ADR-016](../../adr/ADR-016-artifact-aggregate-root.md)).
- Task **не хранит и не интерпретирует git-механику** (сам Pull Request, ход ревью) — это модуль `git`; Task лишь несёт связь с Branch/PullRequest.
- Task **не решает, кто персонально назначен исполнителем** — подбор исполнителя относится к Application/Orchestrator.

## Non-Goals

Сформулированы как сознательные ограничения этой версии, а не как список отсутствующих функций.

- В этой версии Task не специфицирует Epic как самостоятельную сущность со своим Lifecycle и Invariants — вне scope этой задачи.
- В этой версии Task не вводит операции записи scope/критериев приёмки отдельно от Create — расширение контракта запланировано на этапе 2 (см. Decision Log), не в scope этой спецификации.
- В этой версии Task не определяет приоритезацию внутри Backlog/Ready.
- В этой версии Task не определяет SLA или сроки на отдельных этапах жизненного цикла.

## Removal Test

Если убрать Task из модели, платформа теряет организующую единицу процесса, отдельную от факта исполнения и от результата: работу пришлось бы организовывать либо напрямую вокруг Execution — стирая грань между организацией работы и отдельными техническими попытками её выполнения, и лишая Epic возможности декомпозироваться на что-либо, кроме отдельных запусков, — либо напрямую вокруг Artifact, смешивая «то, что нужно сделать» с «тем, что уже произведено», при том что многие Task (например, чисто исследовательские) не производят Artifact вовсе. Именно эта потеря — невозможность организовать и отследить объём работы с явным владельцем на каждом этапе, независимо от техники исполнения и результата, — оправдывает Task как самостоятельный Aggregate Root.

## Open Questions

- **Формат идентификатора Task/Epic** — [ADR-011](../../adr/ADR-011-task-identifiers.md), Decision Required; эта спецификация только фиксирует зависимость, не блокирует утверждение.
- **Приоритезация внутри Backlog/Ready** — не решена, возможный кандидат Future Extensions, а не текущая ответственность Task.

### Three-Pass Review — ответы на диагностические вопросы

- **Pass 1, самое слабое место модели:** связь Task↔Epic (Structural Invariant 2) заявлена как опциональная, но уже принятый контракт `Create` вообще не даёт механизма её установить — инвариант, для которого сейчас нет пути исполнения через принятый код.
- **Pass 2, сильнее всего влияющая соседняя сущность:** Execution — именно порождение Execution при переходе в In Progress связывает организующую сторону (Task) с фактическим исполнением; второе по силе влияние — Artifact, через явный запрет владения ([ADR-016](../../adr/ADR-016-artifact-aggregate-root.md)).
  - Delta Review: (1) *Требует ли новая спецификация изменить уже утверждённую/черновую модель?* Нет — Task использует уже зафиксированные Artifact/Execution/Executor стороны связей, не меняет их. (2) *Использует ли она уже утверждённые понятия одинаковым образом?* Да — «владеет»/«ссылается», «породившее Execution» применены согласованно. (3) *Не вводит ли она новое понятие, уже существующее под другим названием?* Нет — Epic уже существовал в [domain-model.md](../../architecture/domain-model.md) до этой спецификации, здесь только используется по ссылке, не переопределяется.
- **Pass 3, наиболее вероятное будущее изменение, требующее пересмотра:** расширение `internal/domain/task` на этапе 2 (`epicID`, операции scope/AC — уже решённое направление, см. Decision Log) и принятие [ADR-011](../../adr/ADR-011-task-identifiers.md) — оба, вероятно, потребуют локального пересмотра формулировок Commands, но не самой модели (Structural Invariant 2 и 4 уже корректно предвосхищают оба расширения).
