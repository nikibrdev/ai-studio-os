# Стратегия тестирования

## Назначение

Определяет уровни тестирования и требования к тестам в AI Studio OS. На этапе v0.1 Foundation создание тестов запрещено; документ задаёт правила для последующих этапов.

## Содержание

### Уровни тестирования

| Уровень | Что проверяет | Инструменты |
| --- | --- | --- |
| Unit | Доменная логика в изоляции | стандартный `go test`; фреймворк для frontend не выбран |
| Интеграционные | Взаимодействие модулей и инфраструктуры (PostgreSQL, Redis) | `go test` + Docker-окружение |
| End-to-End | Пользовательские сценарии через интерфейс | Playwright |

### Требования

1. Тестируемость закладывается в дизайн: зависимости — через интерфейсы, подмена — без монки-патчинга и магии.
2. Критическая бизнес-логика покрывается unit-тестами обязательно.
3. Тест читается как спецификация: имя описывает сценарий и ожидание.
4. Тесты детерминированы: без зависимости от времени, сети и порядка выполнения.
5. Интеграционные тесты используют реальные зависимости в контейнерах, а не моки инфраструктуры.
6. С момента включения тестирования в scope (v0.4 QA Engine) тесты — часть Definition of Done.

### Роль QA Engineer

QA Engineer (человек или агент) проверяет соответствие результата критериям приёмки, полноту тестов и прогоняет end-to-end сценарии. Обязанности роли: [.claude/agents/qa.md](../../.claude/agents/qa.md).

### Открытые вопросы

- Фреймворк тестирования frontend (unit-уровень) — [ADR-009](../adr/ADR-009-toolchain.md).
- Целевые пороги покрытия кода не определены (решение — при проектировании v0.4 QA Engine).
- Организация тестовых окружений не спроектирована (после снятия ограничений Foundation; связано с [ADR-006](../adr/ADR-006-agent-execution-environment.md)).
