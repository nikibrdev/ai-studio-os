# ADR-012: Модель пользователей и аутентификации

## Статус

**Принято** (решение архитектора проекта, 2026-07-22) — **Вариант 1**.

## Дата

2026-07-19 (заготовка); решение принято 2026-07-22 при открытии [EPIC-008](../roadmap/EPIC-008-api-layer.md) (v0.9 API)

## Проблема

Доменная модель содержит User, Role, Session ([domain-model.md](../architecture/domain-model.md)), но не определено: когда вводится аутентификация, каким способом, как авторизуются действия ролей и как аутентифицируются агенты (их действия должны атрибутироваться).

## Контекст

- До v0.6 (Dashboard) человеческий доступ — файлы и git; собственная аутентификация платформы фактически не нужна.
- С v0.6 Dashboard и API требуют аутентификации и привязки действий к User.
- Действия агентов тоже атрибутируются (инициатор в событиях) — механизм их идентификации нужен раньше UI.
- GitHub уже является общим знаменателем для всех участников.

## Возможные варианты

### Вариант 1: Отложить identity до v0.6; до этого — доверенная однопользовательская установка

- **Плюсы:** MVP быстрее; KISS — не строить auth без UI.
- **Минусы:** события до v0.6 атрибутируются условно; переделка проекций при вводе User.

### Вариант 2: Минимальный identity с v0.2 (локальные пользователи, токены), полноценный auth в v0.6

- **Плюсы:** атрибуция действий с самого начала; модель данных стабильна.
- **Минусы:** ранняя реализация при отсутствии UI; часть работы вслепую.

### Вариант 3: OAuth через GitHub с v0.6, идентификаторы GitHub — с v0.2

- **Плюсы:** без собственного хранения паролей; GitHub-логины уже есть у всех участников; ранняя атрибуция через GitHub-идентификаторы.
- **Минусы:** жёсткая привязка к GitHub; оффлайн/enterprise-сценарии потребуют альтернативы.

## Решение

Принят **Вариант 1**: identity/аутентификация откладывается. `apps/api` (v0.9) и будущий `apps/dashboard` (v0.8, строится после API — [decision](../../engineering/decisions/2026-07-22-api-before-dashboard-build-order.md)) работают в модели **доверенной однопользовательской установки** — без собственной аутентификации запросов. Тот же принцип, что решение по эмбеддингам Memory (ADR-018) и отказ от раннего Orchestrator (EPIC-006): не строить механизм, для которого пока нет реального внешнего потребителя/пользователя — платформа сейчас эксплуатируется одним доверенным оператором, у внешних потребителей API ещё нет.

Атрибуция инициатора в событиях (`actor`, уже параметр каждого use-case-метода Application Layer, EPIC-004) продолжает определяться вызывающим кодом (человек за терминалом или Executor), а не механизмом аутентификации запроса — этот механизм не меняется этим решением. Модуль `identity` (User/Session как отдельные доменные сущности) не вводится.

Пересмотр — при появлении первого реального внешнего потребителя API или многопользовательского сценария; тогда более уместен Вариант 3 (GitHub OAuth), учитывая, что GitHub уже общий знаменатель проекта — эта оценка не меняется, просто её реализация откладывается до реальной потребности.

## Влияние на систему

Состав модуля `identity` и его наличие в MVP — не вводится в v0.9/v0.8. Поле «инициатор» в событиях ([events.md](../architecture/events.md)) — не затронуто, определяется вызывающим кодом, как и раньше. API (аутентификация запросов, [ADR-003](ADR-003-api-protocol.md)) — без аутентификации в первой версии. Dashboard (v0.8) — тот же доверенный доступ, без экрана логина. Модель прав ролей — не вводится до появления identity.
