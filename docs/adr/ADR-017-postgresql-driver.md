# ADR-017: Драйвер PostgreSQL

## Статус

**Принято** (решение архитектора проекта, 2026-07-21)

## Дата

2026-07-21

## Контекст

[stack.md](../../.claude/context/stack.md) фиксирует PostgreSQL как технологию — источник истины задач ([ADR-004](ADR-004-task-storage.md)) и место хранения журнала событий ([ADR-002](ADR-002-event-delivery.md)). Стандартная библиотека Go не содержит драйвера PostgreSQL — конкретный пакет не выбран. Решение нужно до начала EPIC-005 (Infrastructure Layer, v0.5): адаптеры хранения (`internal/infrastructure`) не могут быть написаны без выбранного способа доступа к БД. Новая зависимость требует ADR по правилу `stack.md`: «Новая библиотека или сервис — только через ADR.»

## Рассмотренные варианты

### Вариант 1: `database/sql` + `lib/pq`

Классическая пара: стандартный интерфейс `database/sql` с драйвером `github.com/lib/pq`. Плюсы: `database/sql` — знакомый идиоматичный интерфейс. Минусы: `lib/pq` официально в режиме поддержки без развития («not accepting new features», по заявлению самого проекта), нет нативного пулинга — нужен `database/sql`'s встроенный (менее гибкий, чем специализированный).

### Вариант 2: `pgx` (нативный интерфейс, `pgxpool`)

`github.com/jackc/pgx/v5` — активно поддерживаемый драйвер, де-факто стандарт экосистемы Go для PostgreSQL. Нативный интерфейс (не через `database/sql`) даёт прямой доступ к специфичным для Postgres возможностям (COPY, нативные типы, LISTEN/NOTIFY на будущее) и собственный пул соединений (`pgxpool.Pool`) без дополнительной зависимости. Минус: не подчиняется абстракции `database/sql`, код адаптеров работает конкретно с `pgx`, а не с любым SQL-драйвером — но проект уже не закладывается на смену СУБД (PostgreSQL зафиксирован ADR-004), так что это не ограничение на практике.

### Вариант 3: `pgx` в режиме совместимости с `database/sql` (`pgx/v5/stdlib`)

`pgx`, зарегистрированный как драйвер `database/sql`. Совмещает активную поддержку `pgx` со стандартным интерфейсом. Минус: теряется часть нативных возможностей `pgx` (typed params, batch, пул) без выигрыша, так как проект не нуждается в переносимости между разными SQL-СУБД.

## Решение

Принят **Вариант 2**: `github.com/jackc/pgx/v5`, доступ через `pgxpool.Pool`.

Обоснование:

- активно поддерживаемый пакет, стандарт де-факто для Go + PostgreSQL;
- собственный пул соединений — не нужна отдельная зависимость или обёртка;
- проект не абстрагируется от конкретной СУБД (PostgreSQL зафиксирован ADR-004) — терять переносимость `database/sql` не жаль, выигрыш от неё нулевой.

Миграции схемы — не через отдельную библиотеку (`golang-migrate` и подобные не вводятся), а через минимальный самописный раннер: `.sql`-файлы, встроенные `embed.FS`, применяются по возрастанию имени, применённые версии — в служебной таблице `schema_migrations`. Обоснование: единственная используемая возможность таких библиотек в проекте — применить пронумерованные `.sql`-файлы один раз при старте; вводить внешнюю зависимость ради этого нецелесообразно (тот же принцип, что и решение не вводить UUID-библиотеку в EPIC-004 — `internal/application/id.go`).

## Последствия

### Положительные

- Единственная новая зависимость на весь Infrastructure Layer для доступа к БД.
- Прямой контроль над пулом соединений и SQL — нет прослойки ORM.
- Миграции не требуют внешнего CLI или зависимости — `go run`/тесты видят один и тот же встроенный набор `.sql`.

### Отрицательные

- Код адаптеров `internal/infrastructure` привязан к API `pgx`, а не к абстрактному `database/sql` — смена драйвера потребует переписать адаптеры (не абстракцию), но смена СУБД уже не предполагается проектом.
- Самописный раннер миграций не имеет возможностей промышленных инструментов (down-миграции, dry-run, параллельная защита через advisory lock и т.п.) — при реальной необходимости будет пересмотрено отдельным решением.

### Влияние на существующие документы и код

`go.mod` — добавлена зависимость `github.com/jackc/pgx/v5`; [stack.md](../../.claude/context/stack.md) — строка PostgreSQL дополнена указанием драйвера; `internal/infrastructure` (EPIC-005) — реализуется на этом драйвере.
