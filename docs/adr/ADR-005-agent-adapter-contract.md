# ADR-005: Контракт адаптера агента

## Статус

**Decision Required** — ключевое решение этапа v0.3 (Developer Engine): формат обмена между платформой и агентом определяет реализацию первого адаптера (Claude Code) и всех последующих. Требуется решение архитектора: варианты по-разному ограничивают круг подключаемых провайдеров.

## Дата

2026-07-19 (заготовка)

## Проблема

Концептуальный контракт Agent описан ([interfaces.md](../architecture/interfaces.md)), но не выбран технический способ обмена: как платформа передаёт агенту назначение (задачу, контекст, инструменты) и как получает события хода работы и Result.

## Контекст

- Первый адаптер — Claude Code (CLI-инструмент, работает с файловой системой и git; поддерживает программный запуск через Agent SDK).
- Будущие адаптеры: OpenAI Codex, OpenHands, Gemini — у всех разные интерфейсы (CLI, API, контейнеры).
- Ядро не должно знать о провайдерах ([agents.md](../architecture/agents.md)); добавление провайдера — только новый адаптер.
- Контекст назначения включает документы проекта, файл задачи, правила — уже существующие в репозитории артефакты.

## Возможные варианты

### Вариант 1: Файловый контракт + запуск CLI

Назначение материализуется в файлах (задача, контекст, инструкции), адаптер запускает CLI-агента в рабочем каталоге; результат собирается из файлов (отчёт в задаче, git-изменения).

- **Плюсы:** идеально ложится на Claude Code и текущий файловый процесс; человекочитаемо; минимум связующего кода.
- **Минусы:** события хода работы ограничены (парсинг вывода); слабая структурная валидация; для API-провайдеров придётся эмулировать файловую среду.

### Вариант 2: Структурированный программный контракт (запрос/ответ + поток событий)

Единый структурированный формат назначения и результата; адаптер транслирует его в native-интерфейс провайдера (SDK, API, CLI).

- **Плюсы:** строгий контракт и валидация; единообразные события хода работы; проще подключать API-провайдеров.
- **Минусы:** больше связующего кода для CLI-агентов; риск переусложнения на этапе одного провайдера.

### Вариант 3: Гибрид — структурированный контракт, файловая доставка контекста

Назначение и Result структурированы; объёмный контекст (документы, репозиторий) передаётся ссылкой на рабочий каталог.

- **Плюсы:** валидируемый контракт без дублирования файлового контекста; естественно для git-процесса.
- **Минусы:** два канала данных — сложнее описать и тестировать.

## Влияние на систему

Реализация `agents/` (первый адаптер — Claude Code); модуль `execution` (создание Execution, приём Result); Orchestrator; формат событий работы агента ([events.md](../architecture/events.md)); связано со средой запуска ([ADR-006](ADR-006-agent-execution-environment.md)).
