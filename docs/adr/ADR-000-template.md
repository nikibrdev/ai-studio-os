# ADR-000: Шаблон архитектурного решения

## Назначение

Шаблон для оформления Architecture Decision Record. Копируется при создании нового ADR: файл именуется `ADR-NNN-short-name.md` со сквозной нумерацией. Когда требуется ADR — см. [CONSTITUTION.md](../../CONSTITUTION.md).

## Содержание

```markdown
# ADR-NNN: <Название решения>

## Статус

<Предложено | Принято | Отклонено | Superseded (заменено ADR-MMM)>

## Дата

YYYY-MM-DD

## Контекст

<Какая проблема решается. Какие требования и ограничения действуют.
Почему решение нужно принять сейчас.>

## Рассмотренные варианты

### Вариант 1: <название>

<Суть, плюсы, минусы.>

### Вариант 2: <название>

<Суть, плюсы, минусы.>

## Решение

<Выбранный вариант и обоснование выбора.>

## Последствия

### Положительные

<Что улучшается.>

### Отрицательные

<Какие издержки и риски принимаются.>

### Влияние на существующие документы и код

<Какие документы, контракты и модули требуют обновления.>

## Связанные материалы

<Ссылки на задачи, ADR, документы, обсуждения.>
```

### Правила

1. ADR неизменяем после принятия: пересмотр решения — это новый ADR, переводящий старый в статус Superseded.
2. Один ADR — одно решение.
3. ADR пишется до реализации решения (Documentation First).
4. AI-агенты создают ADR только по явному поручению.
5. **Решение принимается, только когда оно практически необратимо.** Если решение может измениться в обозримое время — оно остаётся в статусе Decision Required, проектирование продолжается, зависящий от него код не пишется ([правило](../../engineering/decisions/2026-07-19-irreversible-decisions.md)).
