# ADR-006: Среда выполнения и изоляция агентов

## Статус

**Принято** (решение архитектора проекта, 2026-07-21)

## Дата

2026-07-21 (заготовка — 2026-07-19; принято — 2026-07-21)

## Проблема

Не было определено, где и как выполняются AI-агенты: в каком окружении запускается процесс агента, к чему он имеет доступ и как ограничивается ([agents.md](../architecture/agents.md), открытый вопрос «модель прав и ограничений агентов»).

## Контекст

- Агент (Claude Code) нуждается в: рабочей копии репозитория, git, инструментах сборки/тестов, сети до своего AI-провайдера.
- Агент не должен иметь доступа к: чужим проектам, секретам платформы, произвольной сети (желательно).
- Docker входит в утверждённый стек; Docker Compose на этапе Foundation был запрещён — ограничение снято переходом к реализации (v0.3 завершён).
- Один хост в MVP; массовый параллелизм агентов не требуется до v0.9.

## Рассмотренные варианты

### Вариант 1: Локальный процесс на хосте платформы

- **Плюсы:** минимальная сложность; прямой доступ к рабочим каталогам; простая отладка.
- **Минусы:** нет изоляции — агент видит всё, что видит процесс платформы; конфликты окружений инструментов; риск для секретов.

### Вариант 2: Docker-контейнер на исполнение

Каждое Execution — контейнер с рабочей копией, инструментами и ограниченной сетью.

- **Плюсы:** изоляция файловой системы и сети; воспроизводимое окружение; чистое завершение.
- **Минусы:** подготовка образов; передача git-креденшелов внутрь; медленнее старт; сложнее интерактивная отладка.

### Вариант 3: Внешний сервис исполнения (удалённые раннеры)

- **Плюсы:** масштабирование и полная изоляция от хоста платформы.
- **Минусы:** несоразмерная MVP сложность (очереди, доставка кода, безопасность канала); преждевременно до v0.9.

## Решение

Принят **вариант 2 — Docker-контейнер на исполнение** как целевая модель запуска агентов платформой (реализация — v0.6 AI Agent Runtime), с явно оформленным переходным режимом:

1. **Целевая модель (v0.6+):** одно Execution — один контейнер. Внутри: рабочая копия ветки задачи (клонируется при старте, уничтожается при завершении — согласовано с жизненным циклом Execution: Queued → Running → терминальное состояние), инструменты сборки/тестов из воспроизводимого образа (базой служит уже проверенный Dev Container), сеть ограничена allowlist'ом: AI-провайдер агента + git-хостинг; произвольный исходящий трафик запрещён.
2. **Секреты:** короткоживущий git-токен с минимальным scope (один репозиторий, ветки задач) передаётся контейнеру переменной окружения при старте и не сохраняется ни в образе, ни в артефактах; ключ AI-провайдера — аналогично. Секреты платформы (БД, ключи других провайдеров) в контейнер агента не попадают никогда.
3. **Границы доступа:** контейнер видит только рабочую копию своего задания — никаких других проектов и данных платформы (соответствует изоляции знания Project и запрету Executor → Database, ADR-014).
4. **Переходный режим (до v0.6):** агент выполняется локальным процессом **под интерактивным надзором человека** (текущая практика: Claude Code в терминале разработчика) — это режим разработки самой платформы, а не платформенный запуск; на него ограничения контейнера не распространяются, ответственность за границы несёт человек за терминалом. Платформа никогда не запускает агентов локальным процессом без надзора — это единственный жёсткий запрет переходного периода.
5. **Вариант 3** (удалённые раннеры) не отвергнут навсегда: контракт Executor (ADR-005) и модель Execution не зависят от места запуска контейнера — переход к удалённым раннерам при масштабировании (v0.9+) не потребует пересмотра доменной модели, только адаптера запуска.

## Последствия

### Положительные

- Снят блокер v0.6 Developer Engine: модель запуска, границы доступа и обращение с секретами определены до начала реализации Runtime.
- Дизайн тестовых окружений (открытый вопрос [testing.md](../development/testing.md)) получает основание: интеграционные тесты — контейнеры, как и исполнение агентов.
- Изоляция по построению согласована с уже принятыми правилами (ADR-014: Executor не имеет доступа к хранилищу платформы; изоляция Memory по Project).

### Отрицательные

- Подготовка и поддержка образа исполнения — новая операционная обязанность (смягчение: база — уже существующий Dev Container).
- Старт исполнения медленнее локального процесса — приемлемо: скорость старта Execution не является узким местом процесса, где ревью и CI занимают минуты.
- Интерактивная отладка исполнения сложнее — компенсируется журналом Execution и переходным режимом для разработки самой платформы.

### Влияние на существующие документы и код

Модуль `execution` — изменений не требует (модель Queued → Running → терминальное уже согласована с жизненным циклом контейнера); адаптеры `agents/` (v0.6) реализуют запуск контейнера; операционные требования (`docker/`, `docs/operations/`) — при реализации v0.6; модель безопасности Tool Layer ([tools.md](../architecture/tools.md)) наследует сетевые границы контейнера.
