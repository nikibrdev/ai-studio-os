# ADR-008: Git-политики

## Статус

**Принято** (решение архитектора проекта, 2026-07-21)

## Дата

2026-07-21 (заготовка — 2026-07-19; принято — 2026-07-21)

## Проблема

Не были зафиксированы: способ слияния PR, правила защиты `main`, число обязательных ревьюеров, допустимость AI-агента как единственного ревьюера, момент слияния относительно стадии Testing, подпись коммитов, шаблоны Issue ([git-workflow.md](../development/git-workflow.md), [review-process.md](../development/review-process.md), [state-machine.md](../architecture/state-machine.md)).

## Контекст

- GitHub Flow и Conventional Commits уже приняты как процесс ([git-workflow.md](../development/git-workflow.md)).
- Каноническая state machine вводит Testing после Review; где в этой цепочке слияние — не было определено.
- Большинство PR создаются AI-агентами; ревью человеком — узкое место, ревью агентом — вопрос доверия.
- Защита `main` (запрет прямых пушей, обязательный статус-чек `verify`, `enforce_admins`) уже действует с EPIC-002.5 и практически подтверждена (отклонённый прямой push при закрытии этапа 1 EPIC-003).

## Рассмотренные варианты

### Способ слияния

- **Squash:** линейная история, один коммит на задачу (плюс: чистота; минус: теряется структура коммитов агента).
- **Merge commit:** полная история (плюс: прослеживаемость; минус: шумная история).
- **Rebase:** линейность без squash (минус: переписывание истории).

### Момент слияния относительно Testing

- **Слияние после Testing:** QA проверяет ветку; в `main` попадает только проверенное (плюс: чистый `main`; минус: тестируется не итоговое состояние `main`).
- **Слияние после Review, Testing на `main`:** проверяется реальный результат слияния (плюс: тестируется итог; минус: непроверенный код временно в `main`).

### Ревью

- 1 обязательный ревьюер / 2 ревьюера / человек обязателен для всех PR / агент-ревьюер допустим с эскалацией к человеку.

## Решение

1. **Способ слияния — merge commit.** Структура коммитов задачи — часть инженерной истории (пример: три PR одной спецификации, раунды правок после ревью — эта прослеживаемость уже доказала ценность в EPIC-003); squash уничтожал бы её. Заголовок merge-коммита — по Conventional Commits со ссылкой на PR (`... (#NN)`), как уже сложилось.
2. **Момент слияния — после Testing.** QA проверяет ветку задачи; в `main` попадает только проверенное — критично для платформы, где агенты клонируют `main` как отправную точку новой работы. Риск «тестировалась ветка, а не результат слияния» закрывается требованием актуальности ветки относительно базы перед merge (GitHub «Require branches to be up to date» — уже проявилось на практике в PR #42) плюс обязательным прогоном `verify` на `main` после каждого слияния.
   Порядок событий при завершении задачи: `TestsPassed` → `MergeCompleted` → `TaskCompleted`; guard перехода Testing → Done включает факт слияния PR.
3. **Ревью — один обязательный ревьюер; агент-ревьюер допустим** (роль Reviewer может исполнять человек или агент — соответствие модели ролей платформы). Эскалация к человеку — по запросу любой стороны или для изменений Конституции/ADR. Техническое ограничение переходного периода: в защите `main` число обязательных approve остаётся 0, пока у репозитория один мейнтейнер (self-approval GitHub не засчитывает — TASK-016); обязательность ревью обеспечивается стадией Review канонической state machine, а не настройкой GitHub. При появлении второго мейнтейнера/бота-ревьюера — поднять до 1 в настройках.
4. **Прочее:** подпись коммитов в MVP не требуется (источник изменений устанавливается процессом: PR + CI + защита веток); шаблоны Issue (bug/feature/task) уже введены (EPIC-002.5); прямые пуши в `main` запрещены без исключений (уже действует).

## Последствия

### Положительные

- Условие перехода Testing → Done наконец определено — снят блокер оркестрации Application Layer (v0.4).
- Зафиксирован уже фактически действующий процесс (merge commit, ветка-на-задачу, обязательный `verify`) — без миграции.
- Роль Reviewer официально исполнима агентом — соответствует видению платформы и разблокирует полный автоматический цикл в v0.6.

### Отрицательные

- Merge commit даёт более шумную историю `main` — принято сознательно ради прослеживаемости.
- До появления второго мейнтейнера обязательность ревью — процессная, не техническая; фиксируется как известное ограничение.
