# ADR-018: Модель эмбеддингов и схема коллекции Qdrant для Memory System

## Статус

**Принято** (решение архитектора проекта, 2026-07-22)

## Дата

2026-07-22

## Контекст

[memory.md](../architecture/memory.md) фиксирует переход на Qdrant для семантического поиска с v0.7 как открытый вопрос: «Модель эмбеддингов и схема коллекций Qdrant — решение этапа v0.7, потребует ADR.» Qdrant — векторная база данных: она хранит и ищет по векторам, но не порождает их из текста. Нужен отдельный источник эмбеддингов.

`stack.md` не называет ни одного провайдера эмбеддингов (OpenAI, Voyage, Cohere и т.п.) — только Qdrant как хранилище. У текущей сессии нет ключа ни к одному внешнему провайдеру эмбеддингов (та же ситуация, что с `ANTHROPIC_API_KEY` в EPIC-006) — извлечение/переиспользование учётных данных сессии для этой цели не рассматривается. Пользователю задан прямой вопрос; выбор — наивный локальный эмбеддинг сейчас, без внешней зависимости.

## Рассмотренные варианты

### Вариант 1: Внешний API эмбеддингов (OpenAI, Voyage и т.п.)

Реальное семантическое качество. Требует: ключ (недоступен в этой сессии), новую внешнюю зависимость в `stack.md` через отдельный ADR, сетевой вызов на каждую запись/поиск. Отклонено — блокируется отсутствием ключа и решения о провайдере.

### Вариант 2: Локальная нейросетевая модель (ONNX/аналоги)

Реальное семантическое качество без внешнего API. Требует: новую тяжёлую зависимость (ONNX Runtime или аналог), скачивание весов модели, отсутствующую в стеке инфраструктуру инференса. Несоразмерно объёму этого решения — отклонено для MVP.

### Вариант 3: Наивный детерминированный локальный эмбеддинг (feature hashing)

Классический hashing trick: токенизация содержимого, каждый токен хешируется в индекс фиксированномерного вектора, вектор нормализуется (L2). Без нейросети, без внешних вызовов, без новых зависимостей — только `hash/fnv` и `math` стандартной библиотеки.

- **Плюсы:** реальная работающая инфраструктура (Qdrant, коллекция, поиск по вектору) целиком проверяема уже сейчас; ноль внешних зависимостей и секретов; детерминированность упрощает тесты.
- **Минусы:** не улавливает семантику (синонimy, перефразирование) — по сути это разреженное представление совпадения токенов, а не смысла. Осознанно принимается как честное ограничение MVP.

## Решение

Принят **вариант 3**.

**Эмбеддинг:** `internal/infrastructure/memory` вычисляет вектор фиксированной размерности **256** через feature hashing по словам содержимого (нижний регистр, разбиение по границам не-буквенных символов); каждый токен хешируется (`fnv-1a`) в индекс `[0, 256)`, соответствующая координата увеличивается на 1 (со знаком от отдельного бита хеша — классический signed hashing trick, уменьшающий коллизионное смещение); итоговый вектор L2-нормализуется. Метрика близости в Qdrant — косинусная.

Функция эмбеддинга изолирована в одном месте (`embedding.go`, экспортируемая сигнатура `func embed(text string) []float32`) — замена на реальный провайдер в будущем ADR потребует правки одной функции, не схемы коллекции или контракта `MemoryProvider`.

**Схема Qdrant:** одна коллекция `memory_entries` на весь Instance (не по коллекции на проект — по аналогии с тем, как `project_id` в PostgreSQL — колонка, а не отдельная таблица на проект, EPIC-005). Точка: `id` — UUID v4 (генерируется из `crypto/rand`, форматируется как строка UUID — Qdrant принимает точки с ID в виде беззнакового целого или UUID; последнее выбрано, так как остальные строковые идентификаторы проекта уже строки). Payload: `project_id`, `kind`, `content`, `source`, `recorded_at` — самодостаточен для реконструкции `platform.MemoryEntry` без обращения к файлам (аналогично тому, как `event_journal` самодостаточен для перестроения проекций, EPIC-005). Вектор — 256 float32, косинусная близость.

**Источник истины и переиндексация:** файлы в `memory/` (TASK-059) — долговременное, аудируемое хранилище (принцип «Прозрачность», memory.md); Qdrant — производный, перестраиваемый индекс. `Reindex`/пересборка коллекции из файлов — часть TASK-061, тот же принцип, что `eventbus.ReadJournal`/`TaskProjection.Rebuild` (EPIC-005).

## Последствия

### Положительные

- Вся инфраструктура (коллекция, upsert, поиск, изоляция по `project_id`) реальна и проверяема вживую уже в этом эпике — не макет, ожидающий будущего провайдера.
- Ноль новых зависимостей и секретов.
- Замена эмбеддинга на реальную модель в будущем — локализованное изменение одной функции.

### Отрицательные

- Поиск не семантический по сути — совпадение по хешированным токенам, не по смыслу. Явно задокументированное ограничение, не скрытое за формулировкой «семантический поиск».
- 256-мерный хешированный вектор при большом словаре знаний одного проекта может давать коллизии токенов чаще, чем «настоящий» эмбеддинг с сотнями измерений, полученный обучением. Приемлемо для объёма знаний одного проекта на этом этапе.

### Влияние на существующие документы и код
