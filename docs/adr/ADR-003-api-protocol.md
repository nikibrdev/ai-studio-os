# ADR-003: Протокол API

## Статус

**Принято** (решение архитектора проекта, 2026-07-19)

## Дата

2026-07-19

## Контекст

`apps/dashboard` взаимодействует с платформой только через `apps/api` ([module-boundaries.md](../architecture/module-boundaries.md)); протокол требовалось зафиксировать до проектирования API-слоя и Dashboard.

## Рассмотренные варианты

1. **REST + OpenAPI** — универсальность, читаемость, минимальный порог входа.
2. **gRPC (+ gRPC-Web)** — строгие контракты и стриминг ценой прокси для браузера и сложности.
3. **GraphQL** — гибкие выборки ценой дополнительного слоя сложности.

## Решение

Для MVP используется **REST API**.

Схема взаимодействия:

```
Dashboard → REST → Go Core
```

Для внутреннего взаимодействия gRPC не используется — внутри монолита модули взаимодействуют через события и контракты Core ([ADR-014](ADR-014-module-interaction.md)), между приложением и ядром — прямые вызовы контрактов, сетевой протокол не нужен.

## Последствия

### Положительные

- Простой и универсальный внешний интерфейс; естественная интеграция с Next.js.
- Документируемость через OpenAPI; низкий порог для будущих внешних потребителей.
- Никакой сетевой инфраструктуры внутри монолита.

### Отрицательные

- Для доставки событий в UI в реальном времени потребуется дополнительный канал (SSE/WebSocket) — проектируется вместе с Dashboard (v0.6).
- Типизация клиента — через генерацию из OpenAPI (настройка на этапе реализации API).

### Влияние на существующие документы и код

`apps/api` проектируется как REST-слой; спецификации — в `docs/api/` по шаблону [API.md](../../.claude/templates/API.md); [system-design.md](../architecture/system-design.md) и [docs/api/README.md](../api/README.md) обновлены.
