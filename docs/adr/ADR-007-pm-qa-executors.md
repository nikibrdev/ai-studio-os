# ADR-007: Исполнители ролей Project Manager и QA Engineer в MVP

## Статус

**Принято** (решение владельца продукта, 2026-07-23) — **Вариант 2**.

## Дата

2026-07-19 (заготовка); принято — 2026-07-23, при открытии v1.0 First Public MVP

## Проблема

MVP включает три роли; Developer по умолчанию — Claude Code. Кто по умолчанию исполняет роли Project Manager и QA Engineer в MVP — человек, агент или гибрид — не определено ([agents.md](../architecture/agents.md)).

## Контекст

- Роль отделена от исполнителя; процесс одинаков для обоих ([CONSTITUTION.md](../../CONSTITUTION.md)).
- До v0.6 нет Dashboard — человеку-PM работать неудобно (файлы и git напрямую).
- Качество автономного PM/QA-агента без обкатанных промптов и памяти (v0.7) не гарантировано.
- Definition of Done требует контроля качества — чьего именно, влияет на доверие к результату.

## Возможные варианты

### Вариант 1: Обе роли — человек (агенты только Developer)

- **Плюсы:** контроль качества и постановки задач в руках человека; минимум реализации в v0.2/v0.4; безопасный старт.
- **Минусы:** платформа менее «мультиагентна» на практике; узкое место — доступность человека.

### Вариант 2: Обе роли — агенты с подтверждением человеком ключевых решений

- **Плюсы:** демонстрирует ценность платформы; человек — только на контрольных точках (утверждение DoR, приём Done).
- **Минусы:** нужны промпты/адаптеры для двух дополнительных ролей уже в MVP; риск качества без памяти проекта.

### Вариант 3: Гибрид — PM человек, QA агент (или наоборот)

- **Плюсы:** постановка задач остаётся за человеком (наибольший риск ошибок), проверка автоматизируется (наиболее формализуемая часть).
- **Минусы:** асимметрия процессов; всё равно нужен агентский контур для одной из ролей.

## Решение

Принят **Вариант 2**: обе роли (Project Manager, QA Engineer) — агенты, человек подтверждает только на контрольных точках (например, приём Definition of Ready, финальное приёмочное решение Done). К моменту принятия golden path уже полностью автоматизирует роль Developer (`agents/claude-code`, EPIC-006) и проходит все переходы workflow через реальный HTTP (EPIC-008/009) — расширение на PM/QA логически продолжает уже проверенный технический путь, а не начинает его с нуля.

**Технический подход** (не переизобретается для каждой роли отдельно): один и тот же адаптер `agents/claude-code` (Docker-песочница, сетевой allowlist, короткоживущие секреты — ADR-006) используется для всех трёх ролей; различие — в промпте, построенном из `ExecutorTask.Role`, а не в отдельных Docker-образах или адаптерах. PM/QA-агентам для выполнения своей роли нужен доступ к `apps/api` (создание/планирование задач, вынесение вердикта тестирования) — новая запись в сетевом allowlist песочницы, сверх уже существующей (GitHub, AI-провайдер).

Реализация — отдельными эпиками после v1.0 открытия (не одним шагом): Orchestrator (кто решает, когда вызывать какую роль — сейчас пуст, сознательно отложен в EPIC-006), затем role-aware промпты в `agents/claude-code`, затем механизм подтверждения человеком на контрольных точках (ещё не существует ни в API, ни в Dashboard).

## Влияние на систему

Scope v1.0 (декомпозиция на отдельные эпики — Orchestrator, роли PM/QA в `agents/claude-code`, механизм подтверждения); `apps/orchestrator` (первая реализация); сетевой allowlist агентской песочницы (ADR-006) — доступ к `apps/api`; описания ролей `.claude/agents/`; требования к контрольным точкам процесса ([workflow.md](../architecture/workflow.md)).
