# ADR-002: Механизм доставки событий и формат схем

## Статус

**Принято** (решение архитектора проекта, 2026-07-19)

## Дата

2026-07-19

## Контекст

Событийное взаимодействие — базовый принцип системы ([event-model.md](../architecture/event-model.md), [events.md](../architecture/events.md)). Требовалось выбрать механизм доставки событий, гарантии и подход к схемам. Система — модульный монолит; все модули работают в одном процессе.

## Рассмотренные варианты

1. **Внутрипроцессная шина (In-Memory Event Bus)** — простейшее решение, без внешних зависимостей.
2. **Redis Streams** — персистентность и consumer groups ценой усложнения и критической зависимости от Redis.
3. **Выделенный брокер (NATS / RabbitMQ)** — полноценные гарантии ценой новой технологии вне стека.

## Решение

Для MVP используется **In-Memory Event Bus** — внутрипроцессная шина событий.

Обоснование:

- максимально простой вариант (KISS);
- без Redis и внешних брокеров;
- легко тестировать;
- легко заменить.

Ключевое условие: **интерфейс Event Bus ([interfaces.md](../architecture/interfaces.md)) неизменен при смене реализации.** В будущем реализация может быть заменена на Redis Streams или NATS — меняется только адаптер, контракт и код подписчиков не меняются.

Формат схем: события описываются в [events.md](../architecture/events.md); каждое событие несёт версию схемы. Журнал событий (обязателен по [event-model.md](../architecture/event-model.md)) реализуется подписчиком шины, сохраняющим события в PostgreSQL (детали — на этапе Infrastructure).

## Последствия

### Положительные

- Ни одной внешней зависимости для событийной модели в MVP; запуск и тесты тривиальны.
- Синхронная внутрипроцессная доставка упрощает отладку.
- Путь миграции определён заранее (Redis Streams / NATS) и не затрагивает потребителей.

### Отрицательные

- События не переживают падение процесса до фиксации в журнале; доигровка — из журнала.
- Все подписчики — в одном процессе (достаточно для модульного монолита).
