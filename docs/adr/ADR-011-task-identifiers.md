# ADR-011: Идентификаторы задач и эпиков

## Статус

**Decision Required** — формат идентификаторов входит в имена файлов, веток, PR и события; смена формата после старта потребует массовой миграции. Требуется решение архитектора (совместно с [ADR-004](ADR-004-task-storage.md)).

## Дата

2026-07-19 (заготовка)

## Проблема

Документы используют условные обозначения `TASK-NNN` / `EPIC-NNN`, но формат идентификаторов и правила их присвоения не зафиксированы: кто выдаёт номер, как избежать коллизий при параллельном создании задач (людьми и агентами), различаются ли пространства номеров по проектам.

## Контекст

- Идентификатор входит в: имя файла задачи, имя ветки (`feature/TASK-012-...`), ссылку в PR, данные событий.
- До решения [ADR-004](ADR-004-task-storage.md) задачи — файлы в git: последовательный счётчик требует координации (конфликты при параллельных PR).
- Платформа управляет несколькими проектами — нужна ли проектная приставка.

## Возможные варианты

### Вариант 1: Сквозной последовательный номер (TASK-001, TASK-002…)

- **Плюсы:** привычно и читаемо; короткие ссылки.
- **Минусы:** нужен единый выдающий счётчик; в файловом варианте — конфликты параллельного создания.

### Вариант 2: Последовательный номер в рамках проекта (ABC-123, как в трекерах)

- **Плюсы:** читаемость + масштабирование на много проектов; привычно по Jira-подобным системам.
- **Минусы:** требуется реестр проектных префиксов; тот же вопрос выдачи номера.

### Вариант 3: Дата + короткий слаг (TASK-20260719-event-bus)

- **Плюсы:** без координации (коллизии маловероятны); самоописательность; дружелюбно к файловому хранению.
- **Минусы:** длиннее; нет сквозной нумерации «по порядку».
