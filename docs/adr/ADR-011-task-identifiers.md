# ADR-011: Идентификаторы задач и эпиков

## Статус

**Принято** (решение архитектора проекта, 2026-07-21)

## Дата

2026-07-21 (заготовка — 2026-07-19; принято — 2026-07-21)

## Проблема

Документы используют условные обозначения `TASK-NNN` / `EPIC-NNN`, но формат идентификаторов и правила их присвоения не зафиксированы: кто выдаёт номер, как избежать коллизий при параллельном создании задач (людьми и агентами), различаются ли пространства номеров по проектам.

## Контекст

- Идентификатор входит в: имя файла задачи, имя ветки (`feature/TASK-012-...`), ссылку в PR, данные событий.
- До решения [ADR-004](ADR-004-task-storage.md) задачи — файлы в git: последовательный счётчик требует координации (конфликты при параллельных PR). ADR-004 принят: источник истины — PostgreSQL, что снимает проблему координации системно.
- Платформа управляет несколькими проектами — нужна ли проектная приставка.

## Рассмотренные варианты

### Вариант 1: Сквозной последовательный номер (TASK-001, TASK-002…)

- **Плюсы:** привычно и читаемо; короткие ссылки.
- **Минусы:** нужен единый выдающий счётчик; в файловом варианте — конфликты параллельного создания.

### Вариант 2: Последовательный номер в рамках проекта (ABC-123, как в трекерах)

- **Плюсы:** читаемость + масштабирование на много проектов; привычно по Jira-подобным системам.
- **Минусы:** требуется реестр проектных префиксов; тот же вопрос выдачи номера.

### Вариант 3: Дата + короткий слаг (TASK-20260719-event-bus)

- **Плюсы:** без координации (коллизии маловероятны); самоописательность; дружелюбно к файловому хранению.
- **Минусы:** длиннее; нет сквозной нумерации «по порядку».

## Решение

Принят **вариант 2 с уточнениями** — последовательный номер в рамках проекта, в два уровня:

1. **Публичный идентификатор** — строка вида `TASK-NNN` / `EPIC-NNN` (тип + порядковый номер, минимум три цифры, растёт без ограничения), уникальная **в рамках Project**. Полностью квалифицированная форма — пара (Project, публичный ID); внутри контекста одного проекта используется короткая форма, как сейчас. Проектные буквенные префиксы (ABC-123) не вводятся: Project уже является пространством имён, реестр префиксов был бы вторым способом сказать то же самое.
2. **Суррогатный ключ хранения** (внутренний, непубличный) — определяется схемой БД при реализации PostgreSQL-адаптера (v0.5); публичный ID — уникальный индексируемый атрибут, не первичный ключ. Это разводит стабильность внешних ссылок (файлы, ветки, PR, события) и свободу эволюции модели данных.

**Выдача номера** — модуль `task` через единый путь записи ([ADR-004](ADR-004-task-storage.md)): PostgreSQL-последовательность на проект исключает коллизии при параллельном создании людьми и агентами. **Переходный период** (до v0.5, файловый процесс): номер присваивается вручную последовательно, как сложилось на практике (TASK-001…039 платформенного проекта); коллизию обнаруживает ревью PR.

**В данных событий** и внешних ссылках всегда используется публичный строковый ID (плюс идентификатор проекта в метаданных события — уже предусмотрено [events.md](../architecture/events.md), «Общие поля всех событий»).

## Последствия

### Положительные

- Существующая практика (TASK-NNN в файлах, ветках `feature/TASK-NNN-...`, PR) легализована без миграции — формат уже используется корректно.
- Контракты `internal/domain/task` не меняются: идентификаторы остаются строками (`string`), как и было записано в коде «strings until ADR-011» — теперь это постоянное решение для публичного ID, а не временное.
- Коллизии параллельного создания решены системно на уровне ADR-004 (единый путь записи), а не соглашением.

### Отрицательные

- До v0.5 выдача номера остаётся ручной — приемлемо при текущем темпе (один активный проект, последовательная работа).
- Короткая форма ID неуникальна между проектами — любой межпроектный контекст обязан использовать полностью квалифицированную пару (Project, ID); это требование к будущим Dashboard/API (v0.6+).

### Влияние на существующие документы и код

Шаблоны [Task.md](../../.claude/templates/Task.md)/[Epic.md](../../.claude/templates/Epic.md) и [git-workflow.md](../development/git-workflow.md) уже соответствуют решению — изменений не требуют. Модель данных модуля `task` (v0.5) обязана реализовать последовательность на проект и суррогатный ключ. [DECISIONS_INDEX.md](DECISIONS_INDEX.md) обновлён.
