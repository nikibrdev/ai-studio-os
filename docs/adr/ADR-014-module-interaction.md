# ADR-014: Взаимодействие модулей — все проходят через Core

## Статус

**Принято** (решение архитектора проекта, 2026-07-19)

## Дата

2026-07-19

## Контекст

Требовалось определить, как модули и внешние участники (агенты, инструменты, приложения) взаимодействуют: только события или события + синхронные read-контракты; и закрепить направление зависимостей для слоёв исполнения.

## Рассмотренные варианты

1. **Строго только события** — автономия модулей ценой проекций и итоговой консистентности.
2. **События + синхронные read-only контракты** — простота чтения ценой синхронных связей.
3. **Гибрид** — read-контракты только для инвариантов.

## Решение

Принята схема с Core в центре — **все проходят через Core**:

```
Core → Events → Workflow → Agent Runtime → Tools
```

- **Core** — единственный владелец состояния и правил; любое изменение состояния — команда модулю Core.
- **Events** — Core публикует события ([events.md](../architecture/events.md)); подписчики (Workflow-координация, Orchestrator, проекции) реагируют на них.
- **Workflow** — реагирует на события и определяет следующий шаг процесса; результат — команды Core (не прямые действия).
- **Agent Runtime** (Orchestrator + `execution`) — запускает агентов по решениям Workflow.
- **Tools** — вызываются агентами; действуют только во внешней среде.

**Запрещено:**

| Запрет | Смысл |
|---|---|
| `Tool → Core` | Инструмент не вызывает ядро и не меняет состояние платформы; результат действия возвращается агенту, изменения состояния — только через события/команды уполномоченных компонентов |
| `Agent → Database` | Агент не обращается к хранилищам платформы; всё взаимодействие — через контракт Agent и инструменты |
| `Workflow → SQL` | Модуль workflow не содержит SQL и не обращается к БД напрямую; персистентность — только через порты Core |

Междоменное чтение: через события и собственные проекции (вариант 1); синхронные read-контракты не вводятся. Это соответствует действующей формулировке Конституции — поправка не требуется.

## Последствия

### Положительные

- Один центр истины и валидации; невозможно изменить состояние в обход правил.
- Слои исполнения (агенты, инструменты) полностью заменяемы — они не знают о внутренностях платформы.
- Готовность к выделению сервисов: связи уже событийные.

### Отрицательные

- Проекции для чтения требуют кода и дисциплины (итоговая консистентность).
- Инварианты, затрагивающие несколько модулей, проверяются через события/саги — сложнее прямого вызова.
