# ADR-004: Хранение задач — источник истины и отображение жизненного цикла

## Статус

**Принято** (решение архитектора проекта, 2026-07-19)

## Дата

2026-07-19

## Контекст

С v0.1 задачи существовали как markdown-файлы в `tasks/` (перемещение файла = переход состояния). С появлением модуля `task` и PostgreSQL требовалось определить источник истины состояния задач и судьбу файловой системы, включая отображение состояний Testing и Cancelled, для которых нет каталогов.

## Рассмотренные варианты

1. **Файлы — источник истины; БД — проекция** — прозрачность ценой конкурентных конфликтов и парсинга markdown как контракта.
2. **БД — источник истины; файлы — генерируемое представление** — строгая валидация переходов ценой отказа от ручного редактирования файлов.
3. **Гибрид по этапам** — отложенное решение с риском «временного навсегда».

## Решение

Принят **вариант 2**: markdown НЕ является источником истины.

```
PostgreSQL (Source of Truth)
        ↓
   Task Engine (модуль task)
        ↓
   Markdown Export (tasks/)
```

- **PostgreSQL — единственный источник истины** состояния задач и эпиков.
- **Task Engine** (модуль `task`) — единственная точка изменения состояния; переходы валидируются по [state-machine.md](../architecture/state-machine.md).
- **`tasks/` — только экспорт**: человекочитаемое markdown-представление, генерируемое из БД. Ручное редактирование экспортированных файлов не изменяет состояние системы.

Обоснование: это позволит построить веб-интерфейс (v0.6) без миграции данных; состояния Testing и Cancelled существуют в БД полноценно — вопрос отдельных каталогов исчезает (экспорт может отражать все 9 состояний).

### Переходный период

До реализации Task Engine (первые задачи v0.2) файловый процесс `tasks/` остаётся рабочим носителем по правилам v0.1 (включая временное отображение Testing → `review/`, Cancelled → `archive/`). После ввода Task Engine в строй выполняется разовый импорт файлов в БД, и `tasks/` переводится в режим экспорта.

## Последствия

### Положительные

- Строгая валидация переходов state machine; невозможны «нелегальные» состояния.
- Надёжные выборки, статистика, события; готовая основа для Dashboard без миграции.
- Полный жизненный цикл (9 состояний) без костылей файловой структуры.

### Отрицательные

- Прямое редактирование файлов задач перестаёт быть способом управления (после переходного периода) — все изменения через Task Engine (CLI/API).
- Требуется механизм экспорта и разовый импорт существующих задач.
