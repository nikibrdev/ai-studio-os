# Память агентов

## Назначение

Определяет концепцию памяти в AI Studio OS: какие знания накапливает платформа, как они используются агентами и какими принципами регулируется работа с памятью. Реализация запланирована на v0.7 Memory.

## Содержание

### Зачем нужна память

AI-агенты выполняют задачи в длительных проектах. Без памяти каждый запуск агента начинается с нуля: теряются контекст решений, история ошибок и знания о проекте. Память делает работу агентов накопительной: решения, выводы и факты о проекте переиспользуются.

### Виды знаний (концепция)

- **Знания о проекте** — архитектура, соглашения, ограничения управляемого проекта.
- **История решений** — принятые решения и их обоснования (дополняет ADR).
- **Опыт выполнения задач** — что сработало, какие ошибки повторяются.

### Принципы

1. **Память — не источник истины.** Источник истины — документация и код; память ускоряет доступ и дополняет контекст, но при противоречии приоритет у документации.
2. **Прозрачность.** Содержимое памяти хранится в читаемом виде (`memory/`) и доступно для аудита человеком.
3. **Разделение по проектам.** Знания одного управляемого проекта не смешиваются с другими.
4. **Расширяемость.** Хранилище памяти подключается через интерфейс; смена реализации не затрагивает ядро.

### Роль Qdrant

С v0.7 Qdrant реализован как производный, перестраиваемый индекс поверх `memory/` ([ADR-018](../adr/ADR-018-memory-embeddings-and-qdrant-schema.md), [EPIC-007](../roadmap/EPIC-007-memory-system.md)): агенты находят релевантные знания через `platform.MemoryProvider.Search`. Эмбеддинг сейчас — наивный локальный feature hashing (256 измерений, без внешних зависимостей и секретов), сознательно не семантический по сути (совпадение хешированных токенов, а не смысла) — честно задокументированное ограничение MVP; замена на реальную модель — предмет отдельного будущего ADR, локализованный в одной функции (`internal/infrastructure/memory/embedding.go`).

### Открытые вопросы

Концептуальный контракт Memory Provider описан в [interfaces.md](interfaces.md). Закрыты в EPIC-007:

- Структура каталога `memory/` — `memory/<projectID>/<id>.md` (frontmatter + Markdown), решение зафиксировано в [2026-07-22-memory-file-format.md](../../engineering/decisions/2026-07-22-memory-file-format.md).
- Политика записи — `Record`/`Search` вызываются явно кодом-потребителем; автоматический вызов из application-сервисов или будущего Orchestrator — решение уровня v0.9+ (Multi Agent), не этого эпика (тот же документ).
- Модель эмбеддингов и схема коллекций Qdrant — [ADR-018](../adr/ADR-018-memory-embeddings-and-qdrant-schema.md).

Остаётся нерешённым (сознательно, см. [2026-07-22-memory-file-format.md](../../engineering/decisions/2026-07-22-memory-file-format.md)):

- Политика устаревания и очистки памяти — `Record` только добавляет записи; без опыта реальной эксплуатации любое правило было бы придумано, а не выведено из потребности.
