# Инженерные принципы

## Назначение

Детализирует инженерные принципы, закреплённые в [CONSTITUTION.md](../../CONSTITUTION.md), до уровня практических правил проектирования и разработки. Конституция определяет «что обязательно»; этот документ — «как это применять и как распознать нарушение».

## Содержание

### Interface First

- **Суть:** публичное поведение любого модуля, инструмента или интеграции сначала описывается контрактом ([interfaces.md](interfaces.md)), затем реализуется.
- **Зачем:** контракты позволяют заменять реализации (агентов, хранилища, транспорт) без изменения ядра.
- **Как применяем:** новая возможность начинается с описания контракта в документации; реализация не может расширять контракт «по факту».
- **Признаки нарушения:** потребитель зависит от деталей реализации; поведение, не описанное контрактом, используется другим модулем.

### Documentation First

- **Суть:** документ (спецификация, ADR, описание модуля) создаётся до кода; документация меняется в том же PR, что и код.
- **Зачем:** в проекте с AI-исполнителями документация — главный носитель контекста; без неё агент выполняет задачу вслепую.
- **Как применяем:** правила — [documentation.md](../development/documentation.md); недокументированная функциональность не проходит Definition of Done.
- **Признаки нарушения:** «документацию допишем потом»; расхождение документа и поведения.

### Small Modules

- **Суть:** модуль имеет одну ответственность и обозримый размер; крупная функциональность собирается из малых модулей через события и контракты.
- **Зачем:** малые модули проще проверять, документировать и отдавать агентам в работу; ошибка локализуется границей модуля.
- **Как применяем:** перечень модулей и владение данными — [domain-model.md](domain-model.md); новая ответственность — повод для нового модуля, а не разрастания существующего (решение — через ADR).
- **Признаки нарушения:** модуль владеет чужими сущностями; описание ответственности модуля требует союза «и».

### Explicit Dependencies

- **Суть:** все зависимости объявлены явно и направлены по правилам [module-boundaries.md](module-boundaries.md); никаких скрытых связей через глобальное состояние, окружение или порядок инициализации.
- **Зачем:** явные зависимости делают систему читаемой и подменяемой; скрытые — источник «магии» и невоспроизводимых ошибок.
- **Как применяем:** зависимости передаются снаружи (инверсия управления); допустимые направления — только из матрицы границ модулей.
- **Признаки нарушения:** для понимания модуля нужно знать, «что ещё должно быть запущено»; тест требует настройки несвязанных подсистем.

### Immutable Events

- **Суть:** событие — неизменяемый факт; после публикации оно не редактируется и не удаляется. Изменение смысла — новое событие; изменение структуры — новая версия схемы.
- **Зачем:** журнал событий — достоверная история системы; на неизменности держатся проекции, аудит и память.
- **Как применяем:** правила — [event-model.md](event-model.md), каталог — [events.md](events.md); версионирование схем — [ADR-002](../adr/ADR-002-event-delivery.md).
- **Признаки нарушения:** «поправить событие задним числом»; подписчик получает разные данные при повторном чтении.

### Commands, Events, Queries

- **Суть:** запись, факт и чтение — три разных типа операций, разделённые явно на уровне контрактов модуля: **Command** («сделай Х», может быть отклонена) → **Event** («Х случилось», неизменяемый факт) → **Query** («покажи текущее состояние», не меняет данные). Даже без полноценного Event Sourcing разделение вводится сразу.
- **Зачем:** явное разделение делает контракт модуля читаемым (сразу видно, что меняет состояние, а что только сообщает или читает) и не создаёт архитектурного долга при будущем переходе к более event-driven реализации ([ADR-002](../adr/ADR-002-event-delivery.md), [ADR-014](../adr/ADR-014-module-interaction.md)) — рефакторинг не понадобится, изменится только внутреннее устройство, не контракт.
- **Как применяем:** каждый доменный модуль объявляет `Commands` (методы записи, валидируют и меняют состояние), `Queries` (методы чтения для слоя доставки — не межмодульный контракт, [ADR-014](../adr/ADR-014-module-interaction.md)) и публикует `Event`ы через платформенный `EventBus`. Уже применено: `internal/domain/task` (`Commands`, `Queries`, [events.md](events.md)) — образец для остальных доменных модулей.
- **Признаки нарушения:** метод одновременно меняет состояние и возвращает данные для отображения (смешение Command/Query — нарушение [CQS](https://en.wikipedia.org/wiki/Command%E2%80%93query_separation)); Query-метод используется как способ изменить состояние «между делом».

### Backward Compatibility

- **Суть:** опубликованные контракты (интерфейсы, события, API) эволюционируют без слома потребителей; ломающее изменение — осознанное решение через ADR.
- **Зачем:** платформа расширяема внешними агентами и инструментами; слом контракта ломает чужой код.
- **Как применяем:** добавление — предпочтительнее изменения; устаревание — через пометку Deprecated и период сосуществования; слом — только с ADR и планом миграции.
- **Признаки нарушения:** обновление ядра требует одновременной правки всех адаптеров; изменение схемы события без новой версии.

### Simplicity over Cleverness

- **Суть:** из работающих решений выбирается то, которое проще прочитать и сопровождать; «остроумные» приёмы, экономящие строки за счёт ясности, запрещены.
- **Зачем:** код и документы читают люди и агенты с ограниченным контекстом; хитрость сегодня — дефект завтра.
- **Как применяем:** KISS из Конституции; конкурентность, обобщения и абстракции вводятся только под доказанную потребность; «минимум магии» — в [coding-standard.md](../development/coding-standard.md).
- **Признаки нарушения:** для понимания строки нужен комментарий «на самом деле здесь…»; абстракция с единственной реализацией без причины.

### Связь с Конституцией

| Принцип Конституции | Детализация |
| --- | --- |
| Clean Architecture | [core.md](core.md), [module-boundaries.md](module-boundaries.md) |
| SOLID | Small Modules, Explicit Dependencies, Interface First |
| KISS | Simplicity over Cleverness |
| DRY | [documentation.md](../development/documentation.md): один факт — один источник |
| Event-Driven | Immutable Events, Commands/Events/Queries, [event-model.md](event-model.md) |
| Расширяемость без изменения ядра | Interface First, Backward Compatibility |
| Минимум магии | Explicit Dependencies, Simplicity over Cleverness |

При кажущемся конфликте принципов приоритет определяет Конституция; спор фиксируется как Open Question.
