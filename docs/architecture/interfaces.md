# Интерфейсы системы

## Назначение

Описывает ключевые интерфейсы (контракты) AI Studio OS на концептуальном уровне: ответственность, входные и выходные данные, ограничения и жизненный цикл каждого. Без кода — контракты формализуются в коде на этапе реализации, строго по этому документу.

## Содержание

### Принципы

1. Интерфейс определяется до реализации (Interface First) и документируется в момент создания.
2. Контракт описывает намерение, а не транспорт: конкретные форматы и протоколы фиксируются соответствующими ADR.
3. Изменение опубликованного контракта — только через ADR с оценкой обратной совместимости.

---

### Agent (адаптер агента)

- **Ответственность:** подключение конкретного AI-провайдера (Claude Code, OpenAI Codex, OpenHands, Gemini и др.) к платформе; выполнение назначенной работы в рамках роли.
- **Входные данные:** назначение на исполнение — задача (цель, scope, критерии приёмки), роль, контекст (документы, память, ограничения), перечень доступных инструментов.
- **Выходные данные:** поток событий о ходе работы; по завершении — Result: статус, отчёт по структуре из [CLAUDE.md](../../CLAUDE.md), артефакты, Open Questions.
- **Ограничения:** агент не выходит за scope назначения; внешние действия — только через доступные инструменты (целевое состояние, v0.8); адаптер не содержит доменной логики платформы; один адаптер — один провайдер.
- **Жизненный цикл:** Registered (адаптер зарегистрирован) → Assigned (получено назначение) → Running (выполняет, публикует события) → Completed | Failed (возвращён Result) → Released.
- **Decision Required:** формат обмена и способ вызова — [ADR-005](../adr/ADR-005-agent-adapter-contract.md); среда выполнения и изоляция — [ADR-006](../adr/ADR-006-agent-execution-environment.md).

---

### Tool (инструмент)

- **Ответственность:** выполнение ровно одного действия во внешней среде (операция с git, файлами, запуск проверок и т.п.) по декларированному контракту.
- **Входные данные:** декларация инструмента (имя, назначение, схема параметров, ограничения) — при регистрации; параметры вызова — при исполнении.
- **Выходные данные:** результат действия (данные или подтверждение) либо структурированная ошибка; событие о вызове (кто, что, с чем, итог).
- **Ограничения:** без доменной логики платформы; идемпотентность или её отсутствие декларируется явно; набор доступных инструментов определяется ролью и задачей; регистрация только явная (без автообнаружения).
- **Жизненный цикл:** Registered → Available → Invoked (многократно) → Disabled → Retired.
- **Decision Required:** формат декларации и механизм регистрации — проектирование v0.8; модель безопасности вызовов — там же ([tools.md](tools.md)).

### Event Bus (шина событий)

- **Ответственность:** доставка опубликованных событий всем подписчикам соответствующего типа; ведение журнала событий.
- **Входные данные:** публикация — событие (тип, версия схемы, данные, метаданные: идентификатор, время, источник, связанная сущность); подписка — тип события и обработчик.
- **Выходные данные:** доставка события каждому подписчику; подтверждение публикации; журнал для аудита и перестроения проекций.
- **Ограничения:** события неизменяемы; публикующий не знает подписчиков; порядок и гарантии доставки (at-least-once / at-most-once, повторная доставка, идемпотентность обработчиков) — свойства контракта, требующие решения; подписка не влияет на публикующую сторону.
- **Жизненный цикл:** событие: Published → Delivered (всем подписчикам) → Archived (в журнале); подписка: Registered → Active → Cancelled.
- **Принято ([ADR-002](../adr/ADR-002-event-delivery.md)):** реализация MVP — In-Memory Event Bus (синхронная внутрипроцессная доставка); журнал — подписчик с сохранением в PostgreSQL; интерфейс неизменен при замене реализации на Redis Streams / NATS.

### Workflow (процесс)

- **Ответственность:** применение правил процесса: проверка допустимости перехода задачи и определение следующего шага и ответственной роли.
- **Входные данные:** текущее состояние задачи, запрашиваемый переход (или произошедшее событие), контекст (кто инициирует, выполнены ли условия шага).
- **Выходные данные:** решение — переход разрешён (новое состояние, ответственная роль следующего шага) или отклонён (причина); события о переходе публикует модуль-владелец задачи.
- **Ограничения:** переходы — строго по [state-machine.md](state-machine.md); решение детерминировано (одинаковый вход → одинаковый выход); workflow не выполняет действий — только решает.
- **Жизненный цикл:** определение workflow: Draft → Published → Deprecated; применение — без состояния (вычисление на каждый запрос).

### Memory Provider (провайдер памяти)

- **Ответственность:** сохранение знаний проекта и их поиск (с v0.7 — семантический).
- **Входные данные:** запись — знание (содержание, тип, проект, источник, метаданные); поиск — запрос (тема/вопрос, проект, ограничения выборки).
- **Выходные данные:** подтверждение записи; результат поиска — релевантные записи с указанием источника и степени релевантности.
- **Ограничения:** знания разных проектов изолированы; память не источник истины ([memory.md](memory.md)); содержимое доступно для аудита человеком; реализация подключается через порт (файловая — до v0.7, Qdrant — после).
- **Жизненный цикл:** запись: Recorded → Indexed → Superseded; провайдер: Connected → Active → Disconnected.
- **Decision Required:** структура записей, политика записи и устаревания — проектирование v0.7 ([memory.md](memory.md)).

---

### Repository Provider (провайдер git-хостинга)

- **Ответственность:** выполнение операций с git-хостингом (GitHub): ветки, коммиты, Pull Request'ы, ревью, слияния, получение состояния.
- **Входные данные:** операция (создать ветку, открыть PR, запросить ревью, слить PR, получить статус) с параметрами; идентификация репозитория.
- **Выходные данные:** результат операции (созданная сущность, статус) либо структурированная ошибка; события об изменениях (PR открыт, ревью завершено, слито).
- **Ограничения:** весь доступ платформы к git-хостингу — только через этот контракт; агенты работают с git через инструменты, которые используют этот же контракт; провайдер не принимает процессных решений (когда сливать — решает процесс).
- **Жизненный цикл:** Connected (репозиторий подключён) → Active (операции и наблюдение) → Disconnected.
- **Decision Required:** способ подключения репозиториев — [ADR-013](../adr/ADR-013-managed-projects.md); политики слияния — [ADR-008](../adr/ADR-008-git-policies.md).
