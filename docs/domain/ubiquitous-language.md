# Ubiquitous Language — единый язык предметной области

## Назначение

Определяет ключевые понятия AI Studio OS как концепции бизнеса — что они значат для человека, ставящего задачу, и для исполнителя, её выполняющего, — до того, как эти понятия становятся Go-структурами. Термины здесь обязательны к единообразному использованию в разговоре, документации и коде ([DDD](https://en.wikipedia.org/wiki/Domain-driven_design): один язык для бизнеса и для инженеров).

## Содержание

### Project (Проект)

Инициатива разработки программного обеспечения, которой управляет платформа: у неё есть цель, репозиторий кода, команда исполнителей ролей и накопленная память. Project — граница, внутри которой существуют Epic и Task; знания одного проекта не смешиваются со знаниями другого.

### Epic (Эпик)

Крупная цель внутри проекта, слишком большая для одной единицы работы. Существует, чтобы человек мог мыслить и планировать на уровне ценности («сделать X»), а не сразу на уровне отдельных задач. Декомпозируется в Task.

### Task (Задача)

Единица работы с чётким началом, концом и владельцем на каждом этапе. Центральное понятие платформы: вокруг её жизненного цикла организован весь «золотой путь» ([golden-path.md](../architecture/golden-path.md)). Задача проходит формализованные состояния (канонически — [state-machine.md](../architecture/state-machine.md)) и на каждом этапе принадлежит определённой роли.

### Workflow (Процесс)

Правило, определяющее, какие состояния проходит Task, кто отвечает за каждый шаг и когда переход разрешён. Существует, чтобы процесс был предсказуемым и одинаковым независимо от того, кто (человек или Executor) выполняет роль на каждом шаге.

### Review (Ревью)

Акт проверки предложенного изменения (обычно — Pull Request) человеком или Executor'ом в роли Reviewer, завершающийся вердиктом: одобрено или требует исправлений. Существует как точка контроля качества до того, как изменение станет частью основной ветки.

### Artifact (Артефакт)

Любое долговременное инженерное произведение, созданное или изменённое системой исполнения: commit, Pull Request, исходный файл, Markdown, ADR, спецификация, test report, build report, диаграмма, скриншот, Figma-файл, release note (в перспективе: video, audio, dataset, prompt, knowledge entry). **Самостоятельный Aggregate Root** ([ADR-016](../adr/ADR-016-artifact-aggregate-root.md), решение архитектора 2026-07-20) — не часть Execution/Task/Project: может существовать до задачи, во время исполнения, после её завершения или независимо от конкретного исполнителя. Существует как доказательство того, что было сделано и как — Artifact не меняется после создания, он часть истории. Execution, которое произвело Artifact, никогда им не владеет — хранит только ссылку.

**Не является Artifact**: временный лог, прогресс выполнения, heartbeat, токен LLM, внутреннее сообщение агента — это принадлежит Execution (процессу), а не результату, который система обязана сохранить.

### Execution (Исполнение)

Один запуск Исполнителя (Executor) для выполнения задачи или шага процесса: от получения назначения до возврата результата. Существует, чтобы можно было ответить на вопрос «кто, что и когда делал» независимо от того, сколько раз задача переоткрывалась. Execution — не Bounded Context, а сквозная техническая возможность, координируемая Application Layer и используемая всеми контекстами ([bounded-contexts.md](bounded-contexts.md)).

### Agent / Executor (Агент / Исполнитель)

Два разных понятия, различаемые по [ADR-005](../adr/ADR-005-executor-contract.md) (принято): **Agent** — логическая роль-исполнитель (например, Developer Agent — «роль Developer, которую кто-то исполняет»); **Executor** — конкретный технический бэкенд, реально исполняющий эту роль (Claude Code, Codex, OpenHands, Aider, Cline, человек, в будущем — собственные модели). Цепочка: Role (Developer) → Agent (Developer Agent) → Executor (Claude Code).

Платформа не делает различия между видами Executor'ов на уровне ядра — она знает только, что Executor принимает задачу и возвращает Artifact, Status ([ADR-005](../adr/ADR-005-executor-contract.md)). В коде платформы (`internal/platform`) используется только термин **Executor**; «Agent»/«агент» остаётся допустимым термином документации о ролях ([agents.md](../architecture/agents.md), [.claude/agents/](../../.claude/agents/)).

### Context (Контекст)

Совокупность информации, которую Executor получает вместе с назначением, чтобы выполнить работу: описание задачи, релевантные документы, записи Memory, ограничения. Существует, чтобы Executor мог действовать осознанно, не «вслепую» — платформа явно собирает и передаёт Context, а не полагается на то, что исполнитель сам его найдёт.

**Не путать** с Bounded Context (DDD-термином о границах контекста в коде/предметной области) — это отдельное понятие, определённое в [bounded-contexts.md](bounded-contexts.md).

### Memory (Память)

Накопленное знание о проекте: решения, опыт выполнения задач, факты, полезные для будущей работы. Существует, чтобы Executor не начинал каждую задачу с нуля. Память — не источник истины (при противоречии выигрывает документация и код, [memory.md](../architecture/memory.md)), а ускоритель контекста. Bounded Context, обслуживающий этот термин, называется **Knowledge** ([bounded-contexts.md](bounded-contexts.md)) — переименование относится только к контексту, не к самой сущности/модулю Memory.

### Открытые вопросы

Нет: терминологическое разделение Agent/Executor и переименование `platform.Agent` → `platform.Executor` решены [ADR-005](../adr/ADR-005-executor-contract.md) (принято, 2026-07-20).
