# EPIC-009: Dashboard — веб-интерфейс

## Цель

Реализовать `apps/dashboard` (v0.8, [ROADMAP.md](../../ROADMAP.md)): веб-интерфейс на Next.js для наблюдения за проектами и задачами — первое человеко-читаемое окно в состояние платформы, работающее исключительно через уже реализованный REST API ([EPIC-008](EPIC-008-api-layer.md)). Результат по ROADMAP: «состояние платформы видно и управляемо через веб-интерфейс».

## Контекст

Строится **после** v0.9 API, несмотря на номер версии — решение и причина зафиксированы в [2026-07-22-api-before-dashboard-build-order.md](../../engineering/decisions/2026-07-22-api-before-dashboard-build-order.md): `apps/dashboard` может обращаться к платформе только через `apps/api` ([module-boundaries.md](../architecture/module-boundaries.md)), которого не существовало до EPIC-008.

При открытии эпика найден и решён блокирующий пробел: у API нет способа выдавать списки — `TaskProjection.Get`/`ProjectStore.Get` отдают ровно одну запись по точному идентификатору (уже отмечено как «самый заметный практический предел» в рисках EPIC-008). Без списковых операций Dashboard не смог бы показать даже список проектов. Решение архитектора: добавить `GET /projects` и `GET /projects/{id}/tasks` первыми задачами этого эпика — то же обоснование, что и точечные дополнения `ProjectService`/`TaskIDGenerator` в EPIC-008 (без них следующий слой не может выполнить свою роль).

Frontend-толчейн, оставленный открытым в [ADR-009](../adr/ADR-009-toolchain.md) («версия Node.js, конфигурация ESLint/Prettier и unit-фреймворк frontend фиксируются при создании `apps/dashboard`»), зафиксирован при открытии этого эпика (дополнение к ADR-009, 2026-07-22): Node.js 22 LTS, TypeScript, ESLint + Prettier, Vitest + React Testing Library.

**Scope первой версии — наблюдение, не управление**: ROADMAP описывает результат как «наблюдаемость процесса для человека», не как полнофункциональную панель управления. Первая версия — read-only: список проектов, список и состояние задач внутри проекта. Формы создания/управления (создать проект, запустить задачу, одобрить ревью и т.п. — все уже поддержаны API, EPIC-008) — сознательно отложены до реальной потребности, тот же принцип, что и во всех предыдущих эпиках («не строить сверх того, что доказанно нужно сейчас»).

**Канал реального времени — не в этой версии**: [ADR-003](../adr/ADR-003-api-protocol.md) предвидел, что «для доставки событий в UI в реальном времени потребуется дополнительный канал (SSE/WebSocket), проектируется вместе с Dashboard» — первая версия использует обычный запрос/перезапрос (Next.js Server Components, свежие данные при переходе/обновлении страницы), без нового канала доставки; выбор SSE/WebSocket — решение при реальной потребности в живом обновлении без ручного обновления страницы, не изобретается заранее.

## Scope

### Входит

- **Списковые операции** (точечное дополнение к EPIC-004/005/008, обоснование — раздел «Контекст» выше): `application.ProjectStore.List`, `application.ProjectService.ListProjects`, `application.TaskProjection.ListByProject`; `GET /projects`, `GET /projects/{id}/tasks` в `apps/api`; `docs/api/*.md` дополнены (Documentation First).
- Дополнение [ADR-009](../adr/ADR-009-toolchain.md) — frontend-толчейн (принято при открытии эпика).
- `apps/dashboard` — приложение Next.js 15 (App Router, TypeScript): каркас, конфигурация (ESLint/Prettier/Vitest), типизированный клиент REST API (`internal/application`-типы не импортируются — только форма ответов `docs/api/*.md`), базовая навигация/layout.
- Страница списка проектов (`/`) — имя, состояние.
- Страница проекта (`/projects/[id]`) — список задач с их состоянием.
- Страница задачи (`/projects/[id]/tasks/[taskId]`) — детали (заголовок, тип, scope, критерии приёмки, состояние).
- Компонентные/страничные тесты (Vitest + React Testing Library) на реальных HTTP-ответах через мок-сервер (не заглушки на уровне компонента).
- CI: `.github/workflows/verify.yml` дополняется job'ом для `pnpm install`/lint/test/build.
- Реальная проверка: `pnpm dev` против реально поднятого `apps/api` (+ PostgreSQL), просмотр в браузере.
- Закрытие эпика: критерии, ROADMAP (v0.8 — Завершено), PROJECT_MANIFEST, PROJECT_HEALTH, CHANGELOG.

### Не входит

- Формы создания/действия (создать проект, запустить задачу, одобрить ревью, тестирование и т.п.) — сознательно отложены до реальной потребности (см. «Контекст»).
- Аутентификация — не входит (ADR-012, Вариант 1, доверенная установка).
- Канал реального времени (SSE/WebSocket) — не входит (см. «Контекст»).
- E2E-тестирование Playwright — область QA Engine, не блокирует эту версию Dashboard (компонентные/страничные тесты на Vitest достаточны).
- Отчёты агентов, детальный журнал событий в UI — упомянуты в ROADMAP как часть v0.8, но не имеют готового источника данных сверх уже показанного состояния задачи; расширение — по реальной потребности после первой версии.

## Критерии завершения

- [ ] `GET /projects`, `GET /projects/{id}/tasks` реализованы, задокументированы в `docs/api/`, покрыты тестами.
- [ ] `apps/dashboard` — рабочее приложение Next.js: список проектов → список задач проекта → детали задачи, всё через реальные HTTP-запросы к `apps/api`.
- [ ] CI собирает, линтует и тестирует `apps/dashboard` наравне с Go-кодом.

## Риски и зависимости

- **Два точечных дополнения к закрытым эпикам** (списковые операции в Application Layer и `apps/api`) — решение архитектора при открытии этого эпика, обоснование — раздел «Контекст».
- **Read-only первая версия** — Dashboard не заменяет прямые вызовы API для управления процессом; формы — решение по реальной потребности, не изобретаются заранее.
- **Без канала реального времени** — обновление состояния только по явному запросу/перезагрузке страницы; ADR-003 предвидел этот вопрос, решение по нему явно отложено, не забыто.
- **Второй язык/тулчейн в репозитории** — Node.js/pnpm/TypeScript наравне с Go; CI и `make verify` должны покрывать оба, иначе frontend-код останется непроверяемым по факту (тот же принцип строгости, что и для всего Go-кода этой сессии).
