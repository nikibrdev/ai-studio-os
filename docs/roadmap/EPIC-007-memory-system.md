# EPIC-007: Memory System — память

## Цель

Реализовать Memory System AI Studio OS (v0.7, [ROADMAP.md](../../ROADMAP.md)): накопленное знание о проекте (`memory/`) с семантическим поиском на Qdrant поверх уже принятого контракта `platform.MemoryProvider`. Результат по ROADMAP: «агенты используют накопленный контекст проекта».

## Контекст

Контракт `platform.MemoryEntry`/`MemoryProvider` (`internal/platform/memory.go`) уже объявлен и стабилен — этот эпик его не меняет, только реализует. [memory.md](../architecture/memory.md) фиксирует принципы (память не источник истины; изоляция по проектам; прозрачность — читаемое хранилище; расширяемость через порт) и три открытых вопроса, закрываемых этим эпиком: структура каталога `memory/`, политика записи/устаревания, модель эмбеддингов и схема Qdrant.

[ADR-018](../adr/ADR-018-memory-embeddings-and-qdrant-schema.md) (принят при открытии эпика) закрывает вопрос эмбеддингов: наивный локальный feature hashing (256 измерений, без внешних зависимостей и секретов) — у сессии нет ключа ни к одному провайдеру эмбеддингов, тот же принцип, что и решение не имитировать реальный вызов AI-провайдера в EPIC-006. Замена на реальную модель — будущее решение, локализованное в одной функции.

Ключевые архитектурные рамки:

- **`memory/` — данные, не код** ([module-boundaries.md](../architecture/module-boundaries.md)): реализация живёт в `internal/infrastructure/memory`, каталог `memory/` в корне репозитория — только файлы знаний, которые эта реализация читает и пишет.
- **Файлы — источник истины, Qdrant — производный индекс**, тот же принцип, что журнал событий и проекции чтения (EPIC-004/005): `internal/infrastructure/postgres`/`eventbus` уже установили паттерн «durable-хранилище + перестраиваемый индекс» (`ReadJournal`/`Rebuild`) — Memory System использует его же (`Reindex` из файлов в Qdrant).
- **Политика записи и устаревания — вне scope.** Кто и когда вызывает `Record` (агент автоматически, человек, оба) — решение уровня Orchestrator/агентских процессов (v0.9+), не этого эпика: здесь реализуется работающая возможность (`Record`/`Search` вызываемы явно), не автоматическое поведение существующих application-сервисов. Политика устаревания/очистки — открытый вопрос, сознательно не решается здесь (см. Риски).
- **Qdrant поднимается тем же способом, что PostgreSQL** (EPIC-005, ADR-017) — сервис в `docker-compose.yml`, реальная проверка требует Docker.

## Scope

### Входит

- [ADR-018](../adr/ADR-018-memory-embeddings-and-qdrant-schema.md) (принят) — модель эмбеддингов, схема коллекции Qdrant.
- Формат файлов `memory/<projectID>/<id>.md` (frontmatter + содержимое) — durable, аудируемое хранилище.
- Naive-эмбеддинг (feature hashing, 256 измерений, стандартная библиотека) и REST-клиент Qdrant (`net/http`, без клиентской библиотеки — тот же принцип, что GitHub-адаптер).
- `internal/infrastructure/memory` — реализация `platform.MemoryProvider`: `Record` пишет файл и индексирует в Qdrant; `Search` ищет по вектору с фильтром `project_id`; `Reindex` перестраивает коллекцию Qdrant из файлов с нуля.
- `docker-compose.yml` — сервис `qdrant`.
- Реальная проверка вживую: запись, поиск, изоляция между проектами, `Reindex` после потери индекса — на настоящем Qdrant (Docker).
- Закрытие эпика: критерии, ROADMAP (v0.7 — Завершено), PROJECT_MANIFEST, PROJECT_HEALTH, CHANGELOG.

### Не входит

- Реальная нейросетевая модель эмбеддингов — решение отдельного будущего ADR, когда появится провайдер/ключ.
- Автоматический вызов `Record` из существующих application-сервисов или будущего Orchestrator — вне scope.
- Политика устаревания и очистки памяти — открытый вопрос, не решается здесь.
- Dashboard/UI для просмотра памяти (v0.8) — не входит.

## Критерии завершения

- [ ] `internal/infrastructure/memory` реализует `platform.MemoryProvider` целиком (`Record`/`Search`), контракт не изменён.
- [ ] Файлы `memory/<projectID>/<id>.md` — durable источник истины, `Reindex` перестраивает Qdrant с нуля из файлов.
- [ ] Qdrant — изоляция по `project_id` подтверждена реальным тестом (поиск в одном проекте не возвращает записи другого).
- [ ] `docker-compose.yml` поднимает Qdrant; интеграционные тесты — реальный прогон, а не только на фейках.
- [ ] `docs/architecture/memory.md` синхронизирован — три открытых вопроса (структура каталога, эмбеддинги/Qdrant) закрыты, оставшийся (политика устаревания) — явно сохранён как открытый.
- [ ] PROJECT_MANIFEST/PROJECT_HEALTH/ROADMAP/CHANGELOG синхронизированы при закрытии.

## Риски и зависимости

- **Наивные эмбеддинги — не семантический поиск по сути** (ADR-018) — принятое, честно задокументированное ограничение; замена на реальную модель эмбеддингов — будущее решение, не блокирует закрытие этого эпика.
- **Docker может быть не запущен** на машине разработчика в моменте — задачи пишутся и тестируются так, чтобы отсутствие Docker не ломало `make verify` (тот же паттерн, что EPIC-005/006); реальная проверка Qdrant — при поднятом Docker.
- **Политика устаревания/очистки памяти** — сознательно не решается (см. Scope); `Record` только добавляет, автоматического удаления нет.
- Наследуется от EPIC-004/005: отсутствие сквозной транзакции между агрегатами — не относится к Memory (файл + Qdrant не являются доменным агрегатом в смысле ADR-014/015), но тот же класс ограничений («несколько операций записи не атомарны между собой») справедлив и для пары «файл + Qdrant upsert» — задокументировать как известное ограничение, не решать здесь.
