# EPIC-006: AI Agent Runtime — исполнение агентов

## Цель

Реализовать AI Agent Runtime AI Studio OS (v0.6, [ROADMAP.md](../../ROADMAP.md)): целевую модель исполнения из [ADR-006](../adr/ADR-006-agent-execution-environment.md) — одно Execution = один Docker-контейнер с рабочей копией ветки задачи, сетевым allowlist (AI-провайдер + git-хостинг) и короткоживущими секретами — и первый реальный адаптер Executor поверх контракта [ADR-005](../adr/ADR-005-executor-contract.md): Claude Code, исполняющий роль Developer. Результат по ROADMAP: «агент-разработчик выполняет подготовленные задачи в исполняемой платформе» — платформа впервые замыкает петлю сама, без человека за терминалом на шаге выполнения.

## Контекст

Перед этим эпиком закрыт EPIC-005 (Infrastructure Layer, v0.5): золотой путь работает на реальном PostgreSQL, производственном EventBus и GitHub. `WorkService.StartTask` уже принимает предварительно выбранного Executor'а параметром — автоподбор исполнителя (Orchestrator) остаётся за скобками этого эпика: [ADR-007](../adr/ADR-007-pm-qa-executors.md) (Decision Required) блокирует только модель исполнителей по умолчанию для ролей **PM и QA**, не Developer («Developer по умолчанию — Claude Code» уже зафиксировано текстом ADR-007) — этот эпик не принимает решение по ADR-007 и не строит авто-оркестратор.

Ключевые архитектурные рамки:

- **`platform.Executor` типы конкретизируются** ([ADR-005](../adr/ADR-005-executor-contract.md): «форма — задача Domain Layer, когда появятся реальные Task/Artifact/Execution» — теперь они есть). `internal/platform` остаётся домен-независимым ([ADR-015](../adr/ADR-015-internal-layering.md)): `ExecutorTask`/`Artifact`/`ExecutionStatus` становятся конкретными структурами с примитивными полями (строки, срезы, время) — не псевдонимами доменных типов; вызывающая сторона (`internal/application`) собирает их из доменных геттеров, адаптер (`agents/`) не импортирует `internal/domain`.
- **Целевая модель контейнера** — по ADR-006 буквально: рабочая копия ветки задачи (клонируется при старте, уничтожается при завершении), воспроизводимый образ на базе уже проверенного Dev Container (`golang:1.24-bookworm`, git и make уже в образе), сеть по allowlist (AI-провайдер + git-хостинг), короткоживущие секреты через переменные окружения, секреты платформы в контейнер не попадают никогда.
- **Tool Layer не строится** — `docs/architecture/module-boundaries.md`: обход Tool Layer запрещён только «после v0.8»; до этого адаптер вправе действовать напрямую (git, файлы) без промежуточного слоя инструментов — Tool Layer (v0.8, [tools.md](../architecture/tools.md)) не входит в scope.
- **ADR-013 (managed-projects) не блокирует** — рабочая копия агента эфемерна (клон на Execution) независимо от того, какой вариант подключения репозиториев будет принят; этот эпик работает с одним жёстко заданным тестовым репозиторием.
- **Реальный прогон требует настоящих секретов** (git-токен, ключ AI-провайдера для вложенного Claude Code) — в отличие от EPIC-005, где отсутствие GitHub PAT только ограничивало один интеграционный тест, здесь это центральный результат эпика. Способ снабжения вложенного Claude Code учётными данными не решён заранее и фиксируется как открытый вопрос при подходе к последней задаче (TASK-056) — платформа никогда не извлекает и не переиспользует секреты текущей сессии для скрытых целей (см. историю сессии: отказ от извлечения GitHub OAuth-токена через curl).

## Scope

### Входит

- Конкретизация `platform.ExecutorTask`/`Artifact`/`ExecutionStatus` — плоские структуры вместо `any`.
- Docker-образ исполнения (`docker/execution/Dockerfile` или аналог) на базе Dev Container.
- Управление жизненным циклом контейнера Execution: клонирование рабочей копии, сетевой allowlist, инъекция короткоживущих секретов, уничтожение по завершении.
- Адаптер `agents/claude-code` — реализация `platform.Executor` (Accept/Artifacts/Status/Finish), запускающая Claude Code внутри контейнера.
- Обновление стек-документов, где ADR-006 всё ещё числится Decision Required (`docs/architecture/agents.md`, `tools.md`) — точечная синхронизация.
- Демонстрационный реальный прогон на одноразовой тестовой задаче.
- Закрытие эпика: критерии, ROADMAP (v0.6 — Завершено), PROJECT_MANIFEST, PROJECT_HEALTH, CHANGELOG.

### Не входит

- Orchestrator — автоматический подбор Executor'а по роли (ADR-007 Decision Required для PM/QA; Developer уже получает Executor параметром от вызывающей стороны).
- Tool Layer (v0.8) — адаптер действует напрямую, не через промежуточный слой инструментов.
- Managed Projects / `projects/`-манифест (ADR-013) — один жёстко заданный тестовый репозиторий, не многопроектное подключение.
- Массовый параллелизм исполнения (v0.9+, один хост в MVP по ADR-006).
- Удалённые раннеры (ADR-006 Вариант 3) — вне MVP.

## Риски и зависимости

- ~~Секреты вложенного Claude Code~~ — снято в TASK-056 как принятый честный предел: пользователю задан прямой вопрос, выбран вариант «пропустить реальный AI-вызов, зафиксировать предел». `Executor.New(image, gitToken, providerAPIKey)` уже готов принять реальный ключ, когда он появится — доработок кода не требуется, только предоставление `ANTHROPIC_API_KEY`.
- **Docker daemon может быть не запущен** на машине разработчика в моменте (как и в начале EPIC-005) — задачи пишутся и тестируются так, чтобы отсутствие Docker не ломало `make verify`; реальная сборка образа и прогон — при поднятом Docker.
- Наследуется от EPIC-005: `RepositoryProvider` не проверен вживую на реальном GitHub — тот же тестовый репозиторий может понадобиться и здесь; секрет один и тот же (`GITHUB_TOKEN`), решение о его добавлении в CI по-прежнему не принято.
- ADR-007/ADR-013 остаются Decision Required — не решаются этим эпиком (см. «Контекст»), не блокируют его результат.
