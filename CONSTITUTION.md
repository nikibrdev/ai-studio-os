# Конституция проекта AI Studio OS

## Назначение

Главный документ проекта. Определяет миссию, ценности, инженерные принципы и обязательные правила работы для всех участников — людей и AI-агентов.

Настоящий документ **не может быть изменён без отдельного ADR** (см. раздел «Правила изменения архитектуры»).

## Содержание

### Миссия

Создать открытую платформу, в которой разработка программного обеспечения ведётся смешанной командой из людей и AI-агентов по единому, прозрачному и воспроизводимому процессу — с качеством, не уступающим лучшим инженерным командам.

### Ценности

1. **Прозрачность.** Каждое решение зафиксировано в документации; каждое изменение проходит через понятный процесс.
2. **Качество выше скорости.** Незавершённая, но корректная работа ценнее быстрой, но небрежной.
3. **Простота.** Из двух работающих решений выбирается более простое.
4. **Ответственность ролей.** Каждая роль имеет явные обязанности и границы; исполнителем может быть человек или агент.
5. **Открытость.** Проект развивается как open-source и пригоден для внешних контрибьюторов.

### Инженерные принципы

Обязательны для всего кода и всех архитектурных решений:

- **Clean Architecture** — зависимости направлены внутрь; бизнес-логика не зависит от инфраструктуры.
- **SOLID** — в первую очередь Single Responsibility и Dependency Inversion.
- **KISS** — не усложнять без доказанной необходимости.
- **DRY** — без дублирования логики и документации.
- **Interface First** — сначала контракт (интерфейс), затем реализация.
- **Documentation First** — сначала документ (описание, ADR, спецификация), затем код.
- **Event-Driven Architecture** — модули взаимодействуют через события, а не прямые вызовы между доменами.
- **Modular Monolith** — на первом этапе один деплоймент с жёсткими модульными границами.
- **Расширяемость без изменения ядра** — новые агенты, инструменты и роли подключаются через точки расширения.
- **Минимум магии** — никаких скрытых соглашений, неявной генерации и неочевидного поведения.
- **Максимальная читаемость** — код и документы пишутся для читателя.

### Требования к качеству

- Код проходит форматирование и статический анализ, принятые в проекте (см. [docs/development/coding-standard.md](docs/development/coding-standard.md)).
- Любое изменение проходит ревью до слияния.
- Публичные контракты (интерфейсы, API, события) документируются в момент создания.
- Изменения, ломающие обратную совместимость, требуют ADR.
- Документация обновляется в том же изменении (PR), что и код, который она описывает.

### Definition of Ready (DoR)

Задача готова к взятию в работу, если:

1. Сформулирована цель и ожидаемый результат.
2. Определены критерии приёмки.
3. Указаны затрагиваемые модули/документы.
4. Указаны ограничения и зависимости от других задач.
5. Задача помещена в `tasks/ready/` и оформлена по шаблону [.claude/templates/Task.md](.claude/templates/Task.md).

### Definition of Done (DoD)

Задача считается выполненной, если:

1. Реализация соответствует критериям приёмки.
2. Код соответствует стандартам и прошёл ревью.
3. Тесты написаны и проходят (для этапов, где тестирование включено в scope).
4. Документация обновлена.
5. Изменения слиты в основную ветку.
6. Сформирован отчёт о выполнении (для AI-агентов — обязательно, см. [CLAUDE.md](CLAUDE.md)).
7. Задача перемещена в `tasks/done/`.

### Правила изменения архитектуры

1. Любое архитектурное решение фиксируется в ADR (`docs/adr/`) по шаблону [docs/adr/ADR-000-template.md](docs/adr/ADR-000-template.md).
2. Изменение ранее принятого решения оформляется новым ADR, который явно ссылается на заменяемый и переводит его в статус «Superseded».
3. Изменение настоящей Конституции допускается только через принятый ADR.
4. AI-агенты **не принимают архитектурные решения самостоятельно**: при недостатке требований агент фиксирует вопрос в разделе Open Questions и останавливается.

### Правила внесения изменений

1. Изменения вносятся только через Pull Request; прямые коммиты в основную ветку запрещены.
2. Один PR — одна задача (одна логическая единица изменений).
3. Каждый PR ссылается на задачу из `tasks/` и заполняет шаблон PR.
4. Коммиты оформляются по Conventional Commits (см. [docs/development/git-workflow.md](docs/development/git-workflow.md)).
5. PR проходит чек-лист [.claude/checklists/PR.md](.claude/checklists/PR.md) до запроса ревью.

### Требования к документации

1. Документация — часть продукта; недокументированная функциональность считается незавершённой.
2. Каждый markdown-документ имеет единую структуру: Заголовок, Назначение, Содержание, Статус, Последнее обновление.
3. Документация не дублируется: один факт — один источник истины, остальные документы ссылаются на него.
4. Язык документации — русский (вопрос об английской версии для публикации зафиксирован в Open Questions).
5. Правила ведения документации: [docs/development/documentation.md](docs/development/documentation.md).

### Требования к тестированию

1. Тестируемость закладывается на уровне дизайна: зависимости инвертируются, интерфейсы позволяют подмену.
2. Уровни тестирования: unit, интеграционные, end-to-end (Playwright).
3. Критическая бизнес-логика покрывается unit-тестами обязательно.
4. Тесты — часть DoD с момента включения тестирования в roadmap (v0.4 QA Engine и далее); на этапе v0.1 Foundation создание тестов запрещено.
5. Подробнее: [docs/development/testing.md](docs/development/testing.md).

### Правила работы с AI

1. AI-агент действует строго в рамках своей роли ([.claude/agents/](.claude/agents/)) и инструкции [CLAUDE.md](CLAUDE.md).
2. Перед началом работы агент обязан прочитать Конституцию, контекст проекта (`.claude/context/`) и описание задачи.
3. Агент не расширяет scope задачи, не добавляет технологии и не меняет структуру проекта без явного указания.
4. При неопределённости агент не додумывает требования — он фиксирует вопрос в Open Questions.
5. Результат работы агента проходит тот же процесс ревью, что и работа человека.
6. Агент обязан формировать отчёт после выполнения каждой задачи.
